# 课程项目架构设计

## 1. 核心概念

课程项目（Course Project）是 Jijian 平台上精准教学的基本容器。一个课程项目对应一节课的完整生命周期，从设计到执行到记录。

底层由**文件系统**提供 source of truth，上层由**Entity 视图**提供业务语义的结构化投影。Agent（Chat 驱动）和教师直接编辑共享同一套文件存储，保证两种操作模式下的数据一致性。

## 2. 项目结构

一个课程项目包含三个内容区域和一个审计视图：

```
course-project/
├── plan/              # 教案设计 — 文档型
├── execution/         # 课程执行设计 — 模块列表型
├── record/            # 课堂记录 — runtime 自动生成
└── (review)           # 审计视图 — 无 source folder，AI 实时产出
```

| 区域 | 编辑模型 | 产出者 | 消费者 |
|------|---------|--------|--------|
| plan/ | Notion-alike 文档编辑器 | 教师 + Agent | Agent, Review 视图 |
| execution/ | 模块列表编辑器（从模块库拖入 → 配参数） | 教师 + Agent | 学生端 runtime, 教师 Observation, Agent |
| record/ | 自动生成，不可手动编辑 | Live-lesson runtime | 教师回顾, Agent |
| Review 视图 | 只读，AI 驱动 | Agent cross-read plan + execution | 教师 |

## 3. 项目视图体系

项目视图分为两类 tab，对应两层不同的关注点。

### 3.1 平台层 Tab — 透明度

这些 tab 呈现 Jijian 平台的通用能力，不绑定任何特定 solution。换一个行业 PoC，它们依然存在。

**文件系统视图**

展示项目的原始文件结构，类似 IDE 的 file explorer。教师可以看到完整的目录树和文件列表，理解"我的项目到底由哪些文件构成"。

关键交互：点击 markdown 文本文件时，不展示原始 markdown 源码，而是切换到 Notion-alike 的富文本编辑模式，教师可以直接编辑内容。非 markdown 文件（JSON、资源文件等）按原始格式展示或以适当的只读视图呈现。

这个视图的价值在于让教师始终能理解系统是文件驱动的——Agent 做的所有操作最终都落到这些文件上，而不是某个不可见的黑盒数据库。

**Skill 视图**

只读视图，展示当前项目中生效的 skill 定义。教师可以看到执行设计中某个模块的 AI 行为（例如 Discussion 模块的引导策略）背后由哪个 skill 驱动、该 skill 的 prompt 内容是什么。

这个视图的目的是"打开引擎盖"——教师不需要也不能编辑 skill（skill 的生命周期和归属不在单个项目内），但可以理解 AI 行为的来源，消除黑盒感。

Skill 与项目文件的区别：skill 是平台预置的、跨项目复用的、甚至跨 solution 复用的资产，它的管理不在项目范围内。

### 3.2 方案层 Tab — 业务操作

这些 tab 是教育 PoC 特有的业务视图，教师在这里完成实际工作。

**教案设计 Tab**

对应 plan/ 文件夹的结构化视图。提供 Notion-alike 文档编辑器，教师在这里定义教学要求、核心素养目标、课程大模块划分、教学任务。

**课程执行设计 Tab**

对应 execution/ 文件夹的结构化视图。提供模块列表编辑器，教师从模块库中添加模块，配置每个模块的参数。这是一个 canvas/flow-centric 的操作界面，教师的心智模型是"我在搭一条流水线"。

### 3.3 两类 Tab 的关系

平台层 tab 和方案层 tab 操作的是同一份底层数据。教师在教案设计 Tab 中编辑的内容，在文件系统视图中能看到对应的 markdown 文件同步变化。教师在文件系统视图中直接编辑 markdown 文件，教案设计 Tab 的内容也会同步更新。

两层视图是同一份文件存储的不同投影——平台层展示"是什么"，方案层展示"做什么"。

## 4. 教案设计（plan/）

### 4.1 性质

Document-centric。教师的心智模型是"我在写一份文档"。核心职责是定义 **WHAT**：这节课覆盖什么教学要求、核心素养目标、课程的大模块划分、教学任务。

### 4.2 数据格式

结构化文档（Markdown），同时支持教师通过 Notion-alike 编辑器直接编辑，以及 Agent 通过文件读写生成和修改。

### 4.3 与执行设计的关系

教案和执行设计之间**不建立 foreign key 关联**。一致性由 Review 审计视图通过 AI lint 保证——Agent cross-read 两者后评估执行设计是否覆盖了教案要求、是否存在冲突。

这一设计选择避免了硬关联的脆弱性：教师修改教案措辞不会导致执行设计侧的引用失效。

## 5. 课程执行设计（execution/）

### 5.1 性质

Canvas/flow-centric。教师的心智模型是"我在搭一条流水线"。核心职责是定义 **HOW**：用什么交互形式来承接教案中定义的教学内容。

### 5.2 Manifest

执行设计的核心产物是一个 **manifest 文件**——一个有序的模块列表（JSON），由学生端执行框架原样消费，不经过编译。

Manifest 定义了学生的学习路径，例如：Read → 选择题 → Discussion → Summary。学生在这个框架内按顺序执行。

### 5.3 模块 Schema

模块类型的 schema 定义由后端提供，**只读、不可扩展**。在 Agent session 启动时注入。同一份 schema 服务三个消费者：

- **执行设计编辑器 UI：** 据此渲染参数表单
- **Agent：** 据此理解约束、生成合法配置
- **校验层：** 据此拦截非法状态

教师和学校不能自定义新的模块类型。

### 5.4 Module Params 中的两类字段

每个模块的参数天然分为两类，schema 设计中必须显式区分：

**结构化配置** — 机器直接消费，schema 硬校验。例如：时长、题目数量、难度等级、选项数量。编辑器渲染为表单控件（下拉、滑块、输入框）。

**Declarative Prompt** — LLM 消费，校验仅限非空检查。例如：给 AI tutor 的引导指令。编辑器渲染为文本框。

### 5.5 AI 相关字段的角色分离

模块中涉及 AI 的字段必须分为两组独立定义，服务于两个不同的 LLM 调用角色：

**Tutor Instruction** — 给 AI tutor 的行为引导。定义 tutor 在该模块内如何与学生互动。例如："引导学生理解比喻修辞的作用，鼓励学生从课文中找出例子，不要直接给出答案。"

**Completion Rubric** — 给 evaluator 的达标判断标准。定义学生在该模块内的完成条件评估逻辑。例如："学生能自主举出一个课文之外的比喻例子即视为达标。"

这两组字段的读取者不同、判断标准不同、甚至可以使用不同的模型。混合定义会导致调整 tutor 行为时意外改变通过条件，或者反过来。

### 5.6 每个模块的三层定义

每个模块节点在 manifest 中包含三层定义：

- **内容定义：** 展示什么
- **交互定义：** 学生做什么
- **完成条件：** 什么时候可以进入下一步

## 6. 完成条件

每个模块的完成条件决定学生何时可以点击"下一步"进入后续模块。三种类型：

**手动推进** — 学生完成内容浏览后点击"下一步"。适用于 Read 等被动消费型模块。

**硬性指标** — 由系统确定性判定。例如：选择题必须答对。执行框架直接校验，不需要 LLM 介入。

**AI-driven 评估 + 时间兜底** — 由 evaluator LLM 根据 completion rubric 判定学生是否达标。例如：Discussion 中学生是否自主得出了目标结论。必须配合 time counter 做兜底，防止学生卡死在某个模块。

## 7. 学生端 Runtime

### 7.1 执行框架

学生端是一个**状态机**，由 manifest 中的 module list 定义状态序列。学生在框架内按顺序执行，不可跳步。

### 7.2 模块内交互

每个模块内部由对应的 module type 实现渲染和交互逻辑。AI tutor 的活动范围被限定在当前 module 内部，读取该模块的 tutor instruction 作为行为引导。

### 7.3 原样消费

执行框架直接读取 manifest JSON，不经过编译或转换。Manifest 的 schema 同时是 design-time schema 和 runtime schema。

## 8. 教师端 Observation

LLM 驱动的观察系统，实时监控课堂执行状态。观察维度包括：

- 每个学生当前所在的模块步骤
- 选择题等硬性指标模块的提交历史
- Discussion 等开放模块的讨论深度评估
- 整体班级进度分布

Observation 是 module-aware 的——它知道每个学生当前在哪个 module，因此可以针对不同 module 类型展示对应的观察指标。

## 9. Review 审计视图

### 9.1 性质

只读视图，无独立 source folder。由 Agent cross-read 教案（plan/）和执行设计（execution/）后实时生成。

### 9.2 审计内容

- 执行设计中的模块是否覆盖了教案中定义的所有教学要求
- 执行设计与教案之间是否存在冲突
- 潜在的遗漏或冗余

### 9.3 设计意图

教案与执行设计之间的一致性是 **soft constraint**，由 AI lint 保证而非 foreign key。这让两个编辑空间保持独立性，避免硬关联带来的级联失效。

## 10. Agent 协作模型

### 10.1 双通道写入

教师直接编辑和 Agent 生成都落到同一个文件系统层。Agent 读取的永远是最新状态——包括教师手动修改的内容。

### 10.2 Schema 注入

Agent session 启动时，后端将所有模块类型的 schema definition 注入 session context。Agent 据此理解每种模块类型的合法参数范围，生成符合约束的 manifest。

### 10.3 Cross-reference

教案和执行设计之间的语义关联由 Agent 在 Chat context 中实时建立，不编码在文件结构中。Agent 同时拥有整个项目的上下文，自行完成 cross-reference。
