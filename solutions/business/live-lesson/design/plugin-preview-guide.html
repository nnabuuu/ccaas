01 —

为什么要做 Preview The Problem

三类受众 · 一条慢链路 · 都被卡住
!

反馈环 > 2 分钟

每次改一行代码要跑完整集成：4 个进程启动、创 session、假扮学生 join、走完 listen / practice / discuss 才看到一次评分。反馈环长、依赖重、噪音多。

!

编辑者改不了 JSON

课程编辑/教学设计师没法在 admin 后台直接编辑 answerKey JSON、切教师视角看效果。每改一处都要找工程师重起服务。

!

客户看不到 demo

插件做完了，给客户/招生看 demo 只能录屏或 screen share。每个题型都没有可分享的公网链接、没有 marketplace 入口。

04 —

Mini Backend NestJS, but Stripped

复用真实 providers · 剥离 classroom 语义
复用 · Reuse

哪些真实代码继续用

插件评分链路在 Preview 和生产是同一份代码，从 Mini Backend 复用：

  • ExerciseTypeRegistry — auto-discover @ExerciseType() 插件
  • GradingService — 调 registry.grade(ctx)
  • AiPromptBuilder — AI 评分类型直连真实 LLM
  • 所有 11 个 Plugin — bundle 内的 sanitize/grade/buildCheckItems 一行没改
替换 · Replace

哪些换成 Preview 专属

"完整 classroom 语义"在单插件预览里是噪音，换成内存版本：

  • InMemoryClassroomState — 替代 ClassroomService，Map 存 session
  • PreviewSessionService — 内存 session 管理，无 SQLite
  • PreviewClassroomController — 6 个轻量 endpoint
  • 动态 bundle 加载 — 命令行参数注入 plugin 包
不加载 · Drop

哪些直接不加载

对"单插件预览"无意义的子系统全部剥离：

  • ObserverEngine — 可选，仅 teacher role + classSubmissions 时按需启
  • DiscussService / PersonalTouch — 非单插件关注点
  • SnapshotService / MetricsAggregator — 单用户无需历史回放
  • SQLite / TypeORM / SSE / 3s polling — 全部去掉
  • 鉴权 / Session code / Join — 跳过身份语义

preview-classroom.controller.ts
TypeScript 
// 关键：gradingService.grade() 和 registry.buildCheckItems() 是从 live-lesson backend
// 完全相同的源代码 — Preview 只换了 controller 和 state 存储

@Controller('preview/sessions/:id')
export class PreviewClassroomController {
  constructor(
    private readonly state: InMemoryClassroomState,
    private readonly gradingService: GradingService,    // ← 真实的
    private readonly registry: ExerciseTypeRegistry,    // ← 真实的
  ) {}

  @Post('check')
  async check(@Param('id') id: string, @Body() body: { ans: unknown }) {
    const session = this.state.get(id);
    const t0 = performance.now();
    const gradeResult = await this.gradingService.grade(session.answerKey, body.ans);  // ← 真实代码
    const checkItems = this.registry.buildCheckItems(session.answerKey, body.ans, gradeResult);
    this.state.recordGrade(id, { key: session.answerKey, data: body.ans }, gradeResult, performance.now() - t0);
    return { gradeResult, checkItems };
  }
}

三个 UI 壳 CLI · Admin · Public Demo

同一套 React 组件 · 三种部署形态
A · CLI · 开发者

CLI Preview

启动：在 bundle 目录跑一行 npx exercise-preview .，CLI 同时拉起 Mini NestJS (4321) + Vite dev server (5283)，浏览器自动打开。

Stories 来源：本地文件系统 watch，*.stories.ts 改动 Vite HMR 实时热重载。

独家特性：Inspector 完整暴露、文件即编辑（IDE 就是你的编辑器）、AI 评分用开发者自己的 API key。

多 bundle 并存：exercise-preview ./pack-math ./pack-reading 一次性预览多个 plugin。

B · Admin · 编辑者

Admin Playground

路由：admin-next 新增 /playground/:bundle。三栏布局：左侧 Monaco JSON 编辑器、中间渲染学生/教师视图、右侧 Inspector。

实时校验：Monaco 内嵌 plugin.answerKeySchema 转出的 JSON Schema，输入时立即得到错误提示。

草稿持久化：admin 后台 playground_drafts 表，按 bundle + 用户 id 索引，跨设备 / 团队协作。

一键导出：把当前 answerKey 复制到剪贴板，或推送成 lesson manifest 的草稿。

C · Demo · 终端用户

Public Demo

部署：exercise-preview build 静态打包，每个 story 一个路由。短码分发：demo.kedge.com/p/long-division-abc123。

限流：demo API key 设 RPM 限制（防刷）；AI 评分类型走后端 proxy。

简化界面：隐藏 Inspector 和编辑器，终端用户只看练习本身。右下角"了解 KedgeAgentic 平台"引流。

分享场景：销售 demo、招生展示、客户体验、marketplace 入口。

三壳能力对比

每个壳的形态是受众诉求决定的：开发者要速度和透明度，编辑者要 GUI 和草稿，终端用户要分享链接和限流。

Why 关键

每个区段解决一个真实痛点

  • AnswerKey 对比 · 暴露"答案泄漏"问题 — sanitize 是否真的剥离了 correct、walkthrough 等敏感字段
  • Current State · 调试 canSubmit / handleCheckResult 错误时一目了然
  • Grade Call · AI 评分类型直接看 prompt + response，定位 LLM 调用错误
  • Lifecycle 追踪 · Proxy 包裹 plugin 方法，发现"应该调用却没调用"的回调缺失
  • Validation · 即时发现 schema 漂移（手写 manifest 时最常见错误）
How 实现

Proxy 追踪所有方法调用

function instrumentPlugin(plugin, tracer) {
  return new Proxy(plugin, {
    get(target, prop) {
      const value = target[prop];
      if (typeof value !== 'function') return value;
      return (...args) => {
        const t0 = performance.now();
        try {
          const result = value.apply(target, args);
          tracer.record(prop, { args, result, duration: performance.now() - t0 });
          return result;
        } catch (err) { tracer.recordError(prop, { args, error: err }); throw err; }
      };
    },
  });
}
!
公网 Demo 例外：Inspector 在 CLI 和 Admin 壳里完整暴露，但公网 Demo 默认隐藏 — 不向终端用户暴露评分内部细节（answers、AI prompt）。这是一个有意的安全边界。

数据流 · 从命令到评分 End-to-End Flow

CLI Preview · 学生视图 · 走完一轮

完整生命周期 · 11 步

下面演示一次最完整的开发者体验：从执行命令到看到评分结果，途中每一步都是显式的、可追踪的。

1
开发者执行 npx exercise-preview .
CLI 命令入口，参数是当前目录（或多个 bundle 路径）。
2
CLI 启动 · 两件事并发
a. StoryLoader 扫描 ./**/*.stories.ts → 得到 Plugin + Stories；
b. BundleLoader 扫描 ./backend/**/*.plugin.ts → 动态 require → 注入 Mini Backend；
c. 启动 Vite dev server (5283) + Mini NestJS (4321)。
3
浏览器打开 http://localhost:5283
PreviewApp 加载，左栏 StoryList 渲染 [Default, PartiallyAnswered, ReviewMode, TeacherObserve]。
4
用户点击 "Default" → 创建 session
POST /preview/sessions { storyId: 'long-division/Default' } → Mini Backend 创建 sessionId，存 answerKey + initialAns → 返回 { sessionId, sanitizedSpec }。
5
StudentStage 渲染 <plugin.Component>
传入 exercise=sanitizedSpec、ans={}、setAns、checkResultState={}、onDone。组件就是真实的 plugin React 组件，没有 mock。
6
学生交互 · ans 状态实时同步到 Inspector
填空、选择、画图…… 每次 setAns 都被 Inspector 捕获，开发者看到状态变化的完整轨迹。
7
点击 "提交" · 三步本地检查
a. plugin.canSubmit(ex, ans, checkResultState) → true；
b. plugin.formatSubmitData(ans, checkResultState) → submitPayload；
c. POST /preview/sessions/:id/check { ans: submitPayload }。
8
Mini Backend 执行评分 · 复用真实代码
a. gradingService.grade(answerKey, submitPayload) ← 真实代码路径；
b. registry.get(type).grade(ctx) → GradeResult；若为 AI 评分：AiPromptBuilder 调真实 LLM；
c. registry.buildCheckItems(key, data, gradeResult) → CheckItem[]；
d. state.recordGrade(...) ← Inspector trace。
9
前端拿到结果 → handleCheckResult
plugin.handleCheckResult(result, ex, currentState) → { checkResultState, allDone, ... }。这是 plugin 的 hook 点。
10
setCheckResultState → 重新渲染
Inspector 面板更新：grade 历史 + checkItems + lifecycle 调用一并展示。开发者看到完整的一轮闭环。
11
改 plugin 代码 → Vite HMR → 状态保留
最重要的一步：开发者改一行代码，组件热重载但 session state、ans、checkResultState 保留。这是迭代速度的核心。
Core Principle

复用真实代码，
零行为偏移。

Preview 的核心承诺：开发者在沙箱里看到的评分结果，与生产环境完全一致 — 因为跑的是同一份 plugin.grade() 代码。Mini Backend 不是 mock，是 NestJS 真实 IoC 容器；剥离的只是"完整 classroom 语义"（多人 session、SSE、Observer 等），评分链路一行不改。

1
同一份 GradingService Mini Backend 直接 import live-lesson 的 GradingService，不复制不重写。代码改了一处，两边一起生效。
2
Zod 校验链路一致 answerKeySchema 在 Preview 和生产是同一份 Zod schema。manifest 在 Preview 通过的，生产也一定通过。
3
AI 评分调真实 LLM AI 评分类型默认调真实 LLM API（开发者自己的 key）。--mock-ai 切到 fixture 是 CI 专用，不是默认行为。
4
Snapshot 回归检测 CI 把 stories 的 grade 结果做 snapshot，PR 任何"偏移"都立刻被发现 — Preview 是行为基线本身。

与插件架构的关系 Built On Top, Not Coupled

第 10 种契约 · 推荐但非强制
依赖方向

Preview 只依赖插件基础设施

依赖链：@kedge-agentic/exercise-preview → ExerciseTypeRegistry / GradingService / AiPromptBuilder（live-lesson backend 已抽取的 plugin 基础设施）→ @kedge-agentic/exercise-pack-*（具体 bundle）。

Preview 包不依赖具体 bundle。具体 bundle 通过命令行参数或 admin UI 选择注入。

第 10 种契约

Stories 是插件架构的扩展点

插件架构定义了 9 种契约（answerKeySchema / sanitize / grade / buildCheckItems / Component / canSubmit / handleCheckResult / enrichFromApi / ObserveClassView）。Preview 引入第 10 种契约：

// 推荐但非强制

风险与缓解 Known Risks

每个风险都有应对策略
风险影响缓解
AI 评分频繁调真实 LLM → 成本 中 --mock-ai 切 fixture；公网 demo 限流
Mini Backend 与真实 backend 行为偏移 高 严格复用 GradingService / Registry，不复制代码；CI 加 snapshot 测试对比两边 grade 输出
Stories 过时（schema 变更后未更新） 中 CI 跑 vitest stories.test.ts：每个 story 做 safeParse(answerKeySchema, story.answerKey)，失败即报错
公网 demo 被滥用（恶意流量、信息提取） 中 demo API key RPM 限流；隐藏 Inspector；短码可过期
Admin Monaco 与 plugin Zod schema 不同步 低 通过 zod-to-json-schema 自动转换；schema 变更触发 admin 缓存失效
iframe 嵌入的 postMessage 被滥用 低 origin 白名单 + 协议字段 Zod 强类型校验
Bundle 多版本共存冲突 中 Preview 同时只允许一个版本的同名 plugin；多版本场景用多个进程
