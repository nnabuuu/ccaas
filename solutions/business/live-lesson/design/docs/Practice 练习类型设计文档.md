# Practice 练习类型 · 设计文档

> 新增 Practice 题型：支持手写 / 拍照作答 + AI 渐进式提示 + 教师端实时观察  
> 示范任务：多项式乘法 (y + 2)(y − 2)

---

## 1. 概述

Practice（练习）是一种新的课堂执行题型，专为**需要书写过程的计算题**设计。与现有的 Matching Card（结构解码）、Matrix（矩阵构建）、Textarea（批判质疑）等题型并列，覆盖数学、物理等需要手写解题的场景。

### 核心特征

| 维度 | 设计选择 |
|---|---|
| **输入方式** | 统一作答区：手写画布 + 拍照上传混合多页 |
| **AI 介入模式** | Submit-then-Analyze：提交后由 AI Skill 判定，错误时渐进展示提示 |
| **提示策略** | 最多两次机会：第一次错 → 展示理思路 + 列公式；第二次错 → 展示完整解答 → pass |
| **教师端** | 实时统计 + 学生列表 + 个人弹窗（多页浏览 + 笔迹回放） |

### 设计原则

沿用平台四项产品哲学，增加练习题型特有原则：

- **任务始终是主角** — 题目卡居中，手写区域在下方自然延展
- **提示面板从右滑入** — 复用课文面板的交互模式（slide-in from right），做完自动收回
- **渐进式脚手架** — 学生先独立尝试 → 提交触发 AI 判定 → 按需展示提示（不主动暴露答案）
- **手写区域是连续的纸** — 多页纵向堆叠，上滑可回看，像真实的草稿纸

### 2.2 输入方式

手写和拍照**不再互斥**，统一在一个混合多页作答区内。学生可自由组合手写页和照片页。

#### 入口：输入方式选择

- 初始状态显示两个并列按钮：**「手写作答」** / **「拍照上传」**
- 点击「手写作答」→ 创建第 1 页手写页，展开作答区
- 点击「拍照上传」→ 触发系统相机或文件选择，照片作为第 1 页照片页展开作答区
- 选择后进入统一作答区，不再显示初始按钮
- 若删除所有页面 → 回到初始输入方式选择状态

#### 统一作答区（混合多页）

作答区支持两种页面类型自由混排：

| 页面类型 | 标识 | 内容 |
|---|---|---|
| **手写页** | teal「手写」badge | 独立 Canvas（1200×480），方格纸网格背景 |
| **照片页** | blue「照片」badge | 上传的照片（max-height 400px，contain 缩放） |

**工具栏**（sticky 在顶部）：笔 / 橡皮 / 撤销 / 清除 / 📷 拍照
- 笔 / 橡皮 / 撤销 / 清除 作用于当前活跃的手写页；若当前为照片页，工具栏显示「当前：照片页」提示
- 📷 拍照按钮触发文件选择 / 相机，上传后添加为新的照片页

**页面管理：**
- **纵向多页滚动** — 所有页面（手写 + 照片）直接可见，上滑回看
- **底部双按钮** — 「+ 新增手写页」和「📷 拍照上传」并列
- **删除页面** — 每页 header hover 出现「删除」按钮
- **拖拽上传** — 整个作答区域支持拖入图片文件，自动添加为照片页
- 活跃页左侧显示 teal 竖线标记
- 每页 header 显示页码 + 类型 badge（「手写」/「照片」）

#### 照片页细节

- 点击「拍照上传」或工具栏 📷 按钮触发系统相机（`capture="environment"`）或文件选择
- 支持一次选择多张照片，每张各创建一个照片页
- 支持拖拽上传（dragover 整个作答区高亮为 teal）
- 照片页以 `object-fit: contain` 展示，保持原始比例

### 2.3 提交与 AI 批改流程

**提交按钮可见条件**：必须有实际内容（手写页有笔迹 OR 照片页已上传照片）。空白手写页或无页面时提交按钮不显示。

```
    ┌─────────┐
    │ 选择输入  │ ← 「手写作答」/「拍照上传」
    └────┬────┘
         │ 选择后进入作答区
         ▼
    ┌─────────┐
    │  作答中   │ ← 学生书写 / 拍照（有内容后显示提交按钮）
    └────┬────┘
         │ 点击「提交答案」
         ▼
    ┌─────────┐
    │  批改中   │ ← 紫色脉冲点 + "AI 助教正在批改…"（~1.5s）
    └────┬────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
 ✓ 正确    ✗ 第一次错
 (完成)       │
              │ 红色结果卡 + 右面板自动弹出（Step 1 + Step 2）
              ▼
         ┌─────────┐
         │ 修改作答  │ ← 「修改答案，再试一次」
         └────┬────┘
              │ 点击「重新提交 (v2)」
              ▼
         ┌─────────┐
         │  批改中   │
         └────┬────┘
              │
         ┌────┴────┐
         │         │
         ▼         ▼
      ✓ 正确    ✗ 第二次错
      (完成)       │
                   │ 琥珀色结果卡 + 面板追加 Step 3 完整解答
                   ▼
              ┌─────────┐
              │  pass    │ ← 「已理解，继续下一题」
              └─────────┘
```

### 2.4 提示面板（右侧 slide-in）

面板由提交结果自动触发（非手动点击），结构：

| 组件 | 触发时机 | 内容 |
|---|---|---|
| **Header** | 面板打开时 | 图标 + "解题思路" + 提示进度 badge + 关闭按钮 |
| **Step 1 · 理思路** | 第一次错 | 点明需要用的知识点（多项式乘法 / 分配律） |
| **Step 2 · 列公式** | 第一次错 | **公式动画组件**：FOIL 展开 (a+b)(m+n) = am+an+bm+bn，四色标注，自动播放 |
| **Step 3 · 写过程** | 第二次错 | **逐行动画**显示完整解题过程，最终答案绿色高亮 |

面板可关闭 / 重新打开（左侧出现「查看解题思路 →」链接）。

### 2.5 公式动画组件（FormulaAnimation）

演示乘法分配律展开过程：

1. **Stage 0** — 显示 (a + b)(m + n)，所有变量中性色
2. **Stage 1** — a 和 m 高亮为 blue → 下方出现 `am`（blue 底色）
3. **Stage 2** — a 和 n 高亮为 green → 出现 `+ an`（green 底色）
4. **Stage 3** — b 和 m 高亮为 amber → 出现 `+ bm`（amber 底色）
5. **Stage 4** — b 和 n 高亮为 coral → 出现 `+ bn`（coral 底色）
6. **Stage 5** — 完成，显示代入提示：a=y, b=2, m=y, n=−2

未高亮的变量 `opacity: .25`，已展开的项带语义色底色。可手动重新播放。

- 进度条显示回放进度，支持暂停/重播

## 4. 状态机

### 4.1 学生端 Phase

| Phase | 描述 | UI 状态 |
|---|---|---|
| `work` | 初始作答 | 有内容时显示提交按钮；无内容时仅显示输入方式选择 |
| `checking` | AI 批改中（第1次） | 紫色脉冲 + "AI 助教正在批改…" |
| `wrong1` | 第1次错误 | 红色结果卡 + 右面板（Step 1+2） |
| `retry` | 修改作答中 | 有内容时显示"重新提交 (v2)"按钮 |
| `checking2` | AI 批改中（第2次） | 紫色脉冲 |
| `wrong2` | 第2次错误 | 琥珀色结果卡 + 面板追加 Step 3 |
| `passed` | 已完成 | 绿色"本题已完成"卡 |

### 4.2 提示面板状态

| 条件 | 面板行为 |
|---|---|
| `wrong1` / `retry` | 显示 Step 1 + Step 2 |
| `wrong2` / `passed` | 显示 Step 1 + Step 2 + Step 3 |
| 面板可手动关闭 | 任务区出现"查看解题思路 →"链接 |

```javascript
  expression: '(y + 2)(y − 2)',
  correctAnswer: 'y² − 4',
  hints: [
    { step: 1, title: '理思路', content: '...' },
    { step: 2, title: '列公式', component: 'FormulaAnimation', config: { a:'y', b:'2', m:'y', n:'−2' } },
    { step: 3, title: '写过程', solution: ['解：原式', '= y·y + y·(−2) + 2·y + 2·(−2)', '= y² − 2y + 2y − 4', '= y² − 4'] }
  ],
  maxAttempts: 2,
  inputMethods: ['handwrite', 'photo']
}
```

### 5.2 学生提交数据

```javascript
{
  studentId: '...',
  taskId: '...',
  attempts: [
    {
      version: 1,
      pages: [
        // 手写页
        { type: 'canvas', strokes: [{ tool: 'pen', color: '#1c1c1a', points: [{x, y}, ...] }, ...] },
        // 照片页
        { type: 'photo', photoUrl: 'https://...', photoData: 'data:image/...' },
```

## 7. 设计决策记录

### 7.1 为什么用纵向多页而不是分页切换？

**问题**：初版使用数字按钮（1, 2, +）切换页面，用户反馈不直观 — 看不出每页内容，容易误加页面。

**迭代过程**：
1. 数字按钮 → 缩略图导航条（每页 mini 预览）→ 纵向堆叠

**最终选择**：纵向堆叠。理由：
- 最接近真实草稿纸体验（上滑回看）
- 所有内容直接可见，无需心理模型
- "新增一页"在底部，自然延伸

### 7.2 为什么提示由提交结果驱动而非手动请求？

**初版**：学生主动点击"需要提示"→ 展示提示。  
**问题**：与 Submit-then-Analyze 模式不一致。学生可能在未尝试的情况下就查看提示。

**最终选择**：提交后由 AI 判定 → 错误时自动展示对应级别的提示。理由：
- 保证学生先独立尝试
- AI 判定结果自然引出提示
- 两次机会的节奏感更强：第一次给方向，第二次给答案

### 7.3 为什么右面板而不是 inline 提示？

**初版**：提示卡片在任务区下方 inline 展示。  
**问题**：提示和输入区域混在一起，学生需要反复滚动。

**最终选择**：右侧 slide-in 面板。理由：
- 复用课文面板的交互模式，学生已有认知
- 提示和作答区域并列可见，方便参考
- 做完可收回，不占空间

---

## 7.4 为什么手写和拍照合并为统一作答区？

**v2 设计**：手写和拍照是互斥的输入方式（二选一），通过两个 toggle 按钮切换。

**问题**：
- 如果学生只想拍照上传，选择「拍照上传」后无法补充手写批注
- 如果学生先手写再想拍照补充，需要切换模式，丢失手写内容的上下文
- 实际场景中，学生可能先在纸上写一部分 → 拍照 → 再在 canvas 上补充 → 混合提交

**v3 最终选择**：统一作答区，手写页和照片页可自由混排。理由：
- 匹配真实作答行为：学生可能部分手写、部分拍照
- 初始仍有输入方式选择（「手写作答」/「拍照上传」），保留明确的入口引导
- 选择任一方式后进入统一区域，之后两种方式随时可用
- 删除所有页面后回到选择状态，避免空白页困扰
- 每页独立标记类型（手写/照片 badge），教师端可据此分别处理

## 8. 可扩展方向

- **更多题型模板** — 将 FormulaAnimation 抽象为通用组件，支持配置任意公式展开
