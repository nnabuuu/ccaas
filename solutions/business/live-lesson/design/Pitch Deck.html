Part 1

问题

AI 进入教育的三个极端

极端 A — AI 替代老师

A

1 对 1 Chatbot Tutor

Khanmigo、Duolingo Max — 没有课堂、没有同伴、没有教师判断。AI 直接面对学生，教师出局。

核心矛盾

学习不只是信息传递 — 缺少课堂社交语境和教师的专业判断，AI 独角戏无法完成真正的教育。

极端 B — 老师用 AI 当工具

B

ChatGPT 生成教案、PPT、出题

AI 停留在备课环节，没有进入课堂过程。教师被迫从技术视角思考 — "prompt 怎么写"，而非教育视角。

核心矛盾

形式使用 — 技术改变了备课方式，但课堂本身没变。AI 的能力被浪费在课前阶段。

极端 C — Agent 黑盒丢给学生

C

用 Coze / GPTs / Dify 搭学科 Agent

让学生自由对话 — 对话驱动，无法约束学习路径。

  • 不可控 — 学生跑偏失焦，无法约束学习路径
  • 不可观测 — 教师不知道 30 个学生和 Agent 聊了什么
  • 不可解释 — Agent 为什么这样回答？无法归因、无法改进

共同问题 — AI 能力 vs. 教师掌控力

现状
缺失
极端 A
AI 直接教学
教师不在场
极端 B
教师用 AI 备课
AI 不在课堂里
极端 C
Agent 在课堂里
教师看不见、管不了

我们要解决的

AI 在课堂过程中发挥作用，同时教师仍然是掌控者。

Part 2

我们的做法

技术侧

核心概念 — Describable Component

一节课 = 一组可描述的 Component。
每个 Component 声明：

  • 类型 — quiz / match / matrix / discuss / …
  • 内容 — 题目、素材、选项
  • 评分规则 — answerKey
  • observe — 关注哪些维度的学生表现
// quiz component 示例 { "type": "quiz", "content": { "stem": "SAS 判定需要…", "options": ["A", "B", "C"] }, "answerKey": "B", "observe": ["accuracy", "time"] }

Part 3

教育侧的意义

转变 1 — 从"怎么用 AI"
到"关注什么信息"

  • 教师不再需要学 prompt engineering
  • 不需要搭 Agent
  • 回到教育本职 — 这个环节我想观察学生的什么？

observe 定义

Component 的 observe 字段就是这个问题的答案。技术把教育问题还给了教育者。

转变 2 — 课堂有效信息透明化

传统课堂

教师凭经验 + 巡视感知学情
信息损失巨大

Live Lesson

每个学生在每个 step 的选择、耗时、AI 对话轮次、卡住位置 — 全部可见

以前不可能观察的信息

思考过程、错误模式、求助行为 — 现在透明了。教师第一次能看到全班每个人在每个环节的真实状态。

转变 3 — Teachingless

机械性固定知识的讲授 → AI 驱动的结构化流程承担

Coach

根据实时数据精准干预个别学生

情绪洞察者

关注 stuck 学生、发现焦虑信号

决策者

"Q1 错误偏高" → 决定是否全班讲解

不是让教师无事可做 — 是让教师做只有人才能做的事。
