# Live-Lesson 教师端 · 学生自学总览 — 页面设计说明

> 配套 Mockup：`live_lesson_summary_zoomable.html`
> 适用范围：Jijian Live-Lesson PoC · Surface 层

---

## 1. 页面定位

**触发时机**：学生在 Anchor 引导下完成自学 → 课堂讨论开始之间的 **Gap 阶段**（约 2–5 分钟）。

**核心矛盾**：教师从「观察者」切回「主导者」前，需要在极短时间内消化全班自学结果，并准备下一阶段的讲解 / 提问 / 干预。这个 Gap 的成败决定了课堂讨论的精度。

**不是什么**：
- 不是教学结束后的复盘报告（那是另一个页面）
- 不是给学生看的成绩页（那是另一个角色）
- 不是数据看板（数据展示是手段，决策支持是目的）

---

## 2. 设计目标

页面在 30 秒内回答教师的三个决策：

| 决策 | 对应页面区块 |
|---|---|
| 整体讲到哪了？需要重讲什么？ | 指标卡 + 知识点掌握情况 |
| 下一步叫谁回答？回答什么？ | 学生分布矩阵 + Detail Panel + 推荐提问候选 |
| 哪些学生需要个体关注？ | 学生分布矩阵的「需要关注」象限 + 异常学生指标 |

**反命题**：不展示「教师其实不需要看的」数据。提交时间分布、答题速度直方图、参与度趋势曲线都不进这个页面 — 那些是数据分析师的页面，不是教师在 Gap 内的页面。

---

## 3. 信息架构（5 层）

| 层级 | 区块 | 角色 |
|---|---|---|
| L1 | Header | 课堂上下文 + 提交进度 |
| L2 | 指标卡 × 4 | 整体快照（What） |
| L3 | 分布矩阵 ＋ Detail Panel | 个体识别（Who + Why） |
| L4 | 行动建议 × 2（提问候选 / 建议重讲） | 决策推荐（How） |
| L5 | 知识点掌握条形图 | 知识点诊断细节 |

视觉层级：L3 占据最大视觉权重（页面中部最大空间），因为「叫谁回答 + 为什么叫 TA」是 Gap 内最复杂的决策。L1 / L5 视觉权重最低。

---

## 4. 组件设计

### 4.1 Header
- 课堂上下文：班级 / 课题 / 用时
- 右上角提交进度：`已提交 / 总数`
- 不重复教师已知信息（如完整教学计划）

### 4.2 指标卡 × 4

| 卡 | 计算 | 触发的下一步决策 |
|---|---|---|
| 整体掌握度 | 全班加权平均（按知识点权重） | 决定讲解深度 |
| 学优 / 学困 | 双轴 4 象限映射后聚类计数 | 判断要不要回顾基础 |
| 异常学生 | 「吃不饱」象限计数 | 是否单独挑战拔高 |
| 薄弱知识点 | 班级错误率 > 30% 的知识点数 | 决定重讲范围 |

每张卡对应一个**独立**决策维度。如果两张卡触发同一个决策，删掉冗余的那张。

### 4.3 学生分布矩阵（核心组件）

**双轴模型**：

- **X 轴 · 掌握度**：答题正确率 × 用时合理性 × 是否需要提示
- **Y 轴 · 参与度**：停留时长 × 操作频率 × 是否走神（长时间无操作）

**4 象限**：

| 象限 | 学生类型 | 教师动作 |
|---|---|---|
| 高掌握高参与 | 学优 | 展示标杆 / 拔高 |
| 低掌握高参与 | 努力但困惑 | 暴露问题 / 精准辅导 |
| 高掌握低参与 | 吃不饱 | 追加挑战 / 警惕「蒙」的可能 |
| 低掌握低参与 | 需要关注 | 重点干预 / 课后跟进 |

**交互**：
- 缩放：`+/−` 按钮、滚轮、双击聚焦象限、拖拽平移
- AI 重点学生（3–5 名）作为 named overlay 永远显示
- 缩放进入更细粒度后，应**逐步浮现**更多 named label（类似 Google Maps 缩放时街道名字陆续出现）
- 单击姓名 → Detail Panel 切换到该学生

**Overplotting 处理**：底层始终运行轻量 force-collide jitter（≤ 5px 偏移），保证完全相同坐标的学生不会互相完全遮挡。jitter 由真实坐标的小幅扰动构成，不引入数据失真。

### 4.4 Detail Panel

页面里**最关键的设计**。结构：

1. **学生姓名 + 象限标签**
2. **4 项关键指标**：掌握度 / 参与度 / 用时 / 完整度
3. **关键卡点**：该学生在自学过程中卡住的具体位置（题号 + 证据，如「3 次错答 → 用提示后才对」）
4. **为什么推荐 [意图]**：把**单点信号 × 班级聚合信号**交叉，回答教师「为什么是 TA」
5. **CTA：加入提问候选**

第 4 段（「为什么」）是这个页面区别于一般「学生信息卡」的关键。教师收到 AI 推荐时的本能反应是质疑，所以推荐必须自带理由。理由必须由数据支撑（不是 AI 主观判断），且必须把个体信号上升到班级意义上去。

例：
> 王芳在 Q3 上的卡点是全班 18 名学生（40%）共同卡点。让她讲解题思路，能把"配方法 → 顶点公式"这个隐性衔接暴露出来。

这个推理只有在 GLM-Flash event log 这层架构同时具备 per-student 和 per-turn 数据时才能生成。

### 4.5 推荐提问候选

三类**意图**，每类各一名候选：

| 意图 | 选谁 | 用来做什么 |
|---|---|---|
| 展示标杆 | 学优 + 解题路径清晰 | 让全班看到「做对了是怎么做的」 |
| 检验掌握 | 中等偏上 | 用确定能答对的题验证掌握度（不是为难） |
| 暴露问题 | 努力但困惑 + 卡点典型 | 让全班看到「卡点在哪」，精准重讲 |

**为什么按意图分类而不是按成绩排序**：教师选人的真实逻辑不是「谁好叫谁」，而是「我下个动作要做什么 → 谁最适合配合这个动作」。AI 推荐应该绑定意图。

### 4.6 建议重讲

班级错误率 > 30% 的知识点。按错误率降序。红色为危险（> 40%），黄色为警示（30–40%）。最多列 2 个 — 列多了教师反而不重讲。

### 4.7 知识点掌握情况

完整知识点列表 + 横向条形图。色阶：≥ 80% 绿、60–80% 灰、40–60% 黄、< 40% 红。这是给教师**校验 AI 推荐合理性**的原始数据视图，不参与决策但允许教师反查。

## 5. 关键设计决策与理由

### 5.1 为什么用双轴 4 象限，而不是简单的成绩排名？

成绩排名只看「正确率」，看不到「过程」。双轴的关键依赖是 GLM-Flash event log 提供的 per-turn 过程信号 — 这是 Live-Lesson 区别于传统在线作业系统的核心能力。如果不用双轴，相当于浪费了这层架构。

「吃不饱」（高掌握低参与）和「努力但困惑」（低掌握高参与）这两类学生在排名视图里完全不可见 — 但他们恰恰是 Live-Lesson 的高价值教学机会。

### 5.2 矩阵选缩放而不是密度网格 binning

考虑过把矩阵切成 6×4 网格、用 cell 颜色显示密度，但放弃了：

- Binning 是「过早聚合」，丢失粒度
- 缩放 + jitter 保留连续坐标的数据保真度
- 缩放是用户熟悉的交互（地图模型），无学习成本
- 量化分数（5 道题只有 6 个掌握度档位）通过 jitter 自然散开

**Trade-off**：缩放是交互成本（教师需要主动操作），binning 是认知成本（教师需要解读色块）。我们选了交互成本，因为：
1. 默认视图已经包含 named overlay（不缩放也能用）
2. 缩放只在教师想看细节时才需要

### 5.3 Detail Panel 的「为什么」段落

这是把 AI 从「推荐人」升级到「推荐教学动作」的关键。如果只显示「王芳 · 努力但困惑」，教师还要自己脑内推理「为什么叫她」；显示「她的卡点 = 班级 47% 共同卡点」之后，教师拿到的是**完整的教学决策包**：人 + 理由 + 预期效果。

### 5.4 命名 highlight 的人数控制（3–5 名）

太少（1 名）= AI 偷懒；太多（10+ 名）= 教师选择疲劳。3 名对应三种意图，5 名包含两个备选。缩放后再披露更多 — 这是 Progressive Disclosure 标准做法。

## 6. 数据来源映射

| 页面元素 | 数据源 | 三层架构定位 |
|---|---|---|
| 整体掌握度 | event log 聚合统计 | GLM-Flash |
| 学生 X / Y 坐标 | per-student 多信号融合 | GLM-Flash → Surface |
| 关键卡点列表 | per-turn 事件 | GLM-Flash |
| 班级共同卡点（用于「为什么」） | event log 跨学生聚合 | GLM-Flash |
| 知识点掌握度 | anchor checkpoint × event | Opus（anchor 定义）+ GLM-Flash |
| 提问意图推荐 | 规则 + 轻量 LLM 排序 | Surface 层 |

「为什么推荐」段落需要的是**单点信号 × 班级聚合信号**的交叉查询能力，这要求 event log 的 schema 设计必须支持 per-student 时序回放和 per-anchor 横切聚合。

## 7. 待解决的设计问题

- **中等学生归属**：处于 4 象限边界（如答对 3/5 题 + 中等参与）的学生，硬归入某一象限会误导。可能需要引入「中等」第 5 类，或在双轴上做软归属。
- **缩放后 label 渲染**：viewBox 整体缩放会让文字一起放大。理想做法是字号锁定（vector-effect 或 React 重渲染），并加 collision detection。MVP 阶段可不做。
- **Detail Panel 的联动策略**：教师缩放矩阵到学优区时，Detail Panel 不应自动切换显示对象（保持教师当前对比意图）。但点击 cell（如果将来加了点击响应）是否切换需要进一步定义。
- **未提交学生**：当前只在 Header 角落显示「42 / 45」。如果 Gap 内教师要 nudge 这 3 人（提醒交卷），需要单独的 affordance（如可点击的「3 人未提交 ↗」入口）。
- **教师手动加入候选** 与 **AI 推荐候选** 的视觉区分：当教师从矩阵主动选了一个 AI 没推荐的人，列表里要不要标记？

## 8. 实施建议

- **首选实现栈**：React + TypeScript + d3-zoom（或 react-zoom-pan-pinch）。SVG 仍是矩阵的最佳载体，避免 Canvas（命名 hit-test 复杂）。
- **CSS Token**：CSS 变量从 Jijian 主题系统注入，不要硬编码颜色。当前 mockup 的颜色仅作演示。
- **响应式**：横屏 13" 以上为主目标。手机端不在本页面范围（手机不该是教师讲台前的设备）。
- **性能**：45 学生的 DOM 节点数完全不是性能瓶颈。担心的是 event log 实时聚合查询 — 数据层应在 Gap 触发时一次性 snapshot，不要在 Surface 层做实时聚合。
