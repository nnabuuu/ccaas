# @ Picker Reference — 设计审查报告

基于 7 位 UX 理论家的方法论，对 @ Picker Reference v4（双触发模型）进行模拟审查。

---

## 1. Jakob Nielsen — 10 条启发式审查

逐条对照，评分 1-5（5=完全合规，1=严重违反），聚焦有问题的条目。

### H1: 系统状态可见性 — 评分: 4/5

**合规点：**
- 暂存区托盘实时显示已添加的项数和内容 ✓
- 高亮项有明显的左侧色条和背景变化 ✓
- 面包屑显示当前导航深度 ✓
- 快捷键提示条动态变化，反映当前可用操作 ✓

**问题 P1-1：暂存区从"空"到"有"的状态跳变缺乏过渡反馈。** 用户第一次按 Space 添加时，暂存区突然从无到有出现一个绿色条。这个状态变化发生在 Picker 顶部，但用户的视觉焦点在列表中间——可能注意不到。

建议：第一次 Space 添加时，在高亮项旁边做一个短暂的"飞入"微动画（项向上飘到暂存区位置），建立视觉因果关系。

**问题 P1-2：`/` 命令插入后，输入框中的 pill 没有状态反馈区分。** `@` Token 和 `/` pill 都静静地待在输入框里，用户不知道"这条消息已经绑定了一个 Skill"。

建议：`/` pill 可以有一个微弱的脉冲动画或不同的边框样式，暗示"这是一个待执行的命令"。

---

### H2: 系统与现实世界的匹配 — 评分: 3/5

**合规点：**
- Entity icon 三层体系用了学生头像、菜谱图标等现实世界隐喻 ✓
- `@` 触发对应 "mention/引用" 的社交媒体心智模型 ✓
- `↗` 箭头直觉传达"跳转到外部" ✓

**问题 P2-1（重要）：`@` 在教师心智中可能不是"引用"的意思。** 社交媒体用户理解 `@` = mention 人。但教师想"我要调出那个视频"时，脑子里的动词是"打开"、"找到"、"调用"，不是 "at"。`@` 的触发符对非技术用户来说缺乏现实世界的 mapping。

严重程度：中高。这不会阻止使用（教师看到提示后会学），但会拖慢 onboarding。

建议：首次使用时，不要只在输入框 placeholder 里写"键入 @"，而是在 chatbox 上方固定一个 onboarding 提示："输入 @ 可以引用教案、学生、课堂记录"，用用户语言描述结果，而不是操作。

**问题 P2-2："暂存区"这个概念在现实世界中没有直接对应物。** 购物车类比部分成立，但 Picker 的暂存区在视觉上不像购物车。用户可能不理解"Space 添加到暂存 → 后续 Enter 或 Esc 提交"这个两步流程的必要性。

建议：暂存区的文案从"已添加"改为"准备引用"或使用购物车比喻——显示一个小购物袋图标 + 数字。

---

### H3: 用户控制与自由 — 评分: 5/5

这是你的设计最强的一条。

- Esc 始终可退出，且永不丢弃已选内容 ✓
- ← 可逐级返回 ✓
- 面包屑每一级可点击回跳 ✓
- 暂存区每个 token 可单独移除 ✓
- 输入框中的 token 可 Backspace 删除 ✓
- Loop ref 不可 drill 但仍可选择 ✓

**唯一的小建议 P3-1：Enter 确认后没有 undo。** 用户按 Enter 提交了 3 个 token，Picker 关闭——如果有一个选错了，只能在输入框里逐个 Backspace 删除。考虑在 Enter 确认后显示一个短暂的 toast："已添加 3 项引用 · 撤销"，点击撤销回到 Picker 的之前状态。

---

### H4: 一致性与标准 — 评分: 3/5

**问题 P4-1（重要）：Enter 在不同层级做不同的事。**

| 你在哪 | 暂存状态 | Enter 做什么 |
|---|---|---|
| 分类项上 | 空 | Drill In（进入） |
| 分类项上 | 有暂存 | 提交暂存（不进入！） |
| 实体项上 | 空 | 选择+关闭 |
| 实体项上 | 有暂存 | 添加+全部提交+关闭 |

Enter 的行为取决于两个隐藏变量（当前层级 × 暂存状态）。用户按下 Enter 前需要同时考虑"我在哪"和"暂存里有什么"——这违反了一致性原则。

特别危险的场景：用户 Space 添加了 2 个工具，然后 ← 回到分类层想进入另一个分类继续选——但 ta 习惯性按 Enter "进入"，结果不是进入而是提交了暂存。

建议（二选一）：
- **方案 A：Enter 始终做"当前高亮项的默认动作"**——在分类上永远是 Drill In，在实体上永远是 Select & Close。暂存的提交**只能通过一个显式的"确认"按钮或快捷键**（如 Ctrl+Enter）。
- **方案 B：保持现在的设计，但在提示条中明确显示 Enter 的当前含义。** 当暂存有项时，Enter 的提示变成 `⏎ 确认(3)` 而非 `⏎ 进入`——当前已经做了，但需要确保用户真的会看提示条。

**问题 P4-2：`@` 有 Space 多选但 `/` 没有。** 这个不一致本身可能是合理的（一条消息通常只有一个命令），但用户可能会尝试在 `/` 模式下按 Space——什么都不会发生，没有任何反馈。

建议：`/` 模式下按 Space，显示一个轻量提示 "每条消息仅支持一个命令"。

---

### H5: 错误预防 — 评分: 4/5

**合规点：**
- Loop ref 被标记 ⟲ 且不可 drill ✓
- 搜索无结果有明确提示 ✓
- 已添加项重复 Space 时 toggle 移除（而非报错） ✓
- Enter 在已选项上去重 ✓

**问题 P5-1：用户可能在错误的上下文中触发 Picker。** 如果用户在输入 email 地址（hello@example.com），`@` 会误触发 Picker。PRD 提到了"@前无空白则不触发"的规则，但这个边界检测在 contenteditable 中实现可能不完全可靠。

建议：增加 300ms 延迟——如果 `@` 后立即有持续的快速输入（连续字符无停顿），延迟 Picker 出现，等用户停顿后再判断。

---

### H6: 识别而非回忆 — 评分: 4/5

**合规点：**
- Smart Landing 的"最近访问"减少了记忆负担 ✓
- Entity icon 三层体系帮助视觉识别 ✓
- Token pill 显示路径（hover 显示完整路径） ✓
- 搜索结果显示分类和父级路径 ✓

**问题 P6-1：暂存区的 token 是 compact 模式，可能难以识别。** 当暂存了 5 个以上的项，compact pill 挤在一起——用户需要逐个阅读才能确认"我选了什么"。

建议：暂存区超过 3 项时，显示分类汇总："2 菜谱 · 1 工具 · 2 学生"，hover 展开完整列表。

---

### H7: 使用的灵活性和效率 — 评分: 5/5

设计在这条上表现优异，核心的 power user 加速器都有：

- 搜索直接跨层级定位 ✓
- Space 多选避免反复 @ 触发 ✓
- ← 快速返回 ✓
- 面包屑跳级导航 ✓
- 鼠标的 + 按钮和 n › 按钮提供替代路径 ✓

---

### H8: 美学与极简设计 — 评分: 4/5

**问题 P8-1：Smart Landing 的三个区域可能信息过载。** 上下文 + 最近访问 + 全部分类，在 Picker 有限的高度里同时呈现。如果最近访问有 4 项 + 5 个分类，用户一打开就面对 10 行信息。

建议：测试一个更激进的简化版——Smart Landing 只显示上下文 + 最近访问，"全部分类"折叠在一个"浏览更多…"入口里。80% 的引用可能通过最近访问就能解决。

---

### H9: 帮助用户识别、诊断和修复错误 — 评分: 4/5

**问题 P9-1：Token 引用的实体如果被删除（后端变化），Token pill 会怎样？** PRD 提到 re-validate，但没有设计错误态的视觉表现。

建议：失效的 Token 变为红色虚线边框 + strikethrough 名称 + tooltip "该资源已不存在"。

---

### H10: 帮助与文档 — 评分: 3/5

**问题 P10-1：快捷键提示条是唯一的学习路径。** 没有 onboarding 引导、没有 tooltip、没有"首次使用"教程。提示条本身也很小——10px 字体在快速浏览时很容易被忽略。

建议：首次打开 @ Picker 时，显示一个一次性的 3 步引导叠加层："↑↓ 浏览 → Space 添加多个 → Enter 确认"，第二次使用时消失。

## 2. Don Norman — 概念模型 / 映射 / 反馈分析

### 2.1 概念模型审查

**用户的概念模型（教师视角）：**
> "我在写给 AI 的消息。我想告诉 AI 用哪些材料来帮我做事。"

**系统的实现模型：**
> "用户构建一条包含 EntityRef[] + SkillCommand + 自然语言的结构化消息，发送后由 Agent 解析执行。"

**Gap 分析：** 教师不会想"我在构建一个带有引用的结构化消息"——ta 想的是"我在和 AI 说话，顺便指给它看一些东西"。这个 gap 其实不大，因为 @ mention 的隐喻已经被社交媒体广泛建立。

但有一个**概念模型的断裂点**：**暂存区 + 批量提交**。在用户的心智中，`@张小明` 应该是"我提到了张小明"——一说出来就生效了。但暂存区引入了一个中间态——"我准备提到张小明但还没提到"。这个概念在日常对话中不存在。

**Norman 的诊断：** 暂存区可能是一个 **"系统模型泄漏到用户模型"** 的案例。工程上需要一个暂存区来支持多选，但用户不需要知道暂存区的存在。

**建议：** 考虑一个替代方案——**每次 Space 都直接在输入框中插入一个 Token**（而不是先放到暂存区），Picker 保持打开。用户在输入框里可以实时看到自己选了什么。这样就没有"暂存"的概念了——每次选择都即时生效，只是 Picker 没有关闭而已。代价是输入框会在 Picker 打开时不断变化，可能造成视觉干扰。

### 2.2 映射（Mapping）审查

| 物理按键 | 映射的语义 | Natural Mapping? |
|---|---|---|
| ↑↓ | 列表中上下移动 | ✅ 完全自然 |
| → | 进入子级 | ⚠️ 部分自然（文件管理器有此习惯，但浏览器中 → 通常是光标右移） |
| ← | 返回上级 | ⚠️ 同上 |
| Enter | 确认 | ✅ 自然 |
| Space | 添加但不关闭 | ❌ **非自然映射**——Space 在几乎所有场景中是"输入空格"或"翻页"，从未被用来表示"添加到集合" |
| Esc | 退出 | ✅ 自然 |
| `/` | 触发命令 | ✅ 有先例（Slack, Claude Code, Notion） |
| `@` | 触发引用 | ✅ 有先例（所有社交媒体） |

**Space 映射是最大的 natural mapping 问题。** Norman 会指出：当 Picker 打开时，用户按 Space 的本能期望是"输入一个空格到搜索框"。你把它重载为"添加"——这需要用户重新学习一个违反直觉的映射。

这就是为什么 **+ 按钮作为鼠标替代路径如此重要**——它是 Space 的可视化 affordance（可供性）。但如果一个操作需要一个可视化 affordance 来解释它的键盘版本，那说明键盘映射本身不够 natural。

**Norman 的建议方向：** 如果 Space 的映射无法改变（因为其他键更不合适），那就要把 + 按钮做得足够显眼，让用户先通过鼠标学习"添加"的概念，然后在提示条中发现 Space 是它的快捷键——学习路径应该是 **视觉 → 键盘**，而不是**键盘 → 理解**。

### 2.3 反馈（Feedback）审查

| 操作 | 反馈 | 充分? |
|---|---|---|
| 键入 @ | Picker 弹出 + 输入框边框变蓝 | ✅ |
| ↑↓ 移动 | 高亮项变化 + 左侧色条 | ✅ |
| Space 添加 | ✓ 徽章 + 暂存区出现 | ⚠️ 首次添加的反馈不够突出 |
| Enter 确认 | Picker 关闭 + Token 出现在输入框 | ✅ |
| Esc 回 Landing | 视图切回 Landing | ⚠️ 没有过渡动画，跳变感强 |
| → Drill In | 视图切换到子项 + 面包屑更新 | ✅ |
| Loop ref 按 → | 什么都不发生 | ❌ **静默失败**，应该有视觉反馈（如 shake 动画） |

**最需要改进的反馈：Loop ref 上按 → 的静默失败。** Norman 的设计原则之一是"所有操作都必须有可感知的反馈"。当用户在 ⟲ loop ref 上按 →，什么都不发生——用户不知道是系统没响应还是按键没被捕获。建议：短暂 shake 动画 + tooltip "已在路径中，无法展开"。

### 2.4 海湾模型（Gulf of Execution / Evaluation）

**场景：教师想引用第三课教案里那段小组讨论的视频片段。**

**执行海湾（Gulf of Execution）：** 从意图到操作需要多少认知转换？

```
意图："找到那个小组讨论的视频"
认知步骤：
1. 我需要用 @ 来找东西（需要知道这个规则）
2. 视频在教案里→课堂记录里（需要知道信息架构）
3. 键入 @，看到 Landing
4. 找到教案分类 → 进入（→ 或 Enter）
5. 找到第三课教案 → 进入（→）
6. 看到自身内容和关联引用 → 理解 ↗ 是跳转
7. 进入课堂记录 → 找到视频片段
8. 按 Enter 选择
```

**总共 8 步，其中 3 步需要预先知识。** Norman 会认为步骤 2 和步骤 6 的认知负荷最高——用户需要知道信息是怎么组织的，以及 ↗ 符号的含义。

**评估海湾（Gulf of Evaluation）：** 操作完成后，用户能否确认"我做对了"？

Token 显示 `[📋 …课堂记录 › 视频片段: 小组讨论]`——这个反馈是充分的。hover 显示完整路径——更充分了。**评估海湾很窄，这是好的。**

**建议：** 缩短执行海湾的关键优化是 **搜索**。如果教师直接键入 `@小组讨论`，搜索结果应该能直接显示这个视频片段，跳过步骤 3-7。当前搜索已经支持跨层级——需要确保搜索性能和结果排序足够好，让它成为主路径而不是备用路径。

## 3. Jef Raskin — 习惯化 / GOMS 击键分析

### 3.1 习惯化（Habituation）审查

Raskin 的核心原则：**一个已经习惯化的动作如果在某个状态下产生不同的结果，会导致用户犯错且无法察觉（因为习惯化操作绕过了意识）。**

审查你的设计中所有"同一按键不同状态做不同事"的情况：

| 按键 | 状态 A 的行为 | 状态 B 的行为 | 习惯化风险 |
|---|---|---|---|
| **Enter** | 分类上 → Drill In | 暂存有项时分类上 → 提交 | 🔴 **高风险**：已养成 Enter=进入 的习惯后，暂存态下会误触提交 |
| **Enter** | 实体上 → Select & Close | 已添加的实体上 → 提交暂存 | 🟡 中风险：行为相似但不完全一致 |
| **Esc** | 深层 → 回 Landing | Landing → 关闭 | 🟢 低风险：符合"逐级退出"的通用心智模型 |
| **Space** | Picker 关闭时 → 输入空格 | Picker 打开时 → 添加到暂存 | 🔴 **高风险**：Space 的习惯化行为是输入空格，重载后用户在搜索框中输入时会误触添加 |
| **→** | 正常输入时 → 光标右移 | Picker 打开时 → Drill In | 🟡 中风险：Picker 拦截了输入，但用户的肌肉记忆可能期望光标移动 |

**Raskin 的判定：Enter 的状态依赖和 Space 的重载是两个习惯化陷阱。**

**Enter 的修复建议（Raskin 视角）：** 让 Enter 的行为**不依赖暂存状态**。Enter 永远 = "对当前高亮项执行默认动作"（分类→进入，实体→选择）。暂存区的提交用一个**独立的、不会被习惯化操作误触的按键**，比如 `Ctrl+Enter` 或底栏的"确认"按钮。

**Space 的修复建议（Raskin 视角）：** 如果用户在 Picker 打开时搜索，键入文字应优先于 Space 的"添加"语义。即：如果搜索框有焦点，Space = 输入空格。只有当搜索框为空或无焦点时，Space = 添加。

### 3.2 GOMS 击键分析

**场景 1：单选——引用一个 Skill**
`@ → ↓ Skills → → 进入 → ↓↓ smart-questions → Enter`

```
操作序列：
K(@)           — 键入 @，Picker 打开           0.2s
M              — 心理准备：扫视 Landing          1.2s
K(↓)           — 跳过上下文区，到"浏览全部"      0.2s
K(↓)           — 移到 Skills 分类 (取决于排序)   0.2s  
K(→)           — 进入 Skills                    0.2s
M              — 扫视列表找到目标                0.8s
K(↓)K(↓)       — 移到 smart-questions           0.4s
K(Enter)       — 选择并关闭                     0.2s
────────────────────────────────────────────────
总计: 3.4 秒, 7 次击键, 2 次心理操作
```

**场景 2：搜索——直接搜索引用**
`@smart → Enter`

```
K(@)           — 触发 Picker                    0.2s
K(s)K(m)K(a)K(r)K(t) — 键入 "smart"            1.0s
M              — 确认搜索结果正确                0.6s
K(Enter)       — 选择并关闭                     0.2s
────────────────────────────────────────────────
总计: 2.0 秒, 7 次击键, 1 次心理操作
```

**场景 3：多选——引用 3 个工具**
`@ → Recipes → Space(鱼香肉丝) → Space(红烧牛腩) → ← → 班级 → → Space(三年一班) → Enter`

```
K(@)           — 触发                           0.2s
M              — 找到 Recipes                    0.8s
K(→)           — 进入 Recipes                   0.2s
M              — 找到鱼香肉丝                    0.5s
K(Space)       — 添加鱼香肉丝                    0.2s
K(↓)           — 移到红烧牛腩                    0.2s
K(Space)       — 添加红烧牛腩                    0.2s
K(←)           — 回到分类                       0.2s
M              — 找到班级                        0.5s
K(↓)           — 移到班级                       0.2s
K(→)           — 进入班级                       0.2s
M              — 找到三年一班                    0.5s
K(Space)       — 添加三年一班                    0.2s
K(Enter)       — 确认全部                       0.2s
────────────────────────────────────────────────
总计: 4.3 秒, 11 次击键, 4 次心理操作
```

**场景 4：深层 ref 导航——引用课堂视频片段**
`@ → 教案 → 第三课教案 → → ↗课堂记录 → → 视频片段 → Enter`

```
K(@)           — 触发                           0.2s
M              — 找到教案                        0.8s
K(↓)...        — 导航到教案分类                  0.4s
K(→)           — 进入                           0.2s
M              — 找到第三课教案                   0.5s
K(→)           — 进入子项                       0.2s
M              — 理解"自身内容"vs"关联引用"       1.5s  ← 首次最慢
K(↓)...        — 移到 ↗ 课堂记录                0.4s
K(→)           — 进入 ref                       0.2s
M              — 找到视频片段                    0.5s
K(↓)           — 移到视频片段                    0.2s
K(Enter)       — 选择                           0.2s
────────────────────────────────────────────────
总计: 5.3 秒, 10 次击键, 4 次心理操作（首次使用约 7 秒）
```

**GOMS 结论：**

| 场景 | 击键 | 时间 | 心理操作 | 评价 |
|---|---|---|---|---|
| 单选（导航） | 7 | 3.4s | 2 | 可接受 |
| 单选（搜索） | 7 | 2.0s | 1 | ✅ 快 |
| 多选（跨分类） | 11 | 4.3s | 4 | 可接受但心理操作偏多 |
| 深层 ref | 10 | 5.3s | 4 | ⚠️ 首次使用需 7 秒 |

**关键发现：** 搜索路径（场景 2）始终是最快的。如果搜索的排序和匹配质量足够好，应该把它推为**主路径**，让导航成为备用。Raskin 会建议：`@` 触发后，**搜索框应该自动聚焦且是最显眼的 UI 元素**——当前 Smart Landing 把搜索隐含在"继续键入"的行为中，不够显式。

## 4. Alan Cooper — 目标导向设计 / Persona 审查

### 4.1 Persona 具体化

PRD 中的三个角色过于抽象。Cooper 会要求具体到一个人：

**Persona A — 王老师（永久中间用户）**
- 45 岁，教数学 20 年
- 用微信和钉钉，不用 Slack
- 不知道什么是 Token、Pill、contenteditable
- 会打字但不快，经常看键盘
- 目标：备课时引用教案和课堂记录让 AI 帮忙出试卷
- 使用频率：每周 2-3 次

**Persona B — 李工（Power User）**
- 28 岁，解决方案工程师
- 日常用 VS Code、Linear、Slack
- 熟悉 @ mention 和 / 命令
- 打字快，从不看键盘
- 目标：快速组合 Tool + Template 做客户演示
- 使用频率：每天 5-10 次

**Persona C — 张局（管理者/只看不用）**
- 50 岁，教育局信息化处长
- 只在演示时看到这个系统
- 不会自己操作，但需要看懂效果
- 目标：确认系统能做到"精准教学"
- 使用频率：每月 1 次

### 4.2 Persona 与设计的适配分析

| 设计特征 | 王老师 | 李工 | 张局 |
|---|---|---|---|
| @ 触发 | ⚠️ 需要教 | ✅ 自然 | ❌ 不会 |
| Space 多选 | ❌ 不会发现 | ✅ 高效 | ❌ 不相关 |
| → 导航 | ⚠️ 需要教 | ✅ 自然 | ❌ 不会 |
| ref 跨实体跳转 | ❌ 概念太复杂 | ✅ 理解 | ❌ 不理解 |
| 搜索 | ✅ 自然 | ✅ 自然 | ⚠️ 能理解 |
| + 按钮（鼠标） | ✅ 自然 | 不需要 | ⚠️ 能理解 |
| Smart Landing | ✅ "最近访问"有帮助 | ✅ 效率提升 | ⚠️ 信息太多 |

**Cooper 的结论：**

王老师（永久中间用户）的主要路径应该是 **搜索 + 鼠标点击**，而不是键盘导航。当前设计的键盘优先策略完美服务李工，但对王老师来说，键盘操作是障碍而不是加速器。

**Cooper 的建议：** 为王老师设计一个**"简单模式"入口**——一个输入框旁边的 📎 按钮（或类似），点击后打开一个全屏的、更像文件选择器的界面（大按钮、大图标、无键盘快捷键依赖）。@ 键盘触发仍然存在，服务李工。两种入口，同一个 Picker 引擎。

### 4.3 "永久中间用户"分析

Cooper 定义的永久中间用户（Perpetual Intermediates）是使用频率中等、已经过了新手期但永远不会成为专家的人。王老师就是典型。

对于这类用户，Cooper 建议：
- **不要隐藏功能，但也不要前置所有功能。** Smart Landing 的做法（最近访问在上面）是对的。
- **提供渐进式披露。** 首次使用只展示搜索 + 点击，第三次使用时在提示条中 hint "试试 Space 键可以选多个"。
- **永远提供鼠标路径。** 当前的 + 按钮和 n › 按钮做到了。

## 5. Steve Krug — "Don't Make Me Think" 模拟走查

模拟一个 5 秒测试。想象把当前原型放在一个从未见过的教师面前：

### 5.1 第一眼（0-2 秒）

教师看到一个 chat 界面，输入框 placeholder 写着"键入 @ 引用资源，/ 触发技能..."。

**Krug 的问题：** "引用资源"是什么意思？"触发技能"是什么意思？这两个概念对教师来说都是程序员术语。

**改写建议：** "输入 @ 找到教案、学生等，输入 / 使用 AI 功能"。

### 5.2 触发 Picker（2-5 秒）

教师键入 @（假设已被教过），Smart Landing 出现。

**教师看到什么？**
```
★ 当前上下文
  🍳 鱼香肉丝（？？？这里为什么有菜谱？）

◷ 最近访问
  📋 第三课教案
  👤 张小明
  🔧 playwright-mcp（？？？这是什么？）

📁 浏览全部
  🍳 Recipes 菜谱
  🏫 班级
  ...
```

**Krug 会指出的问题：**
1. "playwright-mcp" 出现在教师的最近访问里不现实（假设数据），但如果是真实数据，任何教师看不懂的条目都会产生疑惑。
2. 分类标签混用中英文（"Recipes 菜谱"），对中国教师来说英文部分是噪音。
3. Landing 的三个区域没有足够的视觉分隔——教师可能不清楚"浏览全部"是一个独立区域还是最近访问的延续。

### 5.3 选择一个项目（5-15 秒）

教师想引用"第三课教案"，在最近访问里看到了它。

**鼠标用户：** 直接点击 → Token 插入 → 完成。✅ 非常顺畅。

**键盘用户：** ↓ 移到第三课教案 → Enter → 完成。✅ 也顺畅。

**但如果教师想选教案里的某个具体章节呢？**

教师看到 `第三课教案` 行右侧有一个小小的 `2 ›`。

**Krug 的判断：** 90% 的教师不会注意到 `2 ›`，或者即使注意到也不知道它是可点击的。数字 + 箭头的含义（"这个教案有 2 个子项，点击展开"）需要**习得**，不是自解释的。

**改写建议：** 把 `2 ›` 改成更明确的文案，比如 `▸ 展开` 或者在 hover 时显示 tooltip "查看教案的具体章节"。

### 5.4 Krug 的 "Trunk Test"（主干道测试）

在任何层级，用户应该能回答这 5 个问题：
1. 我在什么网站/产品里？→ Chatbox 的头部有产品名 ✅
2. 我在哪个页面/层级？→ 面包屑显示 `@ > 📋 教案 > 第三课教案` ✅
3. 我有哪些主要选项？→ 列表项 ✅
4. 我在导航中的位置？→ 面包屑 + Entity Banner ✅
5. 我怎么搜索？→ ⚠️ **不明显**。没有显式的搜索框——用户需要知道"直接打字就是搜索"。

**问题 P-Krug-1：搜索的可发现性不足。** 当 Picker 打开时，没有任何视觉元素告诉用户"你可以在这里搜索"。搜索是 Picker 最快的路径（GOMS 分析证明了这一点），但它是最隐蔽的功能。

**建议：** 在 Picker 顶部（面包屑行或独立行）放一个轻量级搜索框占位符（灰色文字"键入搜索..."），用户在任何层级都能看到它，并明白可以直接打字搜索。

## 6. Bruce Tognazzini — 交互设计第一原则审查

### 6.1 可探索性（Explorable Interfaces）

**Tog 的原则：** 用户应该能安全探索，不怕犯错。

当前设计：
- Esc 永不丢弃已选 ✅
- ← 可逐级回退 ✅
- 暂存区的每个项可单独移除 ✅

**但缺少 Undo。** 用户 Enter 确认后，没有撤销路径。这在所有竞品中都一样（Slack 的 @ mention 也没有 undo），但 Tog 会指出这是一个可改进的地方——特别是当用户误触 Enter 提交了错误的暂存组合时。

### 6.2 Fitts 定律审查

Fitts 定律：点击目标的时间与距离成正比、与大小成反比。

| 操作区域 | 位置 | 大小 | 问题 |
|---|---|---|---|
| + 按钮 | 行左侧 | 20x20px | ⚠️ 偏小，移动端几乎不可点 |
| n › 按钮 | 行右侧 | ~30x20px 文字 | ⚠️ 和行点击区域紧邻，容易误触——想点 › 但触发了 Select & Close |
| 行主体 | 中间 | 大面积 | ✅ 容易点击 |
| 暂存区 token ✕ | token 右侧 | 9px 字符 | ❌ **太小**，几乎无法精确点击 |
| 面包屑段 | 顶部 | 文字大小 | ⚠️ 偏小 |

**最需要修复的：暂存区 token 的 ✕ 按钮。** 9px 的文字作为点击目标，Fitts 定律预测这会导致大量误点。

**建议：**
- token ✕ 的点击热区扩大到至少 24x24px（视觉上可以保持小）
- n › 按钮和行主体之间增加一个透明的 padding 隔离带（8px），避免点 › 时误触行
- + 按钮在移动端放大到 32x32px

### 6.3 模态性（Modality）

**Tog 的原则：** 避免模态，如果必须使用模态，确保模态的进入和退出明确可见。

Picker 打开时，输入框从"自由输入模式"切换到"Picker 导航模式"——**键盘事件被拦截**（↑↓ 不再移动光标，而是移动高亮；Space 不再输入空格，而是添加项目）。这是一个**隐式模态切换**。

**Tog 会问：** 用户怎么知道自己当前处于"Picker 模式"？

当前的信号：
- Picker 浮层出现 ✅
- 输入框边框变色 ✅

**但缺少一个关键信号：** 输入框的 placeholder 或内联提示应该变化。从"键入 @ 引用资源..."变成"↑↓ 导航 · 键入搜索"，明确告知用户当前的键盘行为已经改变。

## 7. Indi Young — 心智模型 Gap 分析

### 7.1 模拟教师心智模型

基于教育场景的典型教师工作流，构建教师脑中的信息组织方式：

```
教师的心智模型:
┌──────────────────────────────────────────┐
│ 我的课                                    │
│   ├── 三年一班                            │
│   │   ├── 学生们                          │
│   │   ├── 上周的课（第三课）               │
│   │   │   ├── 我讲了什么（教案）            │
│   │   │   ├── 课上发生了什么（记录/视频）    │
│   │   │   └── 作业情况                    │
│   │   └── 下周的课                        │
│   └── 三年二班                            │
│       └── ...                            │
└──────────────────────────────────────────┘
```

### 7.2 系统信息架构

```
系统的架构:
@ 
├── 教案（Category）
│   └── 第三课教案（Entity）
│       └── ↗ 课堂记录（Ref）
├── 班级（Category）
│   └── 三年一班（Entity）
│       └── 学生们（Children）
├── 学生（Category）
│   └── 张小明（Entity）
└── ...
```

### 7.3 Gap 分析

| 教师想的 | 系统怎么组织的 | Gap |
|---|---|---|
| "三年一班上周的课" | 教案和班级在不同分类 | 🔴 **教师按"班级→时间"组织，系统按"类型"组织** |
| "课上的视频" | 视频是教案的 ref 的子项 | 🔴 **教师觉得视频是"那堂课的"，系统需要 教案→ref→子项 三次跳转** |
| "张小明的成绩" | 从班级或学生分类进入 | 🟡 **多路径设计缓解了这个问题** |
| "帮我出试卷" | / 命令 + @ 引用教案 | 🟡 **组合操作需要学习** |

**最大的 Gap：教师的心智模型是"以课为中心"的树状结构，系统的信息架构是"以类型为中心"的扁平分类。**

教师想的是"三年一班 → 上周的课 → 视频"，但系统里 "三年一班"在班级分类，"上周的课"在教案分类，"视频"在教案的 ref 的子项里。教师需要**先理解系统的分类逻辑**，才能找到视频。

**Indi Young 的建议：**

1. **"我的课"视图作为 Smart Landing 的核心。** 不是按"菜谱/教案/班级"分类，而是按教师的工作流分：`本周的课 > 上周的课 > 我的班级`。每堂课聚合了教案 + 记录 + 视频 + 作业——这才是教师的心智模型。

2. **或者，让搜索成为主路径。** 如果信息架构的 gap 太大无法调整，就用搜索绕过架构——教师输入 `@视频` 或 `@小组讨论`，系统直接返回结果，不需要教师理解分类结构。

3. **ref 的概念对教师来说是透明的应该。** 教师不应该知道"课堂记录是教案的 ref"——ta 只需要知道"打开教案后能看到相关的课堂记录和视频"。当前设计的 `◆ 自身内容 / ◇ 关联引用` 分组暴露了系统实现细节。考虑改为更自然的分组：`📝 教案内容 / 📹 相关录像 / 👥 相关学生`。

## 综合诊断：最高优先级问题

| 优先级 | 问题 | 来源 | 建议 |
|---|---|---|---|
| **P0** | Enter 在分类上的行为取决于暂存状态（有暂存=提交，无暂存=进入） | Nielsen H4, Raskin 习惯化 | Enter 始终按高亮项类型执行默认动作，暂存提交用 Ctrl+Enter 或显式按钮 |
| **P0** | Space 重载与"输入空格"的习惯化冲突 | Raskin, Norman Mapping | 搜索状态下 Space 优先输入空格；列表聚焦时 Space = 添加；增加 + 按钮的视觉权重作为主学习路径 |
| **P1** | 信息架构按"类型分类"而非教师心智模型的"以课为中心" | Indi Young | Smart Landing 增加"以课/时间为中心"的视图选项 |
| **P1** | 搜索是最快路径但可发现性最差 | GOMS, Krug | Picker 顶部增加显式搜索框占位符 |
| **P1** | 首次使用无 onboarding | Nielsen H10, Krug | 首次打开显示 3 步叠加引导 |
| **P2** | 暂存区的概念模型泄漏 | Norman 概念模型 | 考虑"Space 直接插入 Token + Picker 不关闭"的替代方案 |
| **P2** | 暂存区 token ✕ 按钮太小 | Tog Fitts | 热区扩大到 24x24px |
| **P2** | Loop ref 上按 → 静默失败 | Norman 反馈 | 短暂 shake + tooltip |
| **P2** | `n ›` 和行点击区域误触 | Tog Fitts | 增加 8px 隔离 padding |
| **P3** | `◆ 自身内容 / ◇ 关联引用` 分组标签暴露系统实现 | Indi Young | 改为面向用户语义的标签 |
