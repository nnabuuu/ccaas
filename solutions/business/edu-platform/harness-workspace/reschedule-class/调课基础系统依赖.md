# 调课功能 · 基础系统依赖分析

> 调课是一个"看起来简单但依赖链很长"的功能。本文梳理它需要哪些基础系统能力，哪些已有，哪些需要建设。

---

## 依赖全景

```
调课功能
  │
  ├─ [1] Chat 融合入口基础能力
  │    ├─ @ picker + 工具栏             ← Context Layer 模块
  │    ├─ 结构化表单弹出                ← 前端组件
  │    └─ Chat ↔ 表单上下文传递         ← 前端状态管理
  │
  ├─ [2] MCP 网关
  │    ├─ 外部教务系统 MCP 接入         ← CCaaS MCP 管理
  │    ├─ MCP 认证 + 权限              ← CCaaS 安全层
  │    └─ MCP 调用超时 + 重试          ← CCaaS 基础设施
  │
  ├─ [3] 结构化输出渲染
  │    ├─ 确认卡片组件                  ← 前端组件
  │    ├─ 方案推荐卡片                  ← 前端组件
  │    └─ 状态通知卡片                  ← 前端组件
  │
  ├─ [4] 审批流
  │    ├─ 审批状态机                    ← Solution 业务 或 外部教务系统
  │    └─ 审批结果回调                  ← Webhook / 轮询
  │
  ├─ [5] 通知系统
  │    ├─ 教师端推送                    ← 平台通知服务
  │    └─ Chat 内状态更新              ← SSE / WebSocket
  │
  └─ [6] 实体追踪
       ├─ 调课申请 @Referenceable       ← Context Layer
       └─ 活动记录                      ← Context Layer Activity
```

## [1] Chat 融合入口基础能力

### 已有（Context Layer 设计中）

| 能力 | 状态 | 说明 |
|------|------|------|
| @ picker | 已设计 | entity-types + suggest + search + browse + resolve |
| 工具栏 pills | 已设计 | session template 配置 shortcuts |
| 实体引用 pill | 已设计 | 选中后显示在输入框上方 |

### 需要新增

**工具栏动作类型扩展：** 当前工具栏 pill 只支持"触发 @ picker 并跳到某个实体类型"。调课需要一种新的动作类型——"弹出结构化表单"。

```typescript
// 当前 toolbar shortcut 只有一种 action
type ShortcutAction = 'open_at_picker_with_type';  // 打开 @ picker 并钻入某类型

// 需要扩展为
type ShortcutAction = 
  | { type: 'at_picker'; entityType: string }      // 📝教案 → @ picker 钻入教案
  | { type: 'form'; formId: string }                // 📅调课 → 弹出结构化表单
  | { type: 'prefill_chat'; text: string }          // 预填 Chat 输入框
  | { type: 'action'; actionId: string };           // 触发自定义动作
```

**表单 ↔ Chat 上下文传递：** 教师在表单中填了一半切换到 Chat 时，已填字段需要作为上下文注入。这不是后端功能——是前端状态管理。

```typescript
// 前端需要的能力
interface FormToChatBridge {
  // 表单切 Chat 时调用
  switchToChat(formState: Partial<RescheduleForm>): void;
  // → 关闭表单
  // → 将 formState 序列化为自然语言预填到输入框
  // → 如 { type: '代课', original: { day: 3, period: 5 }, reason: '教研' }
  //   → "我需要调课：周三第5节因为教研活动，想找人代课，帮我想方案？"
}
```

## [2] MCP 网关

### 需要的基础能力

调课功能的所有数据操作都通过教务系统 MCP 完成。需要 CCaaS 的 MCP 管理具备：

**a) MCP 连接管理**

```typescript
// Solution 注册外部 MCP
interface McpConnection {
  id: string;
  name: string;                    // "成都区教务管理系统"
  type: 'sse' | 'stdio' | 'http';
  endpoint: string;                // MCP server URL
  auth: {
    type: 'api_key' | 'oauth2' | 'basic';
    credentials: string;           // 加密存储
  };
  healthCheck: {
    interval: number;              // 健康检查间隔
    timeout: number;               // 超时
  };
  tools: string[];                 // 该 MCP 提供的 tool 列表（发现或手动配置）
}
```

**b) 权限控制**

不同角色能调用不同的 MCP tools：

```typescript
// Session template 中的 MCP 权限
mcpPermissions: {
  'timetable:query-schedule': ['teacher', 'admin'],      // 所有人可查
  'timetable:find-available-slots': ['teacher', 'admin'], // 所有人可查
  'timetable:check-conflicts': ['teacher', 'admin'],      // 所有人可查
  'timetable:submit-request': ['teacher', 'admin'],        // 所有人可提交
  'timetable:list-my-requests': ['teacher', 'admin'],      // 只能查自己的
}
```

**c) 调用链路**

```
Chat AI → Jijian Agentic Engine → CCaaS MCP Gateway → 教务系统 MCP Server
                                      │
                                      ├─ 认证注入（API key / OAuth token）
                                      ├─ 超时控制（默认 10s，可配置）
                                      ├─ 重试策略（幂等操作可重试）
                                      ├─ 日志记录（审计追踪）
                                      └─ 错误转换（MCP 错误 → 用户友好提示）
```

## [3] 结构化输出渲染（Chat 内卡片组件）

### 问题

当前 Chat 界面只有文本消息。调课功能需要在 Chat 中渲染结构化卡片：

| 卡片类型 | 用途 | 交互 |
|----------|------|------|
| 方案推荐卡片 | AI 推荐 1-3 个调课方案 | 每个方案可点击展开详情 |
| 确认卡片 | 提交前的最终确认 | [提交] / [再调整] 按钮 |
| 状态卡片 | 提交成功 / 审批通过 | 申请号 + 状态标签 |
| 联动建议卡片 | 调课后的后续操作 | [处理] / [生成] 按钮 |

### 需要的前端能力

```typescript
// Chat 消息需要支持 "structured content" 类型
interface ChatMessage {
  role: 'user' | 'assistant';
  content: TextContent | StructuredContent;
}

interface StructuredContent {
  type: 'card';
  cardType: string;               // 'reschedule-confirm' | 'reschedule-status' | 'action-suggestion'
  data: Record<string, any>;      // 卡片数据，按 OutputSchema 结构化
  actions?: CardAction[];         // 卡片上的按钮
}

interface CardAction {
  label: string;
  actionType: 'submit_mcp' | 'prefill_chat' | 'navigate' | 'at_reference';
  payload: Record<string, any>;
}
```

**这不是调课独有的需求——它是 Chat 融合入口的通用能力。** 作业批改、学情分析、教案推荐都需要在 Chat 中渲染结构化卡片。调课只是第一个需要"带操作按钮的卡片"的场景。

### 卡片渲染方案

```
方案 A：前端硬编码卡片组件
  每种 cardType 对应一个 React 组件
  优点：UI 精细控制
  缺点：新增卡片类型需要前端发版

方案 B：Schema 驱动的通用卡片渲染器
  OutputSchema 的 displayConfig 定义布局
  前端有一个通用渲染器按 schema 渲染
  优点：新增卡片类型不需要前端改代码
  缺点：布局灵活性受限

方案 C：混合（推荐）
  通用卡片类型（info / confirm / status）用 schema 驱动
  复杂交互卡片（确认提交 / 多方案选择）前端硬编码组件
  新 Solution 可以先用通用类型，特殊需求再定制组件
```

## [4] 审批流

### 场景

调课申请提交后需要教务处审批。审批可能在教务系统中完成（外部），也可能在我们的 Solution 中完成。

### 两种模式

```
模式 A：审批在外部教务系统
  submit-request → 返回 requestId + status='pending'
  教务系统内部审批
  审批完成 → 教务系统回调我们的 webhook
  我们更新 RescheduleRequest 状态 + 推送通知

模式 B：审批在 Solution 内
  submit-request → 只是记录变更，不直接调教务系统
  Solution 内部审批流（管理员在管理工作台审批）
  审批通过 → Solution 调 timetable:execute-changes 执行

成都项目的情况：可能是模式 A（教务系统已有审批流）
```

### 需要的基础能力

```typescript
// 不管哪种模式，都需要：

// 1. 状态变更监听
interface StatusWatcher {
  // 轮询模式（教务系统没有 webhook 时）
  poll(requestId: string, interval: number): Observable<RequestStatus>;
  // Webhook 模式
  onWebhook(requestId: string, callback: (status: RequestStatus) => void): void;
}

// 2. 状态变更 → Chat 推送
interface ChatNotification {
  // 向教师的 Chat session 推送一条结构化消息
  pushToSession(sessionId: string, message: StructuredContent): void;
  // 如果教师不在 session 中 → 存储待推送，下次进入时显示
}
```

## [5] 通知系统

调课涉及多方通知：

| 事件 | 通知谁 | 通道 |
|------|--------|------|
| 申请提交 | 审批人（教务主任） | 系统通知 / 消息 |
| 审批通过 | 发起人 + 互换教师 + 代课教师 | Chat 推送 + 系统通知 |
| 审批驳回 | 发起人 | Chat 推送 |
| 互换确认 | 对方教师 | 系统通知（需对方确认） |

### 需要的基础能力

```typescript
interface NotificationService {
  // 发送通知
  send(params: {
    recipientIds: string[];
    channel: 'chat_push' | 'system_notification' | 'sms';
    title: string;
    body: string;
    structuredContent?: StructuredContent;  // Chat 推送时带结构化卡片
    actionUrl?: string;                     // 点击跳转
  }): Promise<void>;
}
```

## [6] 实体追踪（Context Layer）

```typescript

  displayName: '调课申请',
  icon: '📅',
  contextFields: ['type', 'changes', 'status', 'reason', 'approver'],

@Controller('reschedule-requests')
export class RescheduleRequestController {
  @Get('search') search() { ... }
  @Get() list() { ... }
  @Get(':id') findOne() { ... }
  @Post() create() { ... }          // auto-track: 'created'
  @Patch(':id/status') updateStatus() { ... }  // auto-track: 'updated'

```

Context Layer 模块已经设计好，这里不需要额外建设——只需要在 Solution 中注册实体。

## 总结：建设优先级

### P0 — 没有就做不了调课

| 系统 | 说明 | 工作量 | 谁做 |
|------|------|--------|------|
| MCP 网关基础 | 外部 MCP 注册 + 认证注入 + 调用链路 | 中 | CCaaS core |
| 工具栏动作扩展 | 支持 form / prefill_chat 类型 | 小 | 前端 |
| 结构化卡片渲染 | Chat 中渲染确认卡片 + 状态卡片 | 中 | 前端 |

### P1 — 没有会降级但能用

| 系统 | 说明 | 降级方案 | 工作量 |
|------|------|---------|--------|
| 审批回调 | 审批结果实时推送到 Chat | 降级：教师手动刷新 / 轮询 | 小-中 |
| 通知服务 | 多方通知（互换教师、代课教师） | 降级：教师自己沟通 | 中 |
| 表单 ↔ Chat 桥接 | 表单数据序列化为 Chat 预填 | 降级：教师重新描述 | 小 |

### P2 — 锦上添花

| 系统 | 说明 |
|------|------|
| AI 推荐优化 | 基于历史调课数据优化推荐（谁经常互换、谁代课评价高） |
| 调课后联动 | 自动触发教案时间线调整建议 |
| 批量调课优化 | 全局最优解（而不是贪心逐个匹配） |

## 与其他融合入口功能的共性需求

调课暴露出的基础系统需求，大部分不是调课独有的：

| 基础能力 | 调课需要 | 其他功能也需要 |
|----------|---------|---------------|
| 工具栏动作扩展（form 类型） | ✓ | 作业发布、成绩录入 |
| Chat 内结构化卡片 | ✓ | 学情分析结果、作业批改报告、AI 推荐 |
| 外部 MCP 接入 | ✓ | 考勤系统、成绩系统、家校通 |
| 审批/状态回调 | ✓ | 请假审批、活动审批 |
| 多方通知 | ✓ | 家校通知、教研活动通知 |
| 操作确认卡片（带按钮） | ✓ | 任何"AI 建议 → 人确认 → 执行"的场景 |

**建设这些基础能力的优先级应该高于调课本身——它们是平台级能力。**
