Decision Archive · @kedge-agentic/ontology v0.1 · 2026-05-30

Phase 1 上线 @kedge-agentic/ontology v0.1 —— Zod 优先的契约层,Tier 2/3 字段在编译期屏蔽。

What. Phase 1 上线了 @kedge-agentic/ontology v0.1.0 包 —— 一个 framework-free 的 TypeScript 包, 提供 ObjectTypeDef / ActionDef / FunctionDef / ManifestDef / AccessBoundary / OntologyRegistry 这套描述业务对象 + 操作 + 权限的原语, 加 Anthropic-tools、MCP、system-prompt 三种 LLM 投影格式,加 SHA-256 内容寻址的 schema digest。 Zod schema 是字段形状的唯一来源,PropertyMeta sidecar 只承载 Zod 表达不了的元数据 (searchable / displayRole / computed)。

Why. ccaas 之前没有统一的契约层来回答"这个 Solution 有哪些业务对象 + Agent 能做什么 + 谁能看到 / 写入什么 + 怎么把这些投影给 Agent" —— 每个 Solution(recipe-book、live-lesson)各自造一套表达。借鉴 Palantir Ontology 的形状, 建一个跨 Solution 的契约层,让 schema 既是设计文档、也是运行时检查、还是 Agent 的可见性投影源。 Phase 1 是 5 阶段 rollout 的第一阶段 —— 只 ship 核心原语 + Tier 1 gap items (FunctionDef、ActionDef.preconditions、LocalizedString), Tier 2/3 留待真实 Solution 触发再做。

Scope. In:8 个 subpath 导出(schema、manifest、accessor、registry、helpers、semantic、distribution、根 barrel), 12 个 Tier 1 ValidationCode,两阶段注册流程(local-eager + cross-def deferred),三种投影格式, framework-free 由架构测试强制(0 NestJS / React / Vue import,0 跨包 import)。 Out:NestJS 桥接 + ManifestAccessor 实现(Phase 3)、 context-layer 改造(Phase 2)、Tier 2 原语 (InterfaceDef / ObjectSetDef / BoundaryPredicate,Phase 4)、 Tier 3 原语(validationRules / StateMachineDef / classification / NotificationRule / ActionDef.returnType,Phase 5)。

D-01

问: 字段形状写在自定义 PropertyDef 里,还是 Zod schema 里?

Design · Schema
触发 设计细化中途,用户直接打回:"我感觉很多定义是不是太细了,如果我们默认就是使用 TypeScript 的话..." 约束 Palantir 风格的 PropertyDef 每个字段都要写 type, required, min, max, pattern, enum。TypeScript + Zod 已经把这些全部表达完了。再做一遍 = 两份 source of truth,迟早会 drift。
Option方案结论
A 保留自定义 PropertyDef 作为 canonical 形状。Runtime 每个字段同时持有 Zod 和 PropertyDef,代码层保证一致。 ❌ 拒 —— 两份真相,必然 drift。
B Zod schema 是字段形状的唯一来源。PropertyMeta sidecar 只承载 Zod 表达不了的:searchable / displayRole / computed / displayName i18n。 ✓ 选 —— 一份真相,一套类型系统。
为什么这么选

一个 TypeScript-native 包的本意,就是不再发明一套平行的类型系统。 Zod 已经能表达 discriminated union、refinement、branded type、optional/nullable —— Palantir 当年要造 PropertyDef,是因为他们没有 Zod。我们有。

Sidecar 模式把真正非 Zod 的元数据(picker 提示、治理标记)隔离到一个很小的接触面。 PropertyMetaMap<S> 用 keyof z.infer<S> 做 key 约束, meta key 拼错就是编译期错误,直接在调用点报。TypeScript 每次都检的东西,你 drift 不了。

Locked in
Solution 写字段就是 z.object({...})。每字段的语义文本走 .describe()。再也没有平行的 "字段类型" 类型。
Locked out
针对非 TS runtime 的 Solution 不能直接复用 Zod schema —— 它们消费的是 distribution/serialize 投影出来的 JSON Schema。

D-02

问: 一个根 barrel,还是 subpath exports?

Design · Package
触发 commit 1 设计 package.json 的 exports map。impl plan 原本写的是单一根 re-export。 约束 兄弟包 @kedge-agentic/agent-runtime 用的是 subpath exports(/workspace / /artifact / /sync)。跨包一致性对认知成本很关键。
Option形状结论
A 单一根 @kedge-agentic/ontology re-export 所有东西。package.json 更简单。 ❌ 拒 —— 跟兄弟包不一致;tree-shaking 弱;import 范围大。
B Subpath exports:/schema、/manifest、/accessor、/registry、/helpers、/semantic、/distribution。根 catch-all 仍然存在。 ✓ 选 —— 跟 agent-runtime 对齐。
为什么这么选

跟 framework-free 兄弟包的一致性,值得 exports map 多写几行。 两个包定了一个 convention,第三个不跟才是异常。

实务收益:tree-shaking 更准(consumer 只 import 它要的, import { ObjectTypeDef } from '@kedge-agentic/ontology/schema' 不会把 projection 代码拖进来)。 测试可以单 subpath 隔离跑。根 catch-all 仍然在,所以这个改动对 consumer 是严格 additive。

Locked in
每个新模块都有自己的 subpath。架构测试(__tests__/architecture.test.ts)强制每个 subpath 都 framework-free。
Locked out
subpath 改名就是 breaking change。今天 ship 的 7 个路径已经是 API 面。

D-03

问: Registry 注册时强制依赖顺序吗?

Runtime · Registry
触发 实现 OntologyRegistry 时遇到:所有 cross-def validator(link / slot / precondition 的引用解析)都要求被引用的 def 已经注册。强行要求顺序 = 注册顺序敏感。 约束 Solution 作者通常先把全套 ontology 写完再注册。逼他们按依赖图拓扑排序很烦。但完全 batch 又会丢掉本地错误(空 semantic、payload 互斥)的及时反馈。
Option方案结论
A 严格顺序 —— register* 跑完整 validation 含 cross-def;引用没解析就抛。caller 自己排序。 ❌ 拒 —— 把拓扑排序的负担压给 caller。
B 两阶段。register* 只跑 LOCAL validation(semantic、meta key、payload)。validate() / seal() 跑 cross-def。幂等。 ✓ 选 —— 保住本地反馈环 + 允许乱序注册。
C 只暴露 registerBatch;不提供单个 register*。 ❌ 拒 —— 杀掉增量开发的体感。
为什么这么选

两阶段在自然接缝处切开 validation —— 本地能查的保持 eager(写错就在调用点报), 需要更广上下文的延迟到一次显式的 seal()。写坏的 def(空 semantic)永远进不了 registry; 引用缺失在启动时作为一个聚合错误报出。

seal() 作为一个独立动词,给 caller 一个"我写完了,锁上"的信号 —— flag 一翻,后续的 register* 直接抛。原子性的保证(seal 先调 validate,通过后才置 flag) 意味着 seal 失败时 registry 仍然可变,可以修完再 seal,不用重启。

Locked in
任何 ontology bootstrap 都是两阶段:注册全部 → 启动时 seal() 一次。
Locked out
预期"注册了就一定有效"的 consumer 会被坑 —— register 和 seal 之间 registry 可能处于 cross-def 无效状态。

D-04

问: Phase 4/5 的字段怎么禁止 Phase 1 的人用?

Contract · Deferral
触发 Phase 1 明确推迟 Tier 2(Interface、ObjectSet、BoundaryPredicate)和 Tier 3(validationRules、stateMachine、classification、NotificationRule、returnType)。如果 consumer 不小心写 ObjectTypeDef.implements: [...],什么挡得住? 约束 Phase 1 现在就要 ship;Phase 4/5 可能几个月之后。这个"推迟"必须由 toolchain 强制,不能靠 reviewer 警觉。
Option方案结论
A Runtime feature flag —— 字段在 type 上存在,Phase 1 注册时如果被 set 就抛。 ❌ 拒 —— Runtime 才在启动时报错,不是写代码时报。
B 字段直接从接口里 omit 掉。set 它就是 TypeScript 编译错。在 tests 里用 @ts-expect-error 钉住覆盖。 ✓ 选 —— 最强约束,在调用点失败。
C README 里写不要用;type 层放过。 ❌ 拒 —— 靠记性。
为什么这么选

对一个还没 ship 的 feature 来说,编译期屏蔽是最强的约束 —— 你的 IDE 把字段名标红了, 推迟就是真的。Runtime error 只有跑代码才触发;review 纪律只能抓 reviewer 看见的东西。

tests 里的 @ts-expect-error gate 还有第二个作用: Phase 4 给 ObjectTypeDef 加 implements 的时候,那行 expect-error 会失败成 "unused expectation" —— 这是给未来实现者的结构化交接信号。(这一点差点失效 —— code review 发现 tsc 默认把测试文件排除了; 靠加一个独立的 tsconfig.test.json 进 typecheck pipeline 补上。)

Locked in
加任何 Phase 4/5 字段都是可发现的变更 —— 各处的 @ts-expect-error 在自己 typecheck 时浮出来。
Locked out
反序列化进来的 def(比如从 JSON)绕过 TS —— runtime validator 必须覆盖同样的不变式。Validator 套件已经覆盖,但 parity 需要记得。

D-05

问: Distribution 依赖 Registry,还是反过来?

Runtime · Layering
触发 在 registry 上实现 getSchemaDigest()。真正算 digest 要用 distribution 层的 serializeRegistry + computeSchemaDigest。 约束 Distribution 函数需要遍历所有注册的 def(横切访问)。直接 distribution → registry import 在 registry 反向 import 时会循环。但让 getSchemaDigest 永久 stub 又别扭。
Option方案结论
A 保持 getSchemaDigest stub。Consumer 外部组合:computeSchemaDigest(serializeRegistry(registry))。 ❌ 拒 —— consumer 用着别扭;registry 自己的 digest 方法应该能用。
B Distribution 函数接收 ValidationContext(Map 形状)而不是 OntologyRegistry。Registry 暴露 context();registry → distribution 是干净的单向 import。 ✓ 选 —— 无循环,distribution 跟 registry 解耦。
为什么这么选

ValidationContext 本来就存在,作为 validator 的最小 cross-def 查询面 —— 三个 ReadonlyMap(object types、manifests、functions)。 Distribution 想要同样的面不是巧合:两边都要 "遍历所有 def", 两边都不需要 register / validate / seal 机制。

方向锁定后,distribution 可以脱离 registry 单独测(手搓一个 Map, 直接调 serializeRegistry(context))。Registry 的 getSchemaDigest() 变成一行,把组合藏在方法后面。层只往下依赖,不往侧面依赖。

Locked in
未来任何 "遍历所有 def" 的特性(export、diff、search)都接收 ValidationContext,不接收 OntologyRegistry。
Locked out
需要 registry 的 seal/validate 状态的特性不能住在 distribution —— 得另起家。

D-06

问: Serialize 是 round-trip codec 还是单向投影?

Design · Distribution
触发 设计 distribution 层 —— 需要一个稳定的 digest 输入。zod-to-json-schema 是天然的投影,但是有损的(refinement、transform、branded type 元数据都不留存)。 约束 Phase 1 的 use case —— 缓存键、wire 传输、diff 工具 —— 只需要单方向。Round-trip 需要一套并行 codec + 它自己的 validation surface,永久维护。
Option方案结论
A 无损 codec —— 设计一个能 round-trip 回 Zod(含所有 refinement、brand)的 JSON 形状。 ❌ 拒 —— 接触面大,当前无 consumer,drift 风险。
B 单向投影,走 zod-to-json-schema。递归 key sort 用 canonicalize。canonical bytes 上跑 SHA-256。 ✓ 选 —— 最小;贴合 Phase 1;复杂度推迟。
为什么这么选

Phase 1 对 serialized 形式的每个使用 —— digest 计算、给 peer 看有哪些字段的 wire 传输、diff 工具 —— 都是只读的。没有一个需要从 JSON 重建一个 live Zod schema。 建那个能力就是在给一个不存在的 consumer 做设计。

有损投影对自己的边界很诚实:"这是可以分享的;这不是源 schema 的替代品。" 等真的来了需要重建的 consumer(浏览器端 validation、schema-as-data 的 CLI), 那是另一个独立的 codec 包,有它自己的维护契约,不是塞进 distribution。

Locked in
Digest 是 "这个 ontology 形状" 的 canonical 身份。同样的 def 不同顺序 → 同样的 digest,由 14 个 distribution 测试验证。
Locked out
Consumer 没法从 serialized 形态重建 Zod schema。真要的时候,是个新 codec 包 —— opt-in 成本。

D-07

问: checkBoundary 从哪拿 precondition 评估需要的 state?

Security · Boundary
触发 实现 checkBoundary —— spec 说它要评估 Tier 1 precondition(stateEquals、slotBound),但这些需要 runtime state(当前 phase、当前绑定的 slot)。spec 从来没说 state 从哪来。 约束 checkBoundary 应该是唯一一个纯函数安全 gate,被每个 ManifestAccessor 实现调用。把 precondition 评估拆到第二个函数 = 有两个 gate 要 audit。
Option方案结论
A checkBoundary 忽略 precondition;caller(ManifestAccessor 实现)走另一个函数评估。 ❌ 拒 —— 两个 gate 要审 security。
B BoundaryCheckInput 可选携带 state + boundSlots 快照。precondition 在同一次调用里评估。缺上下文时 fail-safe 拒绝。 ✓ 选 —— 单 gate;结构上 fail-safe。
为什么这么选

一个单一 audit 面来回答 "这个操作能不能做",远比签名的小别扭重要得多。 两个 gate 存在时,不只是更容易误调一个漏一个 —— 未来代码在系统长大时更容易漏掉其中一个。

可选 state 的设计默认是安全的:caller 忘了传 state context, stateEquals/slotBound precondition 评估为未满足,操作被拒。最坏情况:一个合法操作被拒, 开发者去查为什么;永远不会反过来 —— 一个被拒操作意外被放过去。 第三种 precondition named 是 Phase 1 stub,永远返回未满足 —— 同样的 fail-safe 原则,等 Phase 4 ship predicate registry。

Locked in
每个 ManifestAccessor 实现(Phase 3+)都要把 state + boundSlots 快照塞进 BoundaryCheckInput。Pattern 写在文档里;测试覆盖四种 op kind + precondition 组合。
Locked out
需要更丰富上下文的 precondition(比如跨 manifest 的 state)需要新的 precondition kind。stateEquals/slotBound 形状之内,这个设计是 final 的。

OQ-01

Phase 2 的 context-layer 重构会暴露 Phase 1 的 API 缺口吗?

问题Phase 1 API 是从 spec 设计的 —— 不是从真实 consumer 集成里出来的。OntologyRegistry 的方法集、ManifestAccessor 的 interface、projectManifest 的格式形状,纸面上都合理。没有一个被真实消费验证过。

OQ-02

Digest 在浏览器里怎么工作?

问题computeSchemaDigest 用的是 node:crypto(刻意 —— 保持包依赖轻)。浏览器侧消费要换 SubtleCrypto。

为什么 open今天没有浏览器 consumer 在用 digest。真出现的时候计划:同名 export 走兄弟文件 browser-digest.ts,通过 package.json "browser" 字段条件化。但这是纸面设计,等真实 consumer 出现才落实。

OQ-03

Named-precondition registry 长什么样?

问题Phase 1 ship 了 ActionPrecondition 的 'named' kind 作为永远返回未满足的 stub。意图(spec §3.3)是 Phase 4 ship OntologyRegistry.registerPredicate 跟 BoundaryPredicate 一起。形状:(state, slotContext, params) => boolean?支持 async 吗?跟 checkBoundary 的同步签名怎么集成?

为什么 open形状决定要等 Phase 4 的 BoundaryPredicate 设计(predicate-scoped AccessBoundary)先定下来 —— 它俩是兄弟概念。两个都推迟到 Phase 4。
