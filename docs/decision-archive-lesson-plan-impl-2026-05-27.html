问: Plan Tab 用什么编辑器框架?

Frontend · Editor
触发 设计文档 §8 open question #1 列了三个候选 (TipTap / BlockNote / Lexical),"暂不决定"。实现到 track 5 时必须选一个。 约束 需要支持自定义 inline atom node (req:// chip) + 自定义 block node (HTML 注释保留) + roundtrip stable 的 markdown ↔ JSON 桥。
OptionApproachVerdict
A TipTap v3 (StarterKit + 自定义 Node 扩展) ✓ 选 —— 成熟、可扩展、社区例子多
B BlockNote (现成 Notion-feel, 不开放底层) ❌ 拒 —— 自定义 chip node 困难
C Lexical / 自建 ❌ 拒 —— 工作量过大, 我们要的是富文本不是搜索引擎
为什么这么选

这个决定本来是"两难选一", 但 chip 需求把选项剪掉了 —— BlockNote 的高级 API 不给我们足够口子去定义自定义 inline atom node (chip 是 React-rendered + has-data-attrs + non-editable, 这套组合在 BlockNote 里只能 hack 进去)。 TipTap 走 ProseMirror 的原生扩展机制, 我们写 Node.create({...}) + ReactNodeViewRenderer 就拿到完全控制权。

代价: TipTap "Notion-feel" 不是 out-of-box, 我们自己写了 toolbar + slash command pattern + chip picker modal. 大概 1500 行 React. 但每一行都明确知道在做什么, 比 hack-BlockNote 调试要省时间。

Locked in
ProseMirror schema 是我们的内部表达; bridge 在 tiptap-bridge.ts 集中维护
Locked out
换到 BlockNote 要重写 chip + HtmlBlockNode 扩展 (~500 行)

D-02

问: Chip 在 ProseMirror 里是什么节点类型?

Frontend · Node Type
触发 实现 ReferenceChipNode 时, TipTap 让我选 mark / inline non-atom / inline atom / block。每个都能 "工作", 但语义不一样, 影响光标行为 + 选区 + 撤销。 约束 Chip 内容 (text + title) 是 library 渲染出来的, 用户不应该字符级编辑 (避免误改 → canonicalize 又刷回); 但 chip 整体可以被选中 + 删除。
OptionApproachVerdict
A Inline mark (字符上的标记) ❌ 拒 —— 用户可以在 marked 文本上敲字, 半个 chip 内容混乱
B Inline atom node (atom: true, selectable: true) ✓ 选 —— 原子选区, 整体删, 不可字符级编辑
C Block-level node (独占一行) ❌ 拒 —— chip 要 inline (跟普通文本同一行), block 太重
为什么这么选

ProseMirror 的 atom 语义恰好是我们要的: 节点对编辑器来说是不透明的, 光标跳过, Backspace 整删, 选中是 "选整个 chip" 而不是 "光标在文字中间"。 这跟 markdown 语义吻合 —— [文本](req://id) 是一个不可分割的链接 token, 不是文本流的一部分。

代价: 不能在 chip 文字里直接编辑 (得通过 picker / interpretation 编辑器 modal)。但这正是设计要求 —— canonicalizer 会覆盖手改的文字, 不让用户直接编辑反而避免了"为什么我改的字消失了"的困惑。

Locked in
Chip 选区 / 删除 / 拷贝 行为依赖 ProseMirror atom 语义
Locked out
如果未来允许 chip 内嵌富文本 (粗体名字, 链接等), atom 模型不支持; 要回到 mark 形态

D-03

问: ccaas 怎么把 lib 物化进 workspace?

Architecture · Boundary
触发 第一版实现 (commit 9061e903) 在 ccaas-core 加了 SessionAssetMaterializer.materializeLessonPlanLib(...) + LESSON_PLAN_LIB_URLS env + lessonPlanSubject 字段到 Solution.config。用户审阅时直接打回: "ccaas agentRuntime 为什么会知道 solution 的内容?" 约束 L2 是 per-user, ccaas 必须告诉 solution 当前 user 是谁; 但 ccaas-core 不能知道 "lesson plan" / "teaching requirement" / "subject" 这种 solution-specific 概念。
OptionApproachVerdict
A 保留 ccaas 的 lesson-plan 知识, 仅道歉 + 文档化 ❌ 拒 —— 架构违反不是文档问题
B ccaas 引入 "sidecar" 概念 (新名词, 不带域名词) ❌ 拒 —— 加新概念解决可被现有概念解决的问题
C 全部回滚, 走现有 artifact-sync port + 加 userId context ✓ 选 —— ccaas 零域名词, solution 全权决定返回什么文件
为什么这么选

ccaas 的核心承诺是 domain-agnostic platform —— 它知道 sessions、workspaces、artifacts, 不知道任何 solution 是教育/医疗/金融。Option A 把那条线打破了; 哪怕只是一个方法名, "下一个 solution 来了" 就会发现没有现成的扩展位, 只能再加一组 domain-specific 代码, 复合性病。

真正的洞察: "per-user 文件"和"per-project 文件"在 ccaas 视角下没区别 —— 它都只是 "solution 通过 artifact endpoint 返回的内容"。ccaas 唯一需要加的是 "我告诉 solution 当前 caller 是谁", 也就是给 loadArtifacts(projectId, context?) 加一个可选的 {userId} 字段。Solution 自己决定怎么用这个 userId —— live-lesson 拿它去查 L2, 别的 solution 可以拿它做别的事或者完全忽略。

最终代码 delta: ccaas-core 加了 约 10 行 (context 参数 + 透传)。删了 commit 9061e903 加的约 200 行 (materializer 方法 + SSRF 防御 + env 解析)。负的代码量, 因为做对了。

Locked in
ccaas-core grep "lesson-plan"/"teaching-requirement"/"LESSON_PLAN" 零命中。架构边界检查工具可以纳入 CI。
Locked out
任何"在 ccaas 知道 solution 的什么"的捷径; 未来 solution 想要传入更多 context, 必须扩展 ArtifactCallContext 这个泛用接口 (而不是直接打补丁)

D-04

问: userId 怎么从 ccaas 传到 solution?

Runtime · Port
触发 D-03 定了"通过 artifact sync", 但具体接口形状要选: 给现有 port 加可选参数 vs 新增一个 endpoint vs 用 HTTP header layer 而不动 port 签名。 约束 现有 WorkspaceArtifactSource 接口在 npm 包 @kedge-agentic/agent-runtime, 改动是 type-level breaking change 候选; 但只有 ccaas + live-lesson 在用 (in-tree consumer only).
OptionApproachVerdict
A 扩展 port: loadArtifacts(projectId, context?: {userId?}) ✓ 选 —— 类型化 + 非破坏性 + 自文档
B 新 endpoint: loadArtifactsForUser(projectId, userId) ❌ 拒 —— 与现有 endpoint 双轨, 调用方要选哪条
C 不动 port 签名, 让 REST adapter 偷偷加 header ❌ 拒 —— 类型签名说谎; 非 REST 实现 (in-process 测试 mock) 拿不到
为什么这么选

可选第二参数是 TypeScript 接口扩展的规范做法: 现有 implementer 不用改 (TS 类型签名 backwards-compatible); 新 implementer 可以选择性地 opt-in; 类型签名告诉读者 "这条信息存在, 你可以读"。

关键设计: ArtifactCallContext 是命名对象不是裸 string 或 generic dict。未来添加 tenantId, requestId, locale 等只需要在这个接口加字段, 编辑器和类型检查会指出谁需要更新。比起 Record<string, string> 这种 "拍脑袋加 key 还能跑" 的做法更难误用。

代价: 改了 npm 包接口, 要 rebuild + 更新 in-tree consumer。但 in-tree only, 所以可控。

Locked in
ArtifactCallContext 是 ccaas → solution 上下文传递的 canonical 通道; 未来新字段都加在这
Locked out
不能再让 ccaas 通过裸 header / query 偷偷传 ccaas-specific 信息 —— 必须通过这个 typed 接口

D-05

问: 怎么防 _lib/*.md round-trip 污染?

Solution · Data Integrity
触发 Code reviewer 抓到一个 dangerous edge case: 既然 _lib/*.md 通过 artifact API 返回, SyncEngine 会把它当成普通 artifact。Agent 如果在 workspace 里 touch _lib/x.md, sync 会 plan save_db, 调 live-lesson upsertArtifact, 内容被持久化到 project_files。下次 listArtifacts 返回两份同 path 的内容, 永久污染。 约束 不想加 ccaas 概念 (违反 D-03 刚定的边界); 但又必须阻止 round-trip。
OptionApproachVerdict
A ccaas 加 "path namespace" 概念, 物化但不 sync-back ❌ 拒 —— 又一次域名词侵入 ccaas
B Solution 端 silent no-op on _lib/ writes ❌ 拒 —— SyncEngine 看不到 "写了但没存", snapshot 会刷不掉
C Solution 端拒绝 + 403 ForbiddenException, agent 看到错误 ✓ 选 —— 明显, snapshot 自动回退
为什么这么选

这是 "convention via reservation" 模式: solution 自己声明 _lib/ 是 platform-rendered 不可写, 同时在每一个会接收 path 的 API (upsertArtifact + deleteFile) 加一致的 guard。 ccaas 完全不知道 _lib/ 有什么特殊, 它就是个 path; reject 来自 solution 端, 用 standard HTTP 403 反馈。

额外保险: agent-runtime port JSDoc 加了一行约定 —— "solutions surfacing derived content should reject writes to those prefixes"。 这把 "convention" 提升到 "documented contract", 让别的 solution 实现端也能复用这个模式 (比如某个 solution 想暴露 _audit/ 给 agent 看, 同样写禁 _audit/ 写)。

Locked in
live-lesson 永远不能把 _lib/ 用作正常 project file 路径
Locked out
反向: agent 不能通过 artifact API 写解读 (得调 L2 API)。SyncEngine 看到 403 会把 snapshot rollback, 不再 retry → agent 看到自己的 fs 写消失。这是期望行为, 但需要 agent 知道 (skill 文档有写)

问: Lib 文件的 workspace path: artifacts/_lib/ 还是 workspace-root _lib/?

Solution · Path Convention
触发 设计文档 §4.1 写的是 workspace-root _lib/teaching-requirements.md。但 D-03 之后决定走 artifact-sync, 而 artifact-sync 把所有路径物化到 <workspace>/artifacts/<path>。 约束 要么改 ccaas (加新物化目录, 域名词侵入), 要么改 path 约定 (脚本 / SKILL.md / 设计文档要 sync)。
OptionApproachVerdict
A 坚持设计文档原文 (_lib/ at workspace root), ccaas 加 "sidecar mount" 概念 ❌ 拒 —— 再次往 ccaas 注域名词

Locked out

Deferred 这次显式没做的

5 项 · 每条带 revisit trigger
DEF-01

Per-project subject 配置 (取代 env 单一 subject)

现状Live-lesson 通过 LIVE_LESSON_LESSON_PLAN_SUBJECT env 锁定整个 backend 一个 subject。同一个 backend 不能同时跑 English + Math 项目的 lib materialization。

推迟理由MVP 单一学科是常态; 加 per-project 列要 schema migration + UI 改。等真有跨学科需求再做, 避免 over-engineering。

重新审视的触发第一个用户报告"我的英语和数学项目需要不同的库"。届时加 course_projects.subject 列, 读 column 优先于 env fallback。

DEF-02

saveArtifact / deleteArtifact 也接收 userId

现状只有 loadArtifacts 接收 ArtifactCallContext。写操作没有 user 归属。

推迟理由当前没用例需要 "agent 写文件时记录是哪个 user 写的"; 写操作走 project-scoped, user 由 session 隐式归属。YAGNI。

重新审视的触发需要做 audit log "user X 在 session Y 改了 file Z" 时, 把 context 加到 saveArtifact 签名。non-breaking 同样能做到。

DEF-03

HMAC stamping on X-Caller-User-Id

现状live-lesson backend 信任入站 X-Caller-User-Id header。如果端口 3007 被非 ccaas 进程访问, 任何 client 可以伪造 header 冒充任何 user。

推迟理由单机 dev + 内网部署下风险低, 网络层 (反向代理 / firewall) 就够; HMAC 引入密钥分发 + 时间同步问题。

重新审视的触发部署到非内网 / 多 solution backend 共享 ccaas 时。这是设计文档 §8 open question #6 已标记。

DEF-04

Subject 从 manifest 自动推断

现状Subject 来自 env, 不读 execution/manifest.json 里的 subject 字段。

推迟理由manifest 是 artifact, 在 lib materialization 之前还没物化进 workspace; chicken-and-egg。要么 server-side 预读 manifest (耦合), 要么 lazy materialize after first artifact sync (复杂)。先用 env 配置规避。

重新审视的触发有 backend 实例需要同时跑多 subject 项目 → DEF-01 落地时一起做。

DEF-05

ccaas-side _materialize endpoint 退役

现状live-lesson 还保留 GET /api/teaching-requirements/_materialize endpoint。第一版实现 (commit 9061e903) 用过它, revert 后没有调用方了。endpoint 留着作为 debug 入口。

推迟理由留 ad-hoc debug 入口对生产无害; 没有 caller 也没有维护成本; 完全删除等真的确认无用再说。

重新审视的触发下次 live-lesson backend 改造时清理。或者发现真的有人在调它 (audit log)。

Open Questions 真正还没答案的

3 项 · 没 confident answer
OQ-01

ccaas 该不该有正式的 "sidecar path namespace" 概念?

问题现在 _lib/ 是 solution-side 的 convention + reserved-prefix guard。可行但脆: 每个 solution 都要记得自己的 reserved 前缀, 写 guard。如果 ccaas 提供一个 "sidecar mount" 概念 (固定路径, 不参与 sync-back), 是不是更结构化?

为什么 open没有第二个 use case 验证这个抽象的必要性。一个 case 不算 pattern, 三个 case 才算。等出现第二个 (e.g. demo-sandbox 想暴露 _logs/ 给 agent) 再判断要不要抽象。

OQ-02

Env-based "default user" 模型能撑到 production?

问题LIVE_LESSON_DEFAULT_USER_ID 是 MVP 单用户模式。production 需要真实多用户, 但 live-lesson 还没有 auth 模型。

为什么 openauth 模型设计是 cross-cutting 工作, 不只影响 lib materialization。等 live-lesson 自己有用户登录系统再说; 或者 ccaas 把 session.userId 来源做得更可信 (DEF-03 HMAC), 让 live-lesson 直接信任。

OQ-03

SyncEngine 应不应该感知 reserved prefix?

问题现在 reserved-prefix guard 在 solution 端 (HTTP 403)。但 SyncEngine 看到 403 后会 retry / 把 snapshot rollback, 在某些 edge case (agent 一直在写 _lib/) 可能死循环。如果 SyncEngine 知道某些路径不该 sync-back, 可以更优雅。

为什么 open这又会回到 D-03 的边界问题 —— ccaas 知道哪些路径是 "sidecar" 等于知道 solution 的内部约定。可能的折中: solution 通过 ArtifactCallContext 反向声明 "这些前缀别 sync-back", 但需要双向接口扩展。等真出问题再做。
