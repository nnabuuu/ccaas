D-02

问: 怎么渲染 audit 报告 markdown?

Frontend · Renderer
触发 报告是 markdown + :::pass/warn/guess/error[标题] ... ::: callout directive。 标准 markdown 渲染器 (react-markdown / marked) 不识别 directive 语法。 约束 creator 现有依赖只有 unified + remark-parse (lesson-plan-md 在用), 没 react-markdown。 加新 dep vs 复用现成 dep vs 自己手写。
OptionApproachVerdict
A 加 react-markdown 依赖 + remark-directive 插件 ❌ 拒 —— 两个新 dep 只为一个 tab
B 预切 callout block + remark-parse mdast → 手 walk → React; 跟 PlanRenderer 同 pattern ✓ 选 —— 零新 dep, 与现有渲染层一致
C 完全自写 mini-parser (正则匹配 # / - / **) ❌ 拒 —— 边界 case 无穷; remark-parse 现成
为什么这么选

关键洞察: callout directive 跟 markdown 是两个 orthogonal 关注。Directive 在 line-level 上识别 (开头 :::sev[title], 结尾 :::), markdown 在 directive 之外 + 之内的内容上。 预切策略让两者解耦: splitIntoSegments 纯字符串处理 (无依赖); 每段 markdown 走 remark-parse (用现成 dep); Callout 组件包裹 directive body 内的 segment。

复用 unified 还有风格一致性 的好处。 creator 的 PlanRenderer 也是"remark-parse → walk mdast → emit React" 这条路。 两个 renderer 同 pattern, 后续维护(加新 mdast 节点支持)只学一次。 加 react-markdown 会引入第二种 markdown 处理范式, 团队学习成本翻倍。

代价: 手 walk mdast 写代码量比 react-markdown 多 50-100 行。 但这些代码是可见 + 可调 的 —— 当渲染出 bug 时 (LLM 输出某种格式我们没覆盖) 能直接读代码 + 改 case 分支; react-markdown 内部 plugin 链调试要绕。 对于"审计报告" 这种输出格式相对固定 (LLM 跟着固定 prompt 走) 的场景, 自写更可控。

Locked in
renderer 完全在 creator/ 控制范围内; 加新节点类型 (image / blockquote) 加一行 case
Locked out
支持完整 markdown spec (tables / gfm extensions) 要自己实现 walker; 不像 react-markdown 装个 plugin 就有

问: 纯函数 helper 放 .tsx 还是单独 .ts?

Test · Infra
触发 原本把 splitIntoSegments + stripBannerComment 写在 AuditReportRenderer.tsx 里 export 出去。 写 spec 时发现 vitest.config 是 environment: 'node' + include: '*.spec.ts' —— 从 .ts spec 文件 import .tsx 会把 React 拉进 node 测试 runtime。 约束 改 vitest config 加 jsdom + RTL 是另一个工作 (creator 没 React 测试基建)。 但又想要可测试的解析逻辑。
OptionApproachVerdict
A 加 jsdom + @testing-library/react, 写 component 测试 ❌ 拒 —— 设基建超出 stub scope
B 纯 helper 拆到 audit-markdown.ts, renderer 从那 import; spec 也 import 同一 .ts ✓ 选 —— 零新基建, 核心逻辑覆盖

D-04

问: Callout 解析要不要先归一化 CRLF?

Robustness · Defensive
触发 Code review HIGH: splitIntoSegments 只 split('\n'), 当输入是 CRLF 时每行尾巴挂一个 \r, callout open/close 正则 (anchored with $) 不匹配 → 每个 callout 静默 fall-through 为普通 markdown。 约束 LLM 输出 / Windows 客户端 / 某些 HTTP 中转可能引入 CRLF。不能依赖"输入总是 LF"。
OptionApproachVerdict
A 在 splitIntoSegments 顶上 .replace(/\r\n?/g, '\n'), 后续逻辑只见 LF ✓ 选 —— 单点处理
B 改正则: \r?$ 替代 $, 兼容 CRLF + LF ❌ 拒 —— 每个正则都要改, 容易漏
C 不修, 文档说明 "只支持 LF" ❌ 拒 —— 用户看到的失败是"我的报告渲染坏了", 不是"格式不对"
为什么这么选

这是 "输入 normalization 的位置" 经典问题: 在边界 (parser 入口) 做一次 vs 在每个细节 (各 regex) 处处理。 一次入口 normalize 是 "transform input to canonical form once, then everything downstream is simpler", 这是 robust 软件的常见做法 (compiler 前端的 lexer, web 服务器的 URL canonical 化)。

代价微乎其微: String.replace 在几 KB 报告上的开销纳秒级。 收益: 后面所有正则 / split / index 计算都假设 LF, 永远不用担心"我这个分支考虑 CRLF 了吗"。 一句话, 一个 mental model。

Code review 抓到这个的价值在于: 我自己写时根本没想到 CRLF, 因为开发机 + LF 测试输入都是 LF; 但 LLM 部署到 prod 时通过什么传输路径未知, 假设 LF 是脆弱的。 这是从已知未知到 actively defending 边界条件 的转变。

Locked in
所有 callout 解析无视行终止符差异; 测试包含 CRLF 用例锁契约
Locked out
想保留原 CR 字符 (e.g. 对接古早系统) 不行 —— normalize 一道闸丢了它

D-05

问: handleRun 用 closure 还是 activeProjectId ref?

Correctness · React Closures
触发 Code review HIGH: 原本 const pid = projectId 从闭包读, 如果用户在 render 跟点击之间切了项目, pid 是上一个 render 的 id, runAudit 会发到错的项目去。 约束 React 的闭包陷阱是经典问题。 useEffect / useCallback 重新绑定可以躲, 但 onClick handler 是函数引用, render 时绑定的 closure 就是那一刻的 props。
OptionApproachVerdict
A 保持闭包, 文档说明"切项目期间别点按钮" ❌ 拒 —— 用户不读文档, 这是 footgun
B 读 activeProjectId.current (已经被 useEffect 同步) ✓ 选 —— ref 一致性已存在
C 用 useCallback + 把 projectId 加进 deps, 重新绑定 handler ❌ 拒 —— 每次 projectId 变都重新绑函数, 不如读 ref
为什么这么选

关键观察: activeProjectId ref 已经为别的目的存在 —— fetchOnce 用它做 stale-project guard。 既然 ref 在那, handleRun 借用现成机制最便宜。 不引入第二种状态管理方式。

反思: 我自己写时把 const pid = projectId 视为"自然" —— 函数参数捕获嘛, 哪里有问题? 但 React 的 closure model 让"函数式上对" 跟"事件驱动 UI 上对" 经常错位。 onClick 是异步 触发, 跟 render 时的 props 解耦; ref 是用来跨 render 共享"当前状态" 的标准模式。 这次 fix 的本质是把"事件处理读最新状态" 写成 React idiom, 而不是依赖闭包恰好捕到对的值。

这跟 D-04 (CRLF) 一样, 是 code review 抓到的"代码看起来对但有 edge case 不对" 的情况。 单元测试覆盖不到 (没人 mock React 的 render 时序); 只能靠人眼或 lint 规则提醒。 把它显式写下并加 comment 让未来读者 (包括 future-self) 不踩同一坑。

Locked in
handleRun 永远拿当前 projectId, 不会被切换 race 出 wrong-project POST
Locked out
想加 "audit 触发时记录当时的 projectId 用来 callback 时校验" 需要捕获本地变量; 不能完全靠 ref

D-06

问: nav:// / action:// / ref:// 链接渲染成什么?

Frontend · Link Schemes
触发 Audit prompt 让 LLM 在 callout 里嵌 [让 AI 修复](action://fix?target=step-1) / [Step 1](nav://execution/step-1) 这种 link, 设计上准备给 v7-rich chat-bridge 用。 当前 v7-rich 框架不存在, 这些链接无法 wire。 约束 不能让链接看起来可点但点了无反应 (是 UX 灾难); 也不能完全藏掉 (会丢 LLM 表达出来的意图)。
OptionApproachVerdict
A 渲染成正常 <a>, 点了 navigate to URL —— 浏览器报"unknown protocol" ❌ 拒 —— 浏览器原生错
B inert: 灰色虚下划线 <span>, hover 提示 URL, 不可点 ✓ 选 —— 显示意图但不假装能用
C 完全藏掉 (drop link, 只渲染 text) ❌ 拒 —— 丢了 LLM 想表达"这条要去某地" 的信息
为什么这么选

关键设计原则: 视觉应该反映能力。 看起来是个普通链接 (蓝色下划线) 但点了没反应 = UI 在骗用户; 完全藏掉 = UI 在遮 信息。 Inert 渲染 (灰色, 虚线, 鼠标 hover 显示 title) 是 "I see what this is, I just can't act on it yet" 的诚实表达。

实际意义: 教师看到报告里"[让 AI 修复]" 灰色出现, 直觉理解"哦这是个未来会能点的按钮"; 没出 chat bridge 之前他可以手动切 chat 输入"按报告里说的改", 一样完成任务, 只是多一步。 v7-rich 落地后同一段 LLM 输出 就活了 —— 不用改 prompt, 不用改后端, 只换前端渲染。

这条决定让后端跟前端 v7-rich 完全解耦: prompt + LLM + 报告生成都按"最终能交互" 的设想写, 前端按"当前能力" 渲染。 两边可以独立演化, 不互相阻塞。

Locked in
后端 prompt + 报告内容当前已经向前兼容 v7-rich; 不需要切换时重新生成
Locked out
想让"让 AI 修复" 立刻能点要么 (a) wire chat-bridge (大重构), (b) 改报告生成 prompt 让 LLM 用其他形式 (背离 design)
