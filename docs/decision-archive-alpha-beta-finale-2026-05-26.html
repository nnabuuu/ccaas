D-01

问: 该不该把之前为兼容性保留的 22 处 alias 全删?

Process · Scope
触发 前几轮 α+β refactor 为了 "可能要做数据迁移" 留了 22 处 alias / load-bearing 字符串。我准备进入 "compat window 结束、跑迁移、再删 alias" 这个常规节奏。用户直接打断: "其实可以全改，还没有正式上线，以前的数据不要了"。 约束 每一处保留都有 原因: DB SkillScope 是持久化的 enum 值;POST /bind-project 是 solution 已经在用的 route;agent-runtime npm 包接口是外部消费者的契约。改这些有真实代价 —— 如果有用户。
OptionApproachVerdict
A 保留 22 处。继续走 "ship → compat window → migration → 删 alias" 节奏。安全、慢、多 commit。 ❌ 拒 —— 安全是为不存在的用户买保险。
B 一次删干净。22 处 alias + load-bearing 字符串同步改。一个大 commit。 ✓ 选 —— pre-prod 把所有"为兼容"的代价归零。
为什么这么选

"保留 alias 一个 release 让用户迁移" 这套节奏的前提是 有用户。 没用户的话，alias 不是 "缓冲"，是 永久的双重维护负担: 每次写新代码都要想 "用新名还是兼容旧名"，每个新人都要被解释 "为什么这个有两个名字"。

用户那一句话不是 "可以激进点"，是 "前提条件 (pre-prod, no data) 已经满足， 所以保留的代价从 0 变成净负"。这种翻转不需要新分析 —— 之前没改是因为代价 > 收益， 现在代价归零了，结果就反过来了。

Locked in
代码读起来一致，新人不用学双重词汇表。npm test 跑通即说明 rename 完整。
Locked out
任何持有 pre-rename 数据 / 客户端的人无法 talk to 新 ccaas。不能再回到 "全改" 之前的状态了。

D-02

问: "全改" 在哪儿停下?

Design · Boundary
触发 D-01 给了 "可以全改" 的授权后,出现一个新问题: 哪些是真的能改的,哪些是形式上看似 但其实改不动的? 约束 磁盘路径 tenants/{slug}/skills/ 是 agentfs daemon 跟 ccaas 之间的契约。solution.json 里 "tenant": { name, slug } 是每个 solution 实际 ship 的 wire 格式。CCAAS_API_KEY 是 ops 在 .env 里设的变量名。ADR / CHANGES 是历史记录。
OptionApproachVerdict
A 字面意义的"全改" —— 磁盘路径、wire 格式、env 变量、ADR 一起改。 ❌ 拒 —— 这些每个都有不同的"留下来"理由。
B 代码层面全改;每一个保留都附 明确理由 写在 deferred list 里。 ✓ 选 —— 区分 "懒得改" 和 "有理由不改"。
为什么这么选

"全改" 的意思不是 所有 'tenant' 字符串都变 'solution'，而是 所有 code-level 内部依赖都改。代码内部叫什么是项目自己说了算的事; 但跨进程契约(磁盘路径)、wire 格式(solution.json)、operator-facing 常量(env)、 历史档案(ADR)这四类属于跟其他系统 / 其他时间点 共享的命名 —— 改这些需要协调， 而协调 ROI 是 case-by-case 的。

这次的判断不是 "改不改"，是 "哪些边界改一处会牵连到边界外"。 磁盘路径牵连 agentfs daemon;wire 格式牵连每个 ship 出去的 solution.json; env 变量牵连每份部署文档;ADR 牵连历史叙述。这四类都 暂不改，每条都有 revisit trigger，没有变成隐式技术债。

Locked in
代码内部词汇统一为 solution / workspace。Code review 检查 rename 完整性有明确 ground truth。
Locked out
代码 solutionId 跟磁盘路径 tenants/ 这种不一致永久存在，直到 deferred 项 trigger。

D-03

问: Solution proxy 的 domain 翻译应该放在哪一层?

Design · Boundary
触发 ccaas-core 把 projectId 字段一律改成 sourceIdentity(generic、domain-agnostic)。但 live-lesson 真的有 projects —— browser UI 里那一栏文案是"项目"。如果 browser 也跟着改成 "sourceIdentity"，UI 语义就丢了。 约束 ccaas-core 跟 live-lesson 都是 in-tree、可以一起 ship。但 ccaas-core 应该不知道 "project" 是什么(domain-agnostic 是平台的核心承诺)，live-lesson 的 UI 必须知道(它就是 project editor)。
OptionApproachVerdict
A cascade rename: browser 也用 sourceIdentity。Backend 不做翻译。 ❌ 拒 —— UI 失去 domain 词汇,文案被技术词汇污染。
B live-lesson proxy controller 保留 {projectId} 作为浏览器面;翻译成 {sourceIdentity} 后再 forward 到 ccaas。 ✓ 选 —— 翻译放在 solution backend (boundary),domain 词汇留给 UI。
为什么这么选

Solution backend 本来就是 domain ↔ platform 的边界 —— 它是这条边界唯一同时知道两边词汇的层。 翻译放在这一层是定义，不是 ad-hoc 选择。Browser 只跟 solution 说话(用 domain 词)， ccaas 只跟 solution 说话(用 platform 词)。中间这一层是天然的翻译器。

反过来如果让 browser 直接用 ccaas 词汇，等于 让 UI 替平台说话: 以后 ccaas 平台词汇变了(比如 sourceIdentity 改成别的)，每个 solution 的 UI 都要跟着改。 翻译层抽离掉这个耦合 —— ccaas 词汇变了，只有 solution backend 的翻译器要改。

Locked in
每个新 solution 的 proxy controller 都要做这个翻译(每条 send-message 请求 + 每条 attach-workspace-source 请求)。
Locked out
如果未来需要 browser 直连 ccaas (零 backend)，翻译层无处安放。但 D-04 这条已经在用户的 "API key 不能进浏览器" 原则下封死了。

D-04

问: 拆 PR 还是一个大 commit?

Process · PR Shape
触发 Plan 文件原本把 final phase 拆成 β-1 (route + DTO) → β-2 (service internals) → β-3 (SSE controller + URL)。Sub-PR 化是给 reviewer 友好的常规节奏。 约束 A-D 几个阶段的改动是 compile-time interdependent —— interface 名改了的话所有 import 都要跟着改;DB enum 值改了的话所有读写都要跟着改。半改 = 编译不过。
OptionApproachVerdict
A β-1 → β-2 → β-3 拆成 3 个 PR,每个 ~2-5 天,reviewer 友好。 ❌ 拒 —— 每个 PR 单独 standalone 编译都不过,拆完更难 review。
B 一个 atomic commit (~87 文件)。Holistic review 在最后做。 ✓ 选 —— 半改不编译,拆 PR 是 fake split。
为什么这么选

"拆 PR 给 reviewer 友好" 这条建议预设了 PR 可以独立编译。 当代码改动是 type-level interdependent (interface rename + 所有 caller 必须同步改) 的时候， 所谓的 "拆" 只能拆成故意半残的中间状态，每个 PR 都跑不起来。这种拆法 对 reviewer 不友好，对 git bisect 有毒。

plan 文件的 β-1/β-2/β-3 拆法本来是为 compat-window 设计的 —— 那个场景下每个 PR 加新名 + 保留旧名, 可以独立编译。一旦 D-01 把 compat 干掉了，拆法的前提就消失了。所以拆 PR 不是 "保守做法"，是"应用错了的做法"。

Locked in
Review 一次性看全图;rename 完整性的 ground truth 是 tsc --noEmit + jest 都过。
Locked out
单 commit 87 文件 git bisect 时不能再细分;但 rename 类 commit 本身基本没有 bisectable 故障模式。

D-05

问: Code review 发现 must-fix 该当场修还是 follow-up?

Process · Review Gate
触发 大 commit staged 之后用户问 "code review done?"。我跑 reviewer agent，找到 3 个 must-fix: skill.dto.ts 的 @IsEnum(['tenant','personal']) 漏改(任何带 scope:'solution' 的 POST 会被 validator 拒绝)、RestProjectArtifactSource re-export 没删、admin guard 还在读 x-tenant-id header。 约束 这 3 个 bug 全是 tsc 过、jest 1555 都过 的情况下还存在的 —— test 覆盖盲区。其中 1 个会在 runtime 直接 break (DTO 拒绝所有合法请求)。
OptionApproachVerdict
A 先 push 主 commit,follow-up commit 修。快、main 进度看着好看。 ❌ 拒 —— main 上 broken 几小时,任何 pull 的人会撞到。
B 当场修,加测试,跑 jest,合到同一个 commit。 ✓ 选 —— review gate 的作用就是 "在落 main 之前"。
为什么这么选

Code review 找到的 must-fix 跟 "review 之后想到一个 follow-up improvement" 不是一个范畴的事。 前者是 "这个 commit 在它当前状态下不应该 ship" —— DTO 拒绝所有合法 POST 是 runtime 直接坏。把这个 push 出去等于故意往 main 放一个 known bad commit。

Follow-up 的合理性建立在 "新的需求 / 新的发现" 上。Review 找到的 must-fix 是 review 阶段就该被 block 的 bug。两个 commit 跟一个 commit 在 main history 上的差别很小， 但 "好的 main commit" vs "broken main + fix commit" 在语义上差很多 —— 后者表示 "我们 ship 了 broken 代码 + 然后修了"，前者表示 "ship 之前 review 过"。

Locked in
Pre-commit review gate 是真正的 gate (能让 commit 不发生)，不是 informational。
Locked out
Commit 的"时间-完整性"延后了几十分钟，但这正是 review 该花的时间。

D-06

问: SDK 的 tenantId 是这次一起改，还是 scope-out?

Design · Surface
触发 Code review 第二轮指出: ccaas-core X-Solution-Id header + solutionId 字段都改完了，但 react-sdk / vue-sdk 公共 API 还是 tenantId + X-Tenant-Id。这俩包是 ccaas 平台的 public 表面。 约束 SDK rename 不是 ccaas-core 内部改动，而是每个 SDK 消费者(live-lesson frontend / chat-interface / lesson-plan-designer / ...)的 props / 类型签名都要跟着改。blast radius 比 D-01 大。
OptionApproachVerdict
A SDK 这次不动。Backend 是 solution 词汇,SDK 仍是 tenant 词汇 —— 跨包词汇错位。 ❌ 拒 —— 永久性的双词汇债，每个新 SDK 用户要被解释一次。
B SDK + 所有 in-tree 消费者一起改 (~16 SDK 文件 + 3 consumer 文件)。一个 commit 内。 ✓ 选 —— blast 可控,跟 D-01 同理(pre-prod, 没出 ship 的消费者)。
为什么这么选

D-01 的 "全改" 决定不局限在 ccaas-core 一个 package。整个平台对外暴露的词汇 需要一致 —— 否则 SDK 文档跟 backend 文档会用不同的词，每个用户都要在 mental model 里维护翻译表。 这种 cross-package 词汇债 不会自己消失，只会越来越贵。

SDK 的 blast radius (~3 个 in-tree consumer) 是这次特别便宜的窗口 —— consumer 是 in-tree、能一起 ship。一旦有第一个 out-of-tree 消费者(未来某个 solution 用 SDK 但 repo 在别处)，rename 就会变成 breaking change，要走 deprecation。所以 "现在改" 是 "便宜窗口里改" 的最后机会。

Locked in
SDK + backend + UI 词汇统一。X-Solution-Id 是整个平台认的 header。
Locked out
任何当前不在 repo 里的 SDK 消费者(假设的)会被这次 breaking change 影响。但这个 set 当前是空。

02 —

Deferred 这次显式没做的

5 项 · 每条带 revisit trigger
DEF-01

BaseMaterializer 磁盘路径 tenants/{slug}/skills/

现状变量名是 solutionId，磁盘上的目录路径仍然写 tenants/{slug}/skills/。inconsistency 永久存在。

推迟理由这条路径是 agentfs daemon 跟 ccaas overlay base 的契约。改名意味着 daemon side 的 base-overlay 配置要同步改，还要清掉所有现存的 _agentfs_base dirs。这次改动收益(命名一致)对比代价(跨 daemon 协调 + 清盘)严重不成比例。

重新审视的触发下一次 agentfs daemon 因其他原因要做 base-overlay schema 变更的时候，顺手一起改。或者出现因为路径不一致导致的实际 bug。

DEF-02

solution.json wire schema 字段 "tenant": { name, slug }

现状每个 ship 的 solution(live-lesson、demo-sandbox、lesson-plan-designer 等)的 solution.json 里有 "tenant": { "name": "...", "slug": "..." }。schema 名跟 entity 名不再对应。

推迟理由这是 solution 实际 ship 的 wire 格式，改字段名意味着每个 solution 仓库都要改 solution.json 文件。这跟 ccaas-core 的代码 rename 不是一回事 —— wire 格式属于 ccaas 跟 solution 之间的契约。

重新审视的触发下一次 solution.json schema 出 major version (v4)的时候，作为 schema 升级的一部分一起改。

DEF-03

WorkspaceArtifactSource.loadArtifacts(projectId: string) 参数名

现状interface 名是 WorkspaceArtifactSource，但内部的参数名还叫 projectId。每个方法签名是 loadArtifacts(projectId: string) / saveArtifact(projectId, ...)。

推迟理由判断: 参数名是 local nomenclature，跟 type-level rename 不是一个范畴的事。改参数名不影响调用方(参数按位置传递)，只是局部可读性。这次 scope 已经够大，不强求把参数名也扫一遍。

重新审视的触发下一次为别的原因动 WorkspaceArtifactSource interface 时(比如加新方法 / 改返回值)，顺手把参数名也改成 identity。

DEF-04

CCAAS_API_KEY env var 名

现状env 变量名仍然是 CCAAS_API_KEY。operator 在每份 .env / docker-compose.yml / K8s ConfigMap 都按这个名字配置的。

推迟理由operator-facing constant。改这个名字意味着每份部署文档、每个生产部署的 secret 都要改。"全改" 不包含这种 ops-facing surface —— 代码 rename 跟 ops 流程是两条边界。

重新审视的触发下一次 ccaas 的环境变量做 namespacing(比如 CCAAS_* 全改成 KEDGE_*) 的时候一起改。或者出现真实 operator 困惑。

DEF-05

ADR-0003 / ADR-0011 / CHANGES_2026-05.md 历史记录

现状这些文档里还会出现 "Tenant entity" / "bind-project route" 这些已删除的概念。

推迟理由这是历史记录，不是 reference 文档。它记的是 "在那个时间点 ccaas 长什么样、为什么做某个决定"。把它们 rewrite 等于篡改历史 —— 未来读者会失去"why" 的上下文。

重新审视的触发永远不动。如果有新的 ADR 引用旧名字，那条新 ADR 用 "formerly known as X" 这种 callout 即可。
