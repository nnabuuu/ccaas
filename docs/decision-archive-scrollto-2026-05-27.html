从"切到对的 tab"到"滚到对的位置"。

V7-rich 框架上次 ship 时留了一个 DEF-01: 审计报告里 nav://execution/step-1 这种链接, 点了只切到执行设计 tab, 但 anchor 部分被丢弃 —— 用户得自己在长长的 step 列表里翻。 这次把滚动到目标这半步做完, 顺便定下 LLM ↔ frontend 的 anchor 契约 (V7-rich 的 OQ-03)。

6 个非显然决定: anchor 用 idx 还是 id (D-01), state 怎么从 chat-bridge 传到子 tab (D-02), cold-load 时数据还没到怎么 scroll (D-03), 同一 anchor 连点两次怎么再 fire (D-04), 把 DEF-01 直接做掉 + dual schema 编码 (D-05), 然后 review 时用户继续追问"那为什么不直接 id" —— 重读 deleted-step trade-off 后发现 fallback 是 fail-quiet 而非 fail-loud, 纯 id schema 把 D-01/D-05 都推翻 (D-06)。 cold-load race + nonce 监控 + mid-edit 反复 flash 是 3 轮 review 的发现, 都已修。

D-01

问: nav://execution/? 的 anchor 用 step 的 idx 还是 id?

Contract · LLM ↔ Frontend
触发 V7-rich 决定档案里 OQ-03 标了 anchor schema 待定: step.id 还是 step.idx? Audit prompt 已经在 callout 例子里写了 nav://execution/step-1, 但没明确规定 "1" 是 idx 还是 id 的一部分。 这一次 plan tab 也要支持 anchor (nav://plan/<reqId>), 是个统一定 schema 的好时机。 约束 LLM 写出 anchor 时是基于审计当下的状态;教师可能在审计后重排 step;chip 的 refId (plan 端) 是稳定 doc-author id, 没有相同的问题。
OptionSchemaVerdict
A step.id 不透明 slug (s-1716...-3) —— stable across reorder ❌ 拒 —— LLM 写出的 anchor 不可读,审计报告里 "跳转 Step 1" 变 "跳转 s-1716123-3"。
B step.idx 1-based (step-1 = 第一个 step) —— 跟教师视觉编号一致 ✓ 选 —— 跟"教师所见即所得",reorder 漂移作为已知 tradeoff。
C LLM 同时发 idx + id, frontend 优先 id 兜底 idx ❌ 推迟 —— 需求未现, 现在做就是过度设计。
为什么这么选

Anchor schema 是 LLM prompt ↔ frontend DOM 的契约。 LLM 看到的输入是 manifest JSON (有 idx + id 都行), 但 LLM 输出要让人能读, 因为审计报告本身就是给教师看的——一个写着 "建议修改 Step 1" 的报告比 "建议修改 s-1716...-3" 自然得多。

Reorder 后 idx 会漂移 —— 这是真实代价。 但审计之后立刻 reorder 是低频场景: 审计的产出就是为了 inform 修改, 修改通常是改内容不是改顺序。 真出问题再做 option C 兜底, OQ 里标着等数据。 一开始就上 C 是为不存在的问题付复杂度。

Locked in
StepList 永远暴露 data-step-idx={i+1}; audit prompt 里 N 永远是 1-based。
Locked out
Reorder 后报告 anchor 失效, 没机制重 anchor 旧报告 —— 教师手动重审。

D-02

问: Scroll 信号怎么从 chat-bridge 传到具体 tab?

Frontend · State
触发 ChatBridge.switchToWorkspace(key, anchor) 在 ProjectEditorPage 里实现; 但选完 tab 后, anchor 信息怎么交给即将 mount 的 ExecutionTab / PlanTab? 旧实现里 _anchor 直接被忽略。 约束 Bridge identity 必须稳定 (useMemo([])), 不能每次 anchor 变都 re-create bridge —— 否则消费者 (AuditReportRenderer) 也跟着 re-render。
OptionMechanismVerdict
A Prop drill scrollAnchor + scrollNonce; ExecutionTab/PlanTab useEffect 监听 ✓ 选 —— 跟现有 reloadKey drill 同款 pattern。
B parent 持有 ref + useImperativeHandle, bridge 调 tabRef.current.scrollTo() ❌ 拒 —— parent 要对每个 workspace tab 持 ref; tab 切换时旧 ref 失效; 复杂。
C PageScrollContext, tab 内部 useContext 消费 ❌ 拒 —— 再加一个 context 跟现有 ChatBridgeContext 平行, 没必要。
D 编码到 URL 路径 (/projects/:id#step-1), tab 从 URL 读 ❌ 拒 —— 改路由 schema; tab 切换跟 URL 不同步。
为什么这么选

Bridge identity 必须稳定 (useMemo([])) 是因为 ChatBridgeContext 一变, AuditReportRenderer 重渲染、callouts 重 mount。 所以 bridge 的 method 不能持有 anchor —— 必须用独立 state 承载, bridge 调 setState。

一旦 anchor 落在独立 state, 怎么传给子 tab 就是普通的 React 状态流动问题。 Prop drill 跟 ExecutionTab 已经在用的 reloadKey 完全一致 —— 这是 same shape 复用, 代码读起来不需要切上下文。 Refs / contexts 都能做, 但没有解决 prop drill 不能解决的问题。

Locked in
每个新 workspace tab (将来加 Skills 细节) 都要接 scrollAnchor + scrollNonce 两个 prop。
Locked out
Workspace tab 不能主动触发滚动到自己内部位置 (e.g. block editor drawer 内部); 需要再加一层。

D-03

问: Cold load 时 anchor 已到但 DOM 还没渲染怎么办?

Frontend · Race
触发 初版只用 requestAnimationFrame + setTimeout(200ms) 重试。 Code-reviewer HIGH 标记: ExecutionTab 的 manifest fetch / PlanTab 的 plan+L1+L2 串行 fetch 经常 >200ms, 单次 200ms 重试 悄无声息漏掉冷加载场景。 约束 useEffect deps 必须能反映"数据来了"这件事; 但 useEffect 本身不能查询 DOM——只能跑代码 + 找 element。 失败时没 user-visible signal (静默 no-op)。
OptionApproachVerdict
A 单次 200ms retry, 不行就放弃 ❌ 拒 —— 冷加载实测 >200ms 是常态, 静默漏。
B 指数退避 retry (200ms → 400 → 800 → 1600), 总 3 秒窗口 ❌ 拒 —— 任意时间窗口仍不可靠; 复杂度涨。
C 把"data-ready"信号 (manifest / state.phase) 加进 useEffect deps ✓ 选 —— effect 在数据真到时自然 re-fire, 不靠 timer 估时间。
D 在 fetch 完成的代码路径里直接调 scroll 函数 (跳过 useEffect) ❌ 拒 —— 把"滚动到 anchor"耦合进 data fetch 逻辑, 违反 SoC。
为什么这么选

Timer-based retry 在本质上是猜数据什么时候到——猜得短了漏冷加载,猜得长了浪费时间。 但"数据到了"本来就是 React state 能精确表达的事件 (manifest !== null, state.phase === 'ready'), 直接拿来当 useEffect dep 就是 zero-guess 的解。

这是 React 数据流的 "declare the question, let the framework re-ask it" 的应用: 我不需要主动观察 "数据到了吗", 我只要声明 effect 依赖于这个状态, React 在状态变化时自然 re-run effect。 配合 scrollNonce 解决重复 anchor 问题, 整个 deps 是 [scrollAnchor, scrollNonce, dataReady] —— 三个维度任意变化都重新尝试,直觉上严密。

Locked in
每个 workspace tab 必须暴露一个"data-ready"的 state value 给 effect 监听 (ExecutionTab 用 manifest, PlanTab 用 state.phase)。
Locked out
数据加载失败 (error 分支) 永远不会触发 scroll —— 这是设计上正确的, 但如果将来要 "失败也尝试 scroll到错误条目", 需要换 dep。

D-04

问: 同一个 anchor 连点两次怎么再触发 scroll?

Frontend · Idempotency
触发 如果 child useEffect 只 dep [scrollAnchor]: 用户点 nav://execution/step-1 后滚走, 再点同一链接 —— anchor 不变, deps 不变, effect 不 re-fire, 不滚。 必须有"这是一次新的请求"的 nonce。 约束 Nonce 必须严格单调 (每次 dispatch 都 > 上次), 否则 React 跳过 effect; 必须能 cross-render 保持; 不能跟其它 state 冲突。
OptionSourceVerdict
A Date.now() —— 墙钟时间戳 ❌ 拒 —— NTP slew / 手动改时间能造成 nonce ≤ 前值。
B useRef<number>(0) 计数器, 每次 ++counter.current ✓ 选 —— 单调由实现强制, 跟系统时钟解耦。
C useReducer + action dispatch 模式 ❌ 拒 —— 单一 scroll signal 不值得 reducer 仪式感。
为什么这么选

初版用了 Date.now(), code-reviewer 标记为 MEDIUM: NTP slew / DST 调整能让连续两次 Date.now() 出现 noncenew ≤ nonceold, React 的 dep change detection 看到"没变"会跳过 effect, 用户的第二次点击静默没反应。

换成 ref 计数器, 单调由我们的代码强制 —— ++counter.current 永远比前值大。 这是个 "不要把可靠性外包给系统时钟" 的小例子: 业务语义需要严格单调, 就由代码持有这个不变量, 不依赖 OS。

Locked in
ProjectEditorPage 持有 scrollNonceRef; 任何未来的 scroll 触发都走 ++ref.current。
Locked out
Nonce 不可读 (任意 int) —— 不能用 nonce 做 "什么时候点的" 这种调试推断, 需要时另存。

D-05

问: id 兜底 anchor 的 URL schema 怎么编?

Contract · Encoding
触发 D-01 把 reorder 漂移列为 acceptable tradeoff, DEF-01 punt 到"等抱怨"。 但同一个 review session 用户直接说 "step.id 兜底 anchor 做一下" —— 不等数据, 直接做。 那么 URL 怎么同时编 idx + id? 约束 既要保持向后兼容 (旧报告里的 nav://execution/step-N 还得能 scroll), 又要让 LLM 写新格式时不困惑; URL 不能用 / 当分隔符 (跟 parseNavWorkspace 的 path 切分冲突)。
OptionSchemaVerdict
A 两段 path: nav://execution/step-N/<id> ❌ 拒 —— 跟 parseNavWorkspace 的 / 切分冲突, 要改解析器。
B Query string: nav://execution/step-N?id=<id> ❌ 拒 —— parseNavWorkspace 当前 strip 掉 query string; 解析器要扩展。
C Pipe 分隔: nav://execution/step-N|<id> ✓ 选 —— anchor 内分隔, parseNavWorkspace 不变; 易解析, 易读。
D 两个独立 link: (nav://execution/step-N) (nav://execution/id/<id>) 让 LLM 选 ❌ 拒 —— LLM 大概率选错; 增加 prompt 复杂度。
为什么这么选

Pipe 在 anchor 内部 当分隔符意味着不动 parseNavWorkspace: 那个解析器把 anchor 当不透明字符串提取出来, 内部解析在 parseStepAnchor 里做, 关注点严格分离。 换 query string 或多段 path 都需要修改 parser 的 regex —— 拉大改动面积只为了"看起来正规"。

Pipe 的另一个好处: regex 严格匹配 ^step-(\d+)(?:\|(.*))?$ 让 idx 不能"漏到"id 里去 —— step-1abc 直接被拒绝, 不会被错误解析成 {idx: 1, id: 'abc'}。 严格 strict-match 是对 LLM 输出宽容但不糊涂: 接受不带 id 的简略形式, 但不接受任何夹杂噪声。

Locked in
Pipe 是execution anchor 内部的保留字符; ReadingStep id 不能含 pipe (现实中是 s-<timestamp>-N, 没问题)。
Locked out
Plan anchor 现在只是 reqId 单层, 将来要加"块 id"等需要分隔时不能照搬 pipe (因为 r-id 字符空间可能含 pipe?), 要重新考虑。

D-06

问: 双 anchor 还是直接纯 id schema?

Contract · Final Form
触发 D-05 ship 之后用户立刻问 "用 id 是不是更好, 否则修改一下就错位了, agent 也找不到了"。 重读 ExecutionTab 里的 "deleted-step trade-off" 注释发现里面那句"some scroll less confusing than no scroll" 站不住 —— 用户在删了 step 之后再点旧报告链接, 双 anchor fallback 会滚到位置上的别的 step 上, 静默展示错内容。 这比 "click 没反应" 更糟, 因为用户以为看到的是 LLM 引用的内容。 约束 删 idx 意味着放弃: (a) 旧报告里的 step-N|... 链接停止工作 (silently no-op); (b) URL 不再人类可读 ("Step 1" 这个文本在 markdown 里仍读得到, 但 URL hover 看到的是 opaque id)。 用户接受这两个代价。
OptionApproachVerdict
A 保留 D-05 dual, 但 id 给了找不到时不 fallback 到 idx (minimal patch) ❌ 拒 —— 保留 schema 复杂度跟 parser, 没换来对应收益。
B 纯 id: nav://execution/<stepId>。 删 parseStepAnchor + data-step-idx。 删 spec。 ✓ 选 —— 干净, no parse, no fallback, no wrong-scroll。
为什么这么选

双 anchor 的 fallback 路径本质是"当 id 这条主路失败时, 拿位置猜" —— 但位置只在 step 没被改过时才有意义。 一旦 step 被改过 (而这正是审计后最可能发生的事), 位置猜是系统性错的, 不是随机错。 系统性错很危险: 用户跟报告对比时看到的内容跟 LLM 引用的不是同一条, 但视觉上看不出来。

删掉 fallback 路径就让"id 失效"显式: 不滚, 用户看到链接没反应, 知道这个引用过期了。 这是fail-loud over fail-quiet。 顺带把 parseStepAnchor 跟 12 个 spec 测试删了 —— 没有 parser 就没有 parser bug, 没有 dual schema 就没有 schema 迁移成本。 anchor IS the id, 这是 plan tab 早就在用的形式, 两个 workspace 现在一致了。

Locked in
Execution + plan anchor schema 统一为"workspace 后跟一个 doc-author 稳定 id"。 LLM 不需要算位置。 Agent 解析时 anchor 就是 id。
Locked out
旧报告 (任何含 step-N 的 anchor) 跳转不响应。 URL 不再有人类可读的位置信息 —— 但链接文本仍然可读 (LLM 写 "[Step 1](nav://execution/s-abc)"), 仅 URL 是 opaque。

02 —

Deferred 这次显式没做的

4 项 · DEF-01 在同 session 内 shipped, 其余等触发
DEF-01

step.id 兜底 anchor (reorder 抗漂移)

原现状Anchor 只用 idx; 用户中途重排 step 后再点旧报告的 nav 会跳到位置上对应的 step, 不是审计当时的那个。

原推迟理由"审计之后立刻重排"是低频场景; 实测真实复现率未知。 加 id 兜底要 audit prompt 同时发 idx + id, frontend 优先匹配 id, 增加了 LLM 输出复杂度。

实际触发同一个 review session 里用户直接说 "step.id 兜底 anchor 做一下" —— 不等数据, 立刻做。 推迟时低估了"漂移"的体感成本: 审计完立即在执行 tab 里改 step 的频率比想象中高 (审计本身就是 inform 修改的)。

落地Dual-anchor 方案 (D-05): nav://execution/step-N|<stepId>, 前端优先 id 兜底 idx。 StepList 加 data-step-id; parseStepAnchor 重构返回 {idx, id}。 4 个新测试 lock 行为。

DEF-02

Plan tab 内的 section anchor (heading 级跳转)

现状nav://plan/<reqId> 只能跳 ReferenceChip; 没法 nav://plan/section/教学要求 这种跳到二级标题。

推迟理由当前没有真实场景需要跳 section ——审计报告的 cross-refs 都是 chip-level 或 step-level, 没有 heading-level reference。

重新审视的触发审计 prompt 开始引用 plan 的 "## 教学目标 / ## 教学过程" 章节并希望读者跳过去。 实现要 PlanRenderer 为 heading emit data-section-id, 并扩展 anchor parser。

DEF-03

Audit 报告内部的 #section anchor (报告内跳章节)

现状Audit markdown 渲染时 heading 没有 id 属性, 浏览器原生 #anchor 失效。

推迟理由不在 nav:// 范围 (那是跨 tab 跳); 浏览器原生 hash anchor 是另一个机制。 当前审计报告通常一屏内, 章节跳转不强烈。

重新审视的触发审计报告长到需要 "跳到第三章" 体验 / 用户反馈 TOC 缺失。 AuditReportRenderer 给 renderHeading 加 id={slugify(text)} 即可。

DEF-04

Cross-project nav:// (跨 project 引用)

现状nav:// 只支持当前 project 内的 workspace tab + anchor; 没有 nav://project/<projectId>/execution/step-1。

推迟理由审计也是 per-project 跑的, 报告里不会出现跨 project 引用。 设计上没必要。

重新审视的触发出现 "组级审计" / "教研组对比项目" 这种跨 project workflow。 那时候要重新设计 URL scheme + 路由 + ProjectEditorPage 不再是 leaf 页。

Open Questions 真正还没答案的

3 项 · OQ-01 resolved by D-06; 其余仍 open
OQ-01

idx anchor 漂移在真实使用中频率有多高?

问题选 idx 是因为"reorder 后再看旧报告" 应该是低频。 但多低? 没有数据。

解决D-06 把 idx 整个删了, "idx 漂移"在 schema 层不存在了。 替代风险 (id 失效后 fail-quiet) 不需要频率度量 —— 用户能直接看到链接没反应, 自己判断。

OQ-02

Flash 颜色 (amber-400 35%) 在深色背景下够明显吗?

问题当前 flash 用 rgba(251,191,36,.35), 1.5s 衰减。 亮色 step row 上很显; 但 callout 内部、深色背景下、或 dark mode (未来) 下未必。

为什么 open没真实在不同 callout severity (warn 黄底 / error 红底) 跳进去试过。 需要 manual smoke 不同跳转目标看视觉是否够。 可能要换成边框而非背景的 flash, 或加 outline。

OQ-03

PlanTab 编辑模式下 TipTap 跟 scrollIntoView 会不会冲突?

问题Read mode 跟 edit mode 都用同一个 [data-req-id] querySelector + scrollIntoView; TipTap 的 NodeView 不影响 DOM attr。 理论上没冲突, 但 TipTap 自己也有 editor.view.scrollIntoView 在 selection 变化时调用。

为什么 open没真在 edit mode 下点 audit chip 试过 (smoke test 待做)。 真冲突 (e.g. TipTap 自己又滚回来) 要换成 TipTap 的 imperative API。
