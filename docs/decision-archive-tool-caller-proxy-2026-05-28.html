D-01

问: 这一轮的实施范围设在哪?

Process · Scope
触发 设计文档把工作分成 5 个 phase: 核心 proxy / 注册 API / 权限引擎 / 多 engine 适配器 / 审计 dashboard。 用户起点是 "我想分析一下工具暴露的方式 + 让 agent 在身份限制下工作", 我开 AskUserQuestion 让用户拍板。 约束 范围越大, blast radius 越大。 但范围太小又留下一堆悬空的契约 (e.g. ToolDefinition 写了 requiredPermissions 字段但没人 enforce 它)。 找"最小但自洽"的切割线很难。
OptionScopeVerdict
A Security-critical core only: ToolCallerProxy + ambient ExecutionContext + sanitization。 wire MCP adapter behind 现有 solution.json。 不做权限引擎, 不做 Opencode 适配器。 ✓ 选 —— 关掉攻击面是不可推迟的; 其他 phase 等真有 use case 再做。
B Proxy + SolutionToolkit registration: 把 live-lesson-creator-tools 从 stdio MCP 重写为 typed registration, 走 in-process handler。 ❌ 拒 —— 重写 solution 代码, 干掉刚做的 materializer; 端到端 blast radius 大。
C Full Phases 1-5: proxy + registration + permission engine + scopes + 2nd adapter + audit dashboard。 ❌ 拒 —— 多周工程, 把已经能用的 stdio-MCP 卡片 POC 阻塞掉。
D Design-only iteration: 写一份 ADR, 把契约里没解的问题列清楚, 不写代码。 ❌ 拒 —— 用户已经写完设计文档, 再写一份 ADR 没价值。
为什么这么选

挑出 "security-critical" 作为切割线是关键。 它不是 "scope 最小" 的同义词 —— 比 D 更大, 比 B/C 都小 —— 而是 "如果这块不做, 后续 phase 的存在意义就归零" 的子集。 没有 ambient identity 的权限引擎相当于在沙堡上盖楼。

B 是 表面上的下一步(把 solution 代码迁移过来), 但它把 "改 ccaas 平台" 和 "改 solution 业务代码" 混在一个 commit 里。 我们刚刚 ship 完 cards POC, 任何回归会立刻被发现, 所以解耦平台改动和业务改动 比 "代码量少" 更值钱。 A 保留了 stdio MCP server 不动, 只在它前面插了一层 proxy —— shipped binary 一行没改, attack surface 已经关掉。

Locked in
ToolDefinition 上声明 requiredPermissions + visibility 的字段已经有了, future 加权限引擎不破 contract。
Locked out
权限/scope/visibility 三层只有 strip 起作用; 真要 gate 数据的 solution 还需要等下一轮。

D-02

问: end-user identity 怎么从 solution backend 到 ccaas?

Security · Transport
触发 设计文档说 identity 必须 ambient + session-bound。 但 end-user 是在solution backend用 cookie/JWT 登录的 —— ccaas 只看到 tenant API key。 怎么把 "现在是谁" 透传给 ccaas? 约束 browser 永远不能持有 ccaas API key (已经定下的安全边界)。 transport 必须经过 solution backend。 但 solution backend 是多 tenant 的, 它能不能被信任替任意 user 说话?
OptionTransportVerdict
A Per-call header: X-Ccaas-On-Behalf-Of: <userId> 在 solution backend 的 Bearer tenant key 旁边附带。 ccaas 信任 header, 在 session 创建时 bind 为 ambient identity。 ✓ 选 —— 0 新基础设施, contract 最小, audit 可见。
B 短期 exchange token: solution 用 tenant key 加 userId payload POST /sessions/exchange, 拿回 session-scoped token, 浏览器走 SSE 用那个。 ❌ 拒 —— 多一次 round-trip + 多一张表 (exchange ledger)。 价值在未来公网部署, MVP 不需要。
C Solution 签发 JWT 带 userId + role + tenant, ccaas 校验签名。 共享密钥 / JWKS 管理。 ❌ 拒 —— 引入 JWKS rotation, 重 infra。 适合多家第三方 solution 接入的成熟阶段。
为什么这么选

关键判断: solution backend 已经是 ccaas 的 trust boundary(它持有 tenant key)。 多绕一层 exchange token 不能改变这个事实 —— 一个有 tenant key 的 backend 已经可以替这个 tenant 的任何 user 行事。 A 把这个信任关系显式化: header 让 audit log 看到 "solution backend 声称这次调用是 teacher-42"; B 把它隐藏在一个交换流程后面, 让审计 trail 多一跳但安全模型没变。

A 的代价是 solution backend 必须正确 auth 它的 end user 才能转发对的 header。 但这不是新风险 —— solution backend 本来就要替 end user 拿数据、写数据。 与其在 ccaas 这层增加交换机制再 paper over 同一个信任问题, 不如保留它的形状, 同时把 spoof 攻击 observable 起来: header 若覆盖 API-key 绑定的 userId, ccaas 打 WARN; turn-N payload 跟 first-set 不一致, 也打 WARN。

Locked in
一次 HTTP header, 0 新基础设施。 solution backend 已经在每个 ccaas 调用上加 tenant key 了, 多一个 header 是 trivial。
Locked out
要 audit 显示 "ccaas 自己也校验过 user 而不是只信 solution backend" 必须升级。 公网/多 tenant solution 接入时迟早要做 (OQ-04)。

D-03

问: MCP 调用拦截的边界放在哪?

Runtime · Wire
触发 Claude Code 跟 stdio MCP server 之间是直接的 stdio JSON-RPC pipe ——ccaas-core 看不到中间任何字节。 想给每个 tool call 加 sanitize/validate/audit, 必须把自己插进这条 pipe。 怎么插? 约束 不能改 Claude Code (没源代码)。 也不想改 shipped solution stdio MCP server (才刚 ship 完, 改了等于 invalidate cards POC)。
OptionWire boundaryVerdict
A In-process toolkit registration: solution 把 tool handlers 当 JS 函数注册进 ccaas-core, ccaas 直接 host 一个 MCP server 给 Claude Code, 不需要任何 subprocess。 ❌ 拒 —— 要求 solution 代码住进 ccaas-core 进程; 现有 stdio binary 全部要改。
B Per-session ccaas-owned MCP proxy bundle: ccaas 自己 ship 一个 stdio binary, Claude Code 看见这个而不是 solution 的 binary。 binary 通过 loopback HTTP 回调 ccaas-core 的 ToolCallerProxy。 ccaas-core 内部再用 StdioMcpToolkit 转发到 solution 原本的 stdio binary。 ✓ 选 —— Claude Code 视角不变, solution 代码不动, 拦截点正好在两者之间。
C Wrap each solution's MCP server with a per-server wrapper subprocess that proxies stdio. ccaas 不 host server, 只插一个 wrapper。 ❌ 拒 —— 每个 server 一个 wrapper 进程, scale 更差; 而且 wrapper 之间没共享 audit log。
为什么这么选

B 的优雅在于它把 wire 协议保留为不变量。 Claude Code 不知道发生了什么 —— 它看见一个标准的 stdio MCP server。 Solution 的 stdio binary 也不知道 —— 它收到 JSON-RPC 调用就回复, 跟之前完全一样。 中间的 ccaas-owned 那一段 就是新增的 platform 层。

A 在美学上更纯 (一个进程, 没有子进程), 但要求 solution 代码改写。 而我们刚刚做完 cards POC 的事实告诉我们: 不破坏既有的 solution shipped code 是这一轮最重要的物理学。 C 是 B 的弱化版, 它分散了 audit/sanitization 的 single chokepoint 价值 —— 把同一个 pipeline 复制 N 次, 每加一个 server 都多一个进程, 没有 ROI。

Locked in
一个 ccaas-owned stdio binary 是 wire 边界。 加任何新功能 (audit DB, permission engine, rate limit) 都加在这一层。 多 engine adapter (Opencode) 只需要换 binary, contract 不变。
Locked out
per-session subprocess 数量 = N (每个 active session 一个 proxy binary)。 大规模部署 (1k+ session) 时是 process budget 问题; 当前是对的取舍但未来要 revisit (OQ-02)。

D-04

问: proxy 怎么知道每个 tool 的 schema?

Design · Schema source
触发 proxy 对 Claude Code 暴露的 tool list 需要 JSON Schema 让模型生成正确入参。 ccaas-core 内部的 sanitization pipeline 还想用 Zod 做 reserved-field strip。 schema 从哪里来? 约束 solution 的 stdio MCP server 源代码已经有 schema (hand-mirrored JSON Schema 或 Zod), 但跨 npm package 直接 import 不友好。 hand-mirror 一份在 ccaas-core 就会 drift。
OptionSourceVerdict
A Hand-mirror schema 在 ccaas-core: 每加一个 proxy-enabled tool 都要在两边手写。 ❌ 拒 —— 必然 drift, 出 bug 时不知道哪边是 source of truth。
B solution.json 里再加一个 toolSchemas 块, 操作员手填 JSON Schema。 ❌ 拒 —— 第三种 source of truth, 跟 stdio server 自身的 inputSchema drift。
C Solution loader 在 import 时启动 stdio MCP server 一次, 发送 tools/list, 把 server 自己声明的 inputSchema 捕获, 然后杀掉子进程。 把 JSON Schema 放到 ToolDefinition.jsonSchemaOverride, Zod schema 用 z.unknown() (sanitization pipeline 不依赖结构验证)。 ✓ 选 —— Single source of truth = stdio server 自己。
为什么这么选

C 的关键认知: stdio MCP server 的 inputSchema 已经是 source of truth。 它必须有, 否则 Claude Code 本来就用不了。 我们只是把 ccaas-core 设计成读它而不是另存一份。 probe 是一次性 (开机后 import 时跑一次, 子进程杀掉), 不增加运行时开销。

B 看起来 "操作员友好"(JSON 配置文件), 但实际上是增加了一个永远会跟代码 drift 的配置层。 每次 stdio server 改 schema, 操作员要记得同步; 忘记一次, agent 就拿着错的 schema 调用工具。 A 同问题, 只是 drift 的两边换成了 ccaas-core 和 solution 代码。 C 的代价是 import 路径要能spawn(probe 一次), 但 SolutionLoader 已经在做 materializer 了, 多一个 probe 是边际成本。

Locked in
所有 proxy-enabled solution 的 schema 都从它自己的 stdio server 自动同步。 改一处就够了。
Locked out
import 时 probe 失败 (server 启动慢、有 bug) → 该 server 的 toolkit 注册失败, warning 一条, 走 legacy 路径; 不致命但需要操作员看 log。 长期想要的可能是 schema 作为 solution package 的独立 data file(plan §"Open questions")。

D-05

问: 怎么保证 reserved-field 集合永远跟 ExecutionContext 同步?

Security · Contract
触发 第一轮 code review (H1) 发现: 我加了 apiKeyId 到 ExecutionContext, 但忘了把它加进 RESERVED_ARG_FIELDS。 agent 可以写 args.apiKeyId 绕过 strip。 同样的洞会在每次 ExecutionContext 加新字段时重现。 约束 TypeScript 没法在运行时 列出 interface 的所有 key —— interface 是 type-only。 也不能让运行时 array 自动反射出 type signature。
OptionEnforcementVerdict
A Code review / 文档纪律: 每次改 ExecutionContext 提醒人也改 RESERVED_ARG_FIELDS。 ❌ 拒 —— 已经验证过这个机制不可靠 (这次就忘了)。
B 写一个 spec, 用 TypeScript satisfies 把"reserved-set 的 superset 必须是 ExecutionContext keys"这个性质固化在 type 层。 加新字段不更新 reserved 集合 → spec 编译报红 → 发现得到。 ✓ 选 —— 把"漏字段"从 runtime bug 变成 build-time error。
C 用 zod 把 ExecutionContext 改成 runtime schema, 反射 keys 进 reserved 集合。 ❌ 拒 —— ExecutionContext 不是parse 进来的, 是平台构造的; 用 zod 是误用工具。
为什么这么选

B 用 satisfies 子句让测试代码自己声明它依赖的 ExecutionContext keys, 编译器检查这个 literal 是否真是 ExecutionContext 的子集。 然后运行时检查这些 keys 都在 reserved set 里。 漂亮的地方在于: 编译器和 test runner 各负责一半, 任何一半失败都 fail-loud。

这是把 invariant 钉死的工程模式: 别让"保持同步"成为人的纪律。 真正可靠的 invariant 是编译器知道、测试知道、文档同步三层重合。 reserved-field 的契约从此每个 PR 都 trivially 检查 —— 加新字段不加 reserved 项, CI 报红, 修复 3 行, 防住一类我们 review 时差点错过的 spoof 类。

Locked in
ExecutionContext 加字段 → spec 红 → 提醒加 reserved 项 → 修。 invariant 不会再 drift。
Locked out
ExecutionContext 加嵌套对象字段不会被 catch (reserved-set 只覆盖 top-level)。 当前 ExecutionContext 是 flat 的, 长期看可能要扩展 sanitizer (现在写在 docstring 里作为已知限制)。

D-06

问: shouldProxy 何时求值: 一次 / 每 turn?

Runtime · Routing
触发 第二轮 code review (M5) 指出: 我把 session.useToolCallerProxy 设成 first-turn-only (immutable after first set), 这意味着如果 SolutionLoader 还在 import (boot 慢) 时第一个 session 已经发消息, registry 是空的 → shouldProxy 返回 false → session 一辈子走 legacy 路径。 约束 想要的是: routing 决策是动态的(registry 状态变化能 picked up), 但 identity 决策是静态的(actingUserId 必须 immutable, 否则 spoof 风险)。 两个 invariant 看起来矛盾, 实际不是。
OptionApproachVerdict
A 保持原状: 第一 turn 决定一次, immutable。 用 SolutionLoader 排序 + 加 readiness probe 让 session-startup 等到 loader 完成。 ❌ 拒 —— 增加 startup 同步原语, complicate boot, 解决了 lock-in 但带来了新的 race。
B 每次 CLI spawn 直接调 adapter.shouldProxy(session), 不依赖 cached flag。 routing 跟 registry 状态实时一致, identity 依然 immutable (不同字段, 不同生命周期)。 ✓ 选 —— routing 跟 identity 解耦, 各自适合的可变性。
为什么这么选

这里关键是识别出我把两个本质不同的决策耦合在一个字段里。 identity(actingUserId) 在 session 创建时 bind, 必须 immutable —— 否则 spoof。 routing(走 proxy 还是 legacy) 是每次 spawn 子进程都可以重新决定的部署细节, 跟 identity 无关。

原本的 session.useToolCallerProxy 字段把两者绑在一起, 因为我直觉上把 "session 用了 proxy" 当成 session 的属性。 但session 用了 proxy 不是事实, 是当时的环境; 真正的 session 属性是 identity。 解耦后每次 spawn 都 ask the adapter; cli-process 不需要任何 cached state; registry 是 single source of truth。 这是把 "看起来像状态" 的东西识别为 "其实是 derive 出来的" 的小例子, 但带来了 race 消失 + 代码更简单 + 测试更窄。

Locked in
SolutionLoader 慢启动 / 后续注册 / hot reload 全部正确处理 —— 下一 turn 自动 picked up。 cli-process 无 session 内 state 需要维护。
Locked out
shouldProxy 的查询频率从 "1 次/session" 变成 "1 次/spawn"。 当前是 in-memory Map 查询, 不是问题; 未来 registry 真大了可能要 cache。

5 项 · 触发条件下重新审视
DEF-01

Permission engine: visibility filtering + requiredPermissions + scope types

现状ToolDefinition 上 requiredPermissions + visibility 字段已经声明; 但 ToolCallerProxyService.invoke 的 Step 3 (permission check) 是 stub —— 总是 allow。 ExecutionContext.effectiveScope 永远 undefined。

推迟理由当前 live-lesson-creator-tools 是单 user 的教师创作场景, 没有跨用户数据访问。 没有 use case 驱动权限引擎; 提前做 = 设计在空气中。

重新审视的触发第一个真正跨用户的 solution 接入 (校长能看本校学生, 教师只能看自己班)。 那时 D-01 的 "stub 字段" 就有了真实 enforcer。

DEF-02

SolutionToolkit as in-process module (干掉 stdio subprocess hop)

现状StdioMcpToolkit 把 tool call 转发给 solution 的 stdio binary; 多一次 IPC hop, 多一个进程, schema 通过 probe 同步。 设计文档预期的最终态是 solution 直接把 toolkit 注册成 ccaas-core 进程内的 JS module —— 没有 subprocess。

推迟理由需要 solution 代码住进 ccaas-core (跨 package 依赖、版本管理、code loading 机制)。 D-03 已经选了 "不动 shipped binary" 作为本轮的物理学。

重新审视的触发process count 成为部署瓶颈 (>1k active session) 或 latency budget 紧到 IPC 不可接受; 或第一个 solution 主动愿意改写 (典型场景: solution 跟 ccaas 同语言同 monorepo)。

DEF-03

多 engine adapter (OpenCode / 自定义 engine)

现状只有 McpEngineAdapterService (Claude Code MCP)。 EngineAdapter interface 已经声明, 但只一个实现。

推迟理由OpenCode / 自定义 engine 的工具暴露协议跟 MCP 不一样; 不写出来不知道 abstraction 对不对。 在没有第二个 engine 之前, 抽象出的"通用 contract" 大概率是幻象 contract(只 fits 当前实现)。

重新审视的触发真的有 solution 想用 OpenCode (或 ccaas 自己想 default 切换)。 那时根据真实需求 evolve interface, 而不是猜。

DEF-04

actingUserId 持久化: first-turn mutation 写回 DB

现状Session entity 有 actingUserId column, 但 worker 是在 in-memory ManagedSession 上 first-turn-set 的 —— restart 后从 DB 读回为 null。 (第一轮 review M2 也提到。)

推迟理由当前 audit 路径还在 logger fallback (没接 DB sink), 持久化 actingUserId 还没下游消费者。 等 audit sink 真的接 tool_events 表时一起做。

重新审视的触发第一个 audit 报表 / 治理流程需要按 actingUserId 历史聚合; 或 incident 调查需要 "重启前那条 session 是谁在用"。

DEF-05

Audit dashboard / 结构化 admin UI

现状audit 写在 logger ([audit-fallback] 行), 也可以由调用者注入 sink。 没有 admin UI 让 ops 看历史 strip 记录、failed validation、tool 调用频率。

推迟理由logger fallback 够 ops debug; admin UI 是有了 100x audit volume 才 ROI 正的工程。

重新审视的触发第一次 spoof attempt 真的来 (audit log 里看到大量 stripped fields), ops 需要 dashboard 找 pattern; 或合规审计要求结构化 audit。

03 —

Open Questions 真正还没答案的

4 项 · 没 confident answer
OQ-01

tool schema 长期住在哪?

问题D-04 选了 import 时 probe stdio server 一次。 这对 single-source-of-truth 是好的, 但意味着 solution 升级 schema 后 ccaas 需要重新 probe (现在只有重启才会)。 长期是要做hot reload(solution 通知 ccaas re-probe), 还是把 schema 抽成独立 data file(solution 直接 ship JSON 不需要 spawn)?

为什么 open取决于 solution 怎么发布 — 如果是 ccaas-controlled rolling deploy, hot reload 有意义; 如果是 solution 自己 maintain 的 npm package, data file 更干净。 当前都没有 driver。

OQ-02

proxy MCP server 应该 per-session 还是 singleton?

问题per-session 是D-03 选的: 每个 active session 一个 proxy binary 子进程。 简单, 每个 session 的 token + ExecutionContext 在 env var 里。 但process count = N, 大规模有问题。 singleton 方案: 1 个 proxy server, 每次 tools/call 在 body 里带 sessionId, ccaas 查 ledger。 latency 一样, process budget = 1。

为什么 opensingleton 把 sessionId 显式带进 wire, 给 spoof 多一个 attack surface (proxy server 是 untrusted 的; 它能不能伪造 sessionId?)。 当前没 process pressure 不用决定; 真有了再权衡。

OQ-03

X-Ccaas-On-Behalf-Of 跟 API-key-bound userId 谁赢?

问题API key 可以绑定一个 userId (作为 "这个 key 代表谁" 的语义)。 同时 solution backend 在 call 时又附带了 header。 当前选择 header 赢 (solution backend has fresher knowledge), 但 logging WARN 当两者不同。 真要让 ccaas 独立校验(不只是 trust solution backend 的话) 需要 D-02 升级到 exchange token / JWT。

为什么 open什么场景必须升级? 没有明确标准。 候选: (a) 第一次有非内部 solution 接入 (第三方); (b) 合规要求审计 chain 包含 ccaas 自己的 user 校验; (c) 单实例 ccaas 上有多个 solution backend 同时跑。

OQ-04

WebSocket gateway 路径什么时候必须硬 block (不只是 WARN)?

问题第一轮 review H2: sessions.gateway.ts 上的 chat path 不接 header, 创建的 session 是 actingUserId=null。 当前选择是WARN log, 不 block。 真的 reject 会破坏 (a) 还在用 socket 的 SDK 客户端、(b) AUTH_ALLOW_ANONYMOUS 模式。

为什么 open不知道第一个真的依赖 actingUserId 做 gating 的 solution什么时候出现。 那之前 WARN 已经 observable; 那之后 null actingUserId 意味着访问被拒。 reject vs WARN 的切换需要跟 solution 灰度协调。
