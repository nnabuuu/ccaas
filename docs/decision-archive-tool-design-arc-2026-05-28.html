Backend · Discipline

OptionApproachVerdict
A 留 defensive guard, 写在 comment 里 "覆盖一个 rare TypeORM-SQLite edge case"。 反正零运行时开销, 未来踩到就帮一下。 ❌ 拒 (initial 选了这条, 后来翻案) —— "rare edge case" 没 evidence 时是假阳性, 不是 hedge。

为什么这么选

这个决定是整段弧线里最反直觉的。 直觉说 "防御代码总没坏处", 实际上没 evidence 的防御代码有comprehension 税: 下次 reader 看到 "覆盖 rare edge case" 会以为这个 case 真的存在, 会重视它, 会在 PR 里小心绕开它 —— 全部都是基于错的因果模型。 一段说 "this protects against X" 的代码隐式在告诉所有未来读者"X 是真的"。

Security · Scope

OptionScopeVerdict
A Security-critical core only: ToolCallerProxy + ambient ExecutionContext + reserved-field strip + audit。 permission engine / Opencode adapter / in-process toolkit registration 全 defer。 ✓ 选 —— "关掉攻击面" 是不能拖的; 其他都是可以等的优化。
B Phase 1+2: 加 SolutionToolkit 注册 API, 把 live-lesson 重写成 in-process toolkit。 ❌ 拒 —— 改 solution 代码; 干掉 b7a541f6 的 materializer; 重写刚 ship 的 stdio binary。
C Full 5 phases。 跨周工程。 ❌ 拒 —— blast radius 太大, 任何回归会 invalidate cards POC。
D Design-only ADR, 不写代码。 ❌ 拒 —— 已经有 design doc, 再写 ADR 没新增信息。
为什么这么选

关键不是 "scope 最小"(D 最小), 也不是 "scope 完整"(C 完整), 而是scope 围绕不可推迟的属性 —— identity-spoofing 攻击面是不能拖的 (一旦有 solution 按用户身份 gate 数据, agent 就是 injection 注入面); 其他几 phase 都是等有真 use case 再做性质的优化。

A 还有一个微妙的好处: 它把"不破坏 shipped 代码"作为设计约束明示给后面所有决定。 这个约束直接 shape 了 D-05 (wire 边界选择) —— 我后面所有"in-process vs stdio wrapping"的取舍都要 honor "不动 shipped binary"。 这种 "用 scope 来 lock-in 后续约束" 的取舍 pattern, 比单独看决定的"scope 大小"更深 —— A 不只是范围更小, 它让所有后续决定都更受限, 但限制方向是对的。

问: 把 platform 层插在哪里?

Runtime · Wire
触发 D-04 把 scope 定了, 但在哪条线上加 sanitize/validate/audit? Claude Code 跟 solution stdio MCP server 之间是直接的 stdio pipe — ccaas 看不到中间字节。 想拦截必须 break-and-rejoin。 约束 D-04 的 "不破坏 shipped solution binary" 约束在这里 binding —— 不能改 Claude Code (没源码), 也不能让 solution 把 stdio 改成别的。
OptionWire boundaryVerdict
A In-process toolkit registration: solution 把 tool handler 当 JS 函数注册进 ccaas-core 进程, ccaas 直接 host MCP server。 没 subprocess。 ❌ 拒 —— solution 代码必须住进 ccaas-core 进程; 违反 D-04 约束。

C Per-server wrapper: 每个 solution mcp server 一个 wrapper subprocess。 ❌ 拒 —— process count = M × N (server × session), audit 也分散。
为什么这么选

B 的优雅是把 wire 协议保留为不变量: Claude Code 视角下还是一个标准 stdio MCP server; solution 的 stdio binary 收到的 JSON-RPC call 跟以前一摸一样。 中间多出来的那一段就是 platform 层 —— 所有 sanitize / audit / context inject 都住在这一段。

Locked out

D-06

问: 怎么让契约不漂移?

Security · Contract
触发 第一轮 code review (H1): 我加 apiKeyId 到 ExecutionContext 但忘了加进 RESERVED_ARG_FIELDS。 agent 可以写 args.apiKeyId 绕过 strip。 同样的 mistake 在每次 ExecutionContext 加字段都会重现。 第二轮 review 时 (M5): 类似的 "改一处忘了改另一处" 在session.useToolCallerProxy 跟 shouldProxy 之间也出现 (D-04 的细节 - 一个 cached 一个 live, drift)。 约束 TypeScript 不能在运行时 列出 interface 的所有 key —— interface 是 type-only。 也不能事后 review 一致性 (这次 review 就漏了)。
OptionMechanismVerdict
A Code review / 文档纪律: 加字段时提醒人同步更新另一处。 ❌ 拒 —— 已经验证过这个机制不可靠 (review 自己就漏了)。
B 用 TypeScript satisfies 在 spec 里声明type 级 invariant: "reserved-set 的 superset 必须是 ExecutionContext keys"。 加新字段不更新 reserved 集合 → spec 编译报红。 M5 用同形 pattern: 不存 cached useToolCallerProxy, 每次 spawn 直接 query shouldProxy() —— 让 routing 跟 registry 状态derive 出来, 不能 drift。 ✓ 选 —— "结构上不可能 drift" 比 "约定记得改" 强一个量级。
为什么这么选

这是整段弧线学到的最有 transferable value 的 pattern: "看似需要保持同步的两份数据" 通常是设计错位 —— 真正的 fix 是让一份从另一份 derive, 或在编译器层 enforce 一致性。 B 不只是"防御 H1 这个具体 bug", 它把"加 reserved field 但忘了 spec" 这一类 mistake 全部消除。 M5 同形: session.useToolCallerProxy 跟 adapter.shouldProxy() 不能 drift 的方法是不存它 —— 每次 query 都 fresh。

这个 pattern 后续会一直反复使用。 它的核心是 "don't cache derived state": 缓存随时会跟源 drift, 永远在防御缓存陈旧; 不缓存 (or compile-time enforce) 是更简单的状态空间。 这个直接受益于 D-03 的"无 repro 不写 defense" 习惯 —— 不再用 defensive code 替代 structural correctness。

Locked in
ExecutionContext 加字段 → spec 编译红 → 提醒加 reserved → 修。 routing 跟 registry 状态实时一致 (M5)。 future 加 contract 看到这两个 pattern 会复用。
Locked out
reserved-set 只 cover top-level keys (嵌套对象不 enforce; 已经写在 docstring 里作为已知限制)。 long-term 可能要扩 sanitizer。

SolutionToolkit 作为 in-process module (干掉 stdio subprocess hop)

现状StdioMcpToolkit wrapping 多一次 IPC + 一个 subprocess。 design doc 的终态是 solution 注册 JS 函数, 0 subprocess。

推迟理由需要 solution 代码住进 ccaas-core 进程 (跨 package 依赖, 版本管理, hot reload 机制), 违反 D-04 / D-05 的 non-perturbation 约束。

重新审视的触发process count 成部署瓶颈 (>1k active session) 或 latency budget 不可接受 IPC; 或第一个 solution 主动愿意改写 (典型: solution 跟 ccaas 同 monorepo)。

DEF-03

Audit dashboard + persistent tool_events sink

现状audit 写在 logger ([audit-fallback] 行) 或注入的 sink。 没有 admin UI 让 ops 看 strip 记录 / validation failure / 调用频率。 actingUserId 也只在 in-memory ManagedSession, restart 后从 DB 读回 null (cards-e2e archive 的 M2 未做)。

推迟理由logger fallback 够 ops debug; 当前 audit volume 不大, dashboard 还在未 ROI 正区间。

重新审视的触发第一次真正的 spoof attempt (audit log 看到批量 stripped fields); 或合规审计要求结构化 trail。

Open Questions 弧线没解的

3 项 · 真正没 confident answer 的
OQ-01

整个 tool 架构是per-session 物理模型还是shared multiplexed 物理模型?

问题当前 per-session: 每个 active session 一个 proxy stdio bundle 子进程 + 一个 token + 一份 ExecutionContext。 数据干净, schema 简单, identity 隔离强。 反方案 shared singleton 把 sessionId 显式带进 wire, 减少 subprocess 数, 但sessionId 在 wire 上传递 多一个 spoofing 攻击面。

为什么 open没 process pressure / latency pressure 做 driver, 现在不该决定。 但终态架构的 scale 是 per-session 还是 shared 影响 future 设计选择 (e.g. 如果终态是 shared, 现在的 StdioMcpToolkit 单进程 wrapping 就该重新看)。

OQ-02

solution 跟 ccaas-core 是同 monorepo还是独立 deployable?

问题这个问题决定很多下游: 如果同 monorepo (现状), in-process toolkit 是 cheap 的 (DEF-02 可早做); 如果独立 deployable, 必须维持 stdio MCP + materializer + symlink 这一套, 加 hot reload 机制, schema 加 separate data file (OQ-tool-caller-1)。

为什么 open目前 ccaas + live-lesson 同 monorepo 用得很顺, 但multi-tenant ccaas as a service 的设计文档 (CcaaS) 暗示终态是 solution 由第三方独立 deploy。 两个方向的工程意涵差很多 — 没有用户给最终方向之前没法决定。

OQ-03

identity 链终态是solution 完全 trusted还是ccaas 独立校验?

问题当前 (D-02 / tool-caller archive): solution backend 持有 tenant key 又转发 X-Ccaas-On-Behalf-Of header, ccaas 信任。 如果一个 solution backend 被 compromise, 它能替任意 user 行事 (在那个 tenant 内)。 升级路径是 D-02 的 "短期 exchange token" 或 "solution-issued JWT"。

为什么 open多大压力下要升? 候选触发: 第一个非内部 solution 接入 / 合规要求 audit chain 包含 ccaas 自己的 user 校验 / 单实例 ccaas 上跑多个 untrusted solution backend。
