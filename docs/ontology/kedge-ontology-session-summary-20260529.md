## 1. 调研结论：没有人在教育领域做过 Palantir 式的 Ontology

我们做了系统的调研，确认 Palantir Ontology 的范式（semantic + kinetic + governance 三层一体的 operational OS）在教育领域是一块完全未被占领的理论和实践空白。

### 全球

- **Palantir 自己不做教育。** 核心客户是国防、情报、供应链、医疗、能源。
- **唯一找到的教育相关案例**：乌克兰教育部 2024 年和 Palantir 签了协议，用 Foundry 做学校防空避难设施的资金分配建模和海外儿童远程教育网络建模。本质上是 public sector 的资源分配优化，没有触及教育的核心实体（课程、知识点、学情、教学策略）。
- **美国教育部**用 Palantir 做外国资金报告门户，同样是行政数据管理。
- **没有任何学术论文**把 Palantir 的 Ontology 范式作为框架来分析或设计教育系统。

### 中国

- 教育信息化领域大量使用"知识图谱"概念——学科本体建模（概念、定理、公式、思想、能力、素养、方法、知识点）、实体关系定义（附属、实例、学情、考情）、自适应学习路径推荐。
- **全部停留在传统语义层**：描述性的、静态的、以检索和推荐为目的。
- 没有人把"行动"和"治理"作为一等公民放进教育数据模型里。

### 为什么空白

1. Palantir 的商业射程里没有教育。
2. 教育领域的核心实体不是物理资产（工厂、设备、产品），而是「知识点」「认知状态」「教学策略」「学习行为」——object type 定义本身充满歧义，不同教学理论给出完全不同的本体。
3. Kinetic layer 在教育场景特别有价值但没人做过——学情数据采集后不是用来"看"的，而是要驱动行动（推送习题、调整教学节奏、触发干预提醒），且需要治理（权限、审计、合规）。

**结论：kedge-ontology 在做的事情，在理论和实践上都没有先例。**

## 2. 给非技术人员解释 Palantir Ontology 的核心区分

（可用于向客户、团队成员、文档中解释 kedge-ontology 的设计动机）

### "看到"和"做到"

传统数据整合把散落在各系统的数据汇总到一个大仓库里，然后出报表、做分析。但整合完的数据仍然只是"表格和数字"，没有业务含义——你看到"表A，行3742"，不知道这是一个入院日期还是一台手术的排期。

Palantir 做的第一件事是把数据变成"东西"——不是表和字段，而是"病人张三"、"他的第二次住院"、"3月15日的手术"、"批次号X的心脏支架"，以及它们之间的关系。这叫**语义层**。市面上有不少产品在做类似的事（知识图谱、数据中台）。

**Palantir 做了绝大多数人没做的事：在语义层之上加了行动层。** 不只是让你看到手术排在3月15日，而是让你直接在这张活地图上"改期"——系统自动检查手术室、医生、器材是否到位，满足就执行，同时记录谁在什么时候做了这个决定。这个操作是被定义过、被管控过的业务行为——有前置条件、有连锁影响、有审批流程、有审计记录。这叫**动能层**。

**"看到"和"做到"之间的鸿沟，是大多数数字化项目投入巨大但效果有限的核心原因。** Palantir 主张这两件事应该发生在同一个地方。你看到的地图就是你操作的地图。AI Agent 也在同一张地图上操作，受同样的权限管控，留同样的操作记录。

### 映射到教育场景

**纯语义层**（今天教育知识图谱在做的事）：系统知道"这是三年级二班的数学课，上的是《三角形内角和》，教师是王老师，班上42个学生"。Agent 可以查询这些信息，但它和这堂课的关系就像读百度百科——只能看，不能碰。

**加上动能层**：这堂课不再是一个词条，而是一个可以被进入、被感知、被操作的运行时上下文。Agent 能做的事变成一个被明确定义的清单：

- **接入观察**：订阅实时事件流（学生举手、教师提问、小组讨论开始），不是课后看静态报告
- **读取状态**：当前在哪个教学环节、哪个资源正在使用、已过多少时间
- **写入判断**：标记"这个环节学生参与度下降"、记录"第12分钟出现关键 misconception"
- **触发行动**：推送即时练习、建议调整难度、给特定学生发提示

每个操作都带治理——Agent 能观察但不能改教案，能建议调整难度但需教师确认，所有操作有审计记录，不同角色的 Agent 看到的边界不同。

**一句话总结：语义层让 Agent 知道"这是一堂什么课"，动能层让 Agent 能"参与这堂课"。前者是认知，后者是行动力。光有认知的 Agent 只是一个更聪明的搜索引擎；有了行动力的 Agent 才是一个真正的教学协作者。**

## 3. 这和 kedge-ontology 的关系

kedge-ontology 中的 **ManifestDef** 就是承载这个"从词条到操作空间"跃迁的原语。以 `LessonSession` 为例：

- **Slot** = 这个操作空间里有什么东西（教案、班级、学生列表、事件流）
- **State** = 当前运行时状态（教学阶段、活跃资源、暂停标志）
- **AccessBoundary** = Agent 能看什么、能做什么（能读所有 slot、能写 state、能执行 adjustDifficulty 但不能 modifyLessonPlan）
- **Lifecycle** = 操作空间什么时候激活/销毁（课开始时启动观察，课结束时生成总结）

ObjectTypeDef 描述"世界上有什么东西"，ManifestDef 描述"此时此刻的操作空间"。前者是语义层，后者是动能层的容器。
