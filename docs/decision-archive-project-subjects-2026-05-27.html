D-01

问: Subject 在 DB 上是单值还是数组?

Schema · Shape
触发 用户回应 DEF-01 的描述时直接给方向: "subject 是数组吧"。 约束 SQLite + TypeORM. 数组列需要一个序列化器 (simple-array / simple-json); 老行需要默认值。绝大多数项目其实只有 1 个 subject —— 数组要为常态保留简洁,不能强制 caller 永远 wrap。
OptionApproachVerdict
A 单字段 subject: string | null ❌ 拒 —— 无法表达"英语+数学跨学科项目"
B subjects: string[] · simple-json 列 ✓ 选 —— 数组原生支持多值, 长度 1 就是常态
C 单字段 + 多对多关联表 project_subjects ❌ 拒 —— join 复杂度 ≫ 信息价值
为什么这么选

数组比单字段多一个表达力维度,代价只是一个 JSON serializer。多对多关联表理论上"更正确"(范式化),但在这个域里没有用例 —— 我们不会查"所有有 english 学科的项目",分布也不会扭曲到要建索引。数据规模上,项目 × 学科 永远 < 1000 行,内存里完全装得下。

关键的设计杠杆是 数组 + 显式空集: subjects: [] 不是"未配置"的歧义,而是显式声明"不依赖任何教学要求库"。这让物化器的入口判断变成纯函数: opts.userId && subjects.length > 0,没有隐式 fallback,测试矩阵彻底闭合。

Locked in
所有 subject-aware 操作都按数组迭代; 单值假设永远不能再回来
Locked out
如果未来真的要 N 对 N 关联 (subject 自己也是个实体),迁移成本中等

D-02

问: 已有的 LIVE_LESSON_LESSON_PLAN_SUBJECT env 怎么收尾?

Migration · Cleanup
触发 数组列定下来之后, runtime 还要不要看 env? 这关系到现有项目 (subjects 默认 []) 启动后的"沉默期"会不会让人困惑。用 AskUserQuestion 显式问了用户。 约束 env 是上一次 commit ed451403 才落地的; 立刻删除是"churn"。但保留意味着 runtime 永远有 "env-vs-row" 隐式分支,违反"test coverage 更明确"的初衷。
OptionApproachVerdict
A 完全删除 env, project.subjects 是唯一来源 ✓ 选 (用户) —— 显式 > 隐式, 测试不含 env mock
B 降级为新项目创建时的 default fallback ❌ 拒 —— "什么时候用 env" 是新的认知负担
C 保留作为 subjects=[] 时的 runtime fallback ❌ 拒 —— 双轨, 测试矩阵爆炸
为什么这么选

这是一个"测试可读性优先"的决定。用户原话: "test coverage 更明确"。如果保留 env fallback,每一条 listArtifactsWithContent 测试都要在 beforeEach mock env、在 afterEach restore —— 这是上一版 spec 的真实状态,污染了所有不关心 lib 物化的测试。

代价: dev 老 SQLite 上的项目 subjects = [],启动后看不到 lib 物化输出。但这是loud breakage —— 用户打开 Edit modal 选 subject,文件就回来了; 比"为什么我没改东西就坏了"那种沉默 fallback 更可调试。在 prod 还没起飞 (synchronize:true 仍在用) 的阶段,优化"开发期可读"是正确的权衡。

副产品: 整个仓库 git grep LIVE_LESSON_LESSON_PLAN_SUBJECT 现在只剩两处 —— 历史决定档案 HTML 里的引用,和 entity 注释里"为什么这个列存在"的说明。两处都是有意保留的考古学注释。

Locked in
所有 lib 物化测试用例都通过 fixture 显式声明 subjects, 无 env 隐式分支
Locked out
如果未来想给"未配置 subject 的项目"自动 fallback 一个 default subject, 要重新引入隐式行为

D-03

问: 多 subject 时, _lib/ 物化成几个文件?

Materialization · Layout
触发 数组定了之后立刻冒出: 一个项目带 [english, math], 物化到 workspace 是一个文件 (concat) 还是多个? AskUserQuestion 让用户挑。 约束 Agent 用 Grep 找 reqId。如果两个 subject 都用 "r-1.2.3" 这种命名 (虽然约定上不该), 单文件里 grep 会同时命中两条 —— 难调试。但单文件物化最简单。
OptionApproachVerdict
A 每 subject 一个文件: _lib/teaching-requirements/<subject>.md ✓ 选 (用户) —— grep 隔离 + 文件名即学科签
B 单一拼接文件, H1 分段 ❌ 拒 —— 跨学科 reqId 冲突难分辨
为什么这么选

关键洞察: 文件路径本身可以是一个分类信号。Agent 跑 Grep -r "推断生词" artifacts/_lib/teaching-requirements/ 时, 结果里的 english.md:r-1.2.3 比 H1 分段的 # Math\n## English\n## r-1.2.3 更难误读。前者天然带学科标签,后者要看上下文。

代价: helper script (find-req.sh) 要 glob 多个文件, awk 要从多个文件聚合。bash-3.2 的 nullglob 这次需要 belt-and-suspenders 防御 (代码评审抓到的实际 bug —— 见 D-06)。但脚本只是辅助路径; agent 的 critical path 还是 Grep -r。

副产品: 物化器的代码结构变成"对每个 subject 调一次 renderLibrary",L2 interpretation 按 subject 分桶 —— 后者解决了一个我们差点没注意的cross-subject leak问题: per-user interpretation 表是不分学科的,如果不过滤,英语项目的 _lib/my-interpretations/english.md 会泄露用户对数学 reqId 的笔记。

Locked in
Workspace 路径约定: _lib/<concern>/<subject>.md。新加 concern 类型 (e.g. _lib/glossary/) 可以平行扩展
Locked out
想反过来"按 reqId 聚合 (单文件按 id 排序)" 需要重写物化器 + 改 reserved-path guard

D-04

问: L2 解读怎么按 subject 分桶?

Data Model · Filtering
触发 D-03 决定每 subject 一个 my-interpretations/<subject>.md。但 L2 表的 schema 是 (userId, reqId, notes) —— 没有 subject 列,只有 reqId。怎么决定某条解读属于哪个 subject? 约束 既不能加 schema migration (这次 scope 不含),也不能让英语项目泄露用户对数学的解读。
OptionApproachVerdict
A 给 L2 表加 subject 列, 写时填,读时按 subject 过滤 ❌ 拒 —— 需要 migration + 写路径改造
B 物化时查 L1 库的 reqId 集合, 在内存里过滤 ✓ 选 —— schema 不变, 计算成本低
C 从 reqId 前缀推断 subject (r-* = english, m-* = math) ❌ 拒 —— 约定不是契约, 容易破
为什么这么选

L1 库本身就是"reqId → subject"的权威映射: library.categories[].items[].id。物化器对每个 subject getLibrary(subject),收集所有 reqId,然后用 userInterps.filter(reqIdsInSubject.has) 切出归属此 subject 的解读。一次内存里的 Set lookup,O(n) 不到。

关键的设计哲学是"compute the partition at read time, don't denormalize": L2 的存储模型保持 per-user 全局 (用户对一条标准的看法跨项目稳定 —— 教用户思考的方式)。Subject 归属是library snapshot 的函数,而不是存储事实。如果哪天 L1 库重组分类、把一条 reqId 从 english 移到 math, L2 自然跟着走,不会留 stale partition。

抓到的细节: 跨学科 reqId 冲突 (e.g. 两个 library 都声明 r-1.2.3) 会让那条用户解读同时出现在两个 subject 的物化文件里。这是约定问题不是 bug —— ids 按约定 namespace (r-* vs m-*),但代码不强制。collectReqIds 的 JSDoc 显式记下这个边界。

Locked in
L2 schema 保持稳定; 任何"按 subject 看我的解读" 都可以从 L1 ∩ L2 推出
Locked out
如果未来要支持"我对同一 reqId 在不同 subject 上的不同解读" (跨学科切换思路), L2 必须扩 schema
filesbackend/src/project/project.service.ts:collectReqIds, listArtifactsWithContent 的 L2 过滤段 commite4a1e339

问: 前端 Create + Edit 两个 modal 怎么组织?

Frontend · Components
触发 原 CreateProjectModal 是 160 行 (form + a11y + 状态管理)。Edit 需要同样的字段、同样的 a11y、同样的 subject picker —— 只是 submit 调 PATCH 而不是 POST。 约束 React 没有原生 form 抽象,容易复制粘贴出三份"差不多但又不一样"的代码。修一处 a11y bug 就要改三遍。

D-06

问: Subjects 数组的验证职责在哪一层?

Defense · Layers
触发 第一版只在 Service 层做"是不是合法 subject"。Code reviewer 抓到 HIGH: 没有 array 长度上限 —— caller 提交 Array(10000).fill('english') 还能 pass。 约束 DTO 的 class-validator (HTTP 边界) 跟 Service 的语义验证不重合: 前者管"形状对不对",后者管"内容是不是已知 subject"。要分清两者职责。
OptionApproachVerdict
A 所有验证集中到 Service (含长度上限) ❌ 拒 —— Service 处理形状错误违反单一职责
B DTO 管形状 (@ArrayMaxSize + @MaxLength each), Service 管语义 (catalog 成员 + 去重) ✓ 选 —— 边界各管各的
C 单层验证 + 文档警告 ❌ 拒 —— DoS 入口要类型化拦
为什么这么选

经典defense in depth: HTTP 边界 (DTO + ValidationPipe) 在路由 handler 进入前就把不合理形状打回 400; Service 边界假定形状合法,只关心"这个值在我的 catalog 里吗"。两层各管各的,Service 不需要重复写 "is array? items are strings?",DTO 不需要知道 catalog 是什么。

关键的代码评审收获: "unbounded array on a write endpoint" 是一个被忽略的 attack surface。catalog 检查不能替代长度检查 —— 10k 个 "english" 的 array 全部 pass 语义检查 (都是合法 subject) 之后,被 dedup 成 1 个,但中间的 Set.has 10000 次 + JSON serialize 是真实代价。@ArrayMaxSize(32) 是真正的边界,选 32 是因为任何合理课程都 < 10 个学科,留 3 倍 headroom 防"未来某种变种 track"。

错误响应体故意带上 {unknownSubjects, validSubjects} 而不是单纯 message 字符串 —— frontend picker 可以直接从 400 body 渲染出可选项,不用额外打一遍 GET /_subjects。这是 API design 里"错误也是产品的一部分"的体现。

Locked in
DTO 永远是第一道 wall, Service 假定 input 是合理形状; 新 subject 类输入复用此 pattern
Locked out
想绕过 ValidationPipe (e.g. 内部调用 Service 不走 HTTP) 时, 长度检查不生效 —— 但这是 in-tree 决定

02 —

Deferred 这次显式没做的

4 项 · 每条带 revisit trigger
DEF-01

GET /_subjects 只返回 keys, 没带 subjectLabel

现状endpoint 返回 { subjects: ["english", "math"] }。前端 picker 直接用 key 当显示文字, 教师看到的是英文 slug 而不是 "英语"/"数学"。

推迟理由UI 还在 dev/demo 阶段, 教师不会看到这个 picker; 用 slug 也能区分。扩成 { subjects: [{ key, label }] } 是向后兼容的 (key 字段还在), 等真有教师反馈 "看不懂" 再做。

重新审视的触发第一次教师使用反馈 / i18n 启动 / 任何要在 UI 显示 subject 的新页面 (e.g. 项目卡片 hover tooltip)。

DEF-02

HttpError 不暴露 body, 前端拿不到 validSubjects

现状backend 在 400 body 里返回 validSubjects 列表 (D-06), 但 creator 的 HttpError class 只存了 status + message, 把 body 丢了。前端 modal 显示的是 message 字符串, picker 不能"自动恢复"。

推迟理由需要改 HttpError 的 API (加 body 字段) + 改所有错误处理调用点。当前手动选 subject 的 UX 也 OK, 用户自己看出哪个是 unknown。

重新审视的触发第一个用户报告 "我选了 X 但提交失败,不知道为什么" / 类似 400 with structured body 的新 endpoint 出现时一并改造。

DEF-03

Prod DB migration script

现状新列 subjects 通过 TypeORM synchronize:true 在 dev 启动时自动加。Production migration 不在 scope (live-lesson 还没有 prod 部署)。

推迟理由整个 live-lesson backend schema 都还在 synchronize 阶段; 单独给一列写 migration 是过早的工程化。等启动 prod 时一起做。

重新审视的触发live-lesson 进入 staging / prod / 第一个真实生产 SQLite 实例。届时统一引入 typeorm migration 流程。

DEF-04

Per-subject RBAC / 学科访问控制

现状任何已登录用户都可以把任意 subject 加到自己的项目里。没有 "教师 X 只能选 english+math, 不能选 chemistry" 的策略。

推迟理由当前所有 subject 库都是 platform-shipped, 不涉及付费/权限/机构隔离。多租户 + 学科分配是更大的 feature, 不该夹带做。

重新审视的触发多机构 / SaaS 场景, 或者 subject 库开始按客户定制 (这个客户买了 english, 那个买了 math)。

Open Questions 真正还没答案的

OQ-01

SQLite synchronize:true 真的会给老行 backfill simple-json default '[]' 吗?

问题TypeORM 文档说会, SQLite 3.31+ 的 ALTER TABLE ADD COLUMN DEFAULT 也会, 但中间是否有 race / 特殊情况 (synchronize 决定 recreate table 时)? 我们加了 project.subjects ?? [] 防御兜底, 但不知道这个兜底是不是真的会被触发。

为什么 open需要在一个有真实老数据的 dev DB 上跑一次 synchronize 才能验证。当前 dev 团队都是 fresh DB, 看不到这个边缘场景。

OQ-02

跨学科 reqId 冲突要不要硬性约束?

问题现在的 L1 catalog 加载允许同一个 reqId 出现在多个 subject 库里 (warning log, 不报错)。collectReqIds 的 JSDoc 也承认这种情况下 "两个 subject 的 interpretation 文件都会出现这条解读"。要不要让 catalog 加载在冲突时直接拒绝, 强制约定 "namespace by prefix"?

为什么 open当前内置 catalog 文件遵守约定 (r-* / m-*), 没有出现过冲突。强制约束会拒绝一些合法的极端用例 (e.g. 同一条课标在跨学科课程里被引用)。

OQ-03

Subject picker 是不是 "项目级唯一" 还是有更细粒度?

问题当前 subjects 是 project-level 的。但一个跨学科课程 (e.g. STEM) 在不同 lesson step 上可能只用其中一个 subject 的 reqId。要不要支持 "step-level subject scope" 让物化更精准?

为什么 open没有真实用户报告需要。step-level scope 会让 lesson plan 文件依赖 step → subject 的映射, 复杂度跳一个台阶。等真的有跨学科课程编写者抱怨 "我每步只关心一个学科,为什么 _lib/ 还有别的" 再说。
