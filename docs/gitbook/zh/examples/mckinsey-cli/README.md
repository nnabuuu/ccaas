# 麦肯锡顾问 McKinsey CLI

McKinsey CLI 是一个结构化商业分析工具，展示了**纯 Skill、零 MCP** 架构模式。单个强大的 Skill 通过利用内置能力（网络搜索、文件生成）和全面的领域指令，替代了所有 MCP 工具。

## 架构图

```
┌───────────────────────────────────────────────────────┐
│  客户端层                                              │
│  ┌─────────────────┐  ┌───────────────────────────┐   │
│  │  Vue 3 前端      │  │  Node.js CLI 客户端        │   │
│  │  (聊天 + 文件)    │  │  (终端界面)               │   │
│  └────────┬─────────┘  └──────────┬────────────────┘   │
├───────────┴───────────────────────┴────────────────────┤
│  CCAAS 核心后端 (:3001)                               │
│  会话管理、Skill 路由、文件管理                         │
├────────────────────────────────────────────────────────┤
│  mckinsey-consultant Skill                            │
│  ┌──────────────────────────────────────────────────┐  │
│  │  SKILL.md (300 行) — 导航 + 工作流               │  │
│  │  references/  (7+ 文件) — 按需加载               │  │
│  │  内置工具: WebSearch, Write, Read                 │  │
│  │  不需要 MCP Server                                │  │
│  └──────────────────────────────────────────────────┘  │
├────────────────────────────────────────────────────────┤
│  生成物                                               │
│  PPTX (15-25 页) + XLSX (数据源) + DOCX               │
└───────────────────────────────────────────────────────┘
```

## 为什么零 MCP？

并非每个 Solution 都需要 MCP 工具。McKinsey CLI 展示了 Skill 何时已经足够：

| 需求 | 需要 MCP？ | Skill 方案 |
|------|-----------|-----------|
| 数据收集 | 否 | 内置 `WebSearch`（每次会话 15-30 次搜索） |
| 文档生成 | 否 | 内置 `Write` 工具（Python 生成 PPTX/XLSX） |
| 参考加载 | 否 | 内置 `Read` 工具按需读取参考文件 |
| 业务逻辑 | 否 | SKILL.md + 参考文件中的领域指令 |
| 外部 API | 不需要 | 所有分析使用公开网络数据 |

**经验法则**：如果你的 Solution 的核心逻辑主要是 LLM 原生思考 + 公开数据 + 文件生成，先考虑纯 Skill 模式，再决定是否需要 MCP。

## 9 步工作流

Skill 实现了结构化的咨询方法论：

| 阶段 | 步骤 | 活动 |
|------|------|-----|
| **分析** | 1-3 | 定义问题边界、构建 Issue Tree (MECE)、形成假设 |
| **设计** | 4-5 | 设计 Dummy 页面布局、标注页面依赖关系 |
| **生成** | 6-7 | 通过网络搜索收集数据、逐页生成 PPTX |
| **交付** | 8-9 | 可选 Word 报告、迭代优化 |

每个步骤按需加载参考文件，使用后释放，高效管理上下文。

## 渐进式披露架构

核心创新在于 Skill 如何管理上下文：

```
V2.0（旧版）：1130 行 SKILL.md 一次性加载
  → 每次会话 70% 的行未被使用
  → 约 15 页后上下文耗尽

V3.0（当前）：300 行 SKILL.md + 按需参考文件
  → 仅在 STEP 2-3 加载 methodology.md
  → 仅在 STEP 4-5 加载设计文件
  → 仅在 STEP 6-7 加载 Excel 规格
  → 每次使用后释放
  → 稳定生成 20-25 页
```

### 参考文件一览

| 文件 | 行数 | 加载时机 |
|------|------|---------|
| `SKILL.md` | 300 | 始终加载（核心导航） |
| `methodology.md` | 150 | STEP 2-3（MECE + Issue Tree） |
| `layouts.md` | 120 | STEP 4-5（7 种页面模板） |
| `design-specs.md` | 180 | STEP 4-5（颜色、字体、间距） |
| `page-dependencies.md` | 80 | STEP 4-5（生成顺序） |
| `excel-data-spec.md` | 100 | STEP 6（数据收集格式） |
| `delivery-summary.md` | 100 | STEP 8（Word 报告，按需） |
| `troubleshooting.md` | 120 | STEP 9（仅在出现问题时） |

## 逐页增量生成

Skill 不是一次性生成所有页面（会耗尽上下文），而是一次生成一页：

```
对每一页：
  1. 检查页面依赖（现在能生成这一页吗？）
  2. 执行 5 次网络搜索获取真实数据
  3. 在 Excel 中记录数据（来源 URL + 数值）
  4. 生成 PPTX 页面（严格遵循设计规范）
  5. 自检 6 项标准
  6. 等待用户确认
  7. 清除页面上下文
  8. 进入下一页
```

这个模式使上下文使用量保持有界，无论总页数多少。

## 双客户端支持

同一个 Skill 同时驱动 Web 前端和 CLI：

- **Vue 3 前端**：分屏 UI（聊天 40% + 交付物 60%），文件下载卡片，Token 用量显示
- **CLI 客户端**：Socket.IO 终端界面，支持 `/files`、`/download`、`/new` 命令，持久化会话存储

两个客户端通过同一套会话 API 连接 CCAAS 核心 —— Skill 与客户端无关。

## 这个架构有什么值得关注的

1. **零 MCP 开销** —— 无需构建、部署或维护工具服务器
2. **上下文效率** —— 渐进式披露使活跃的 Skill 指令始终低于 300 行
3. **有界生成** —— 逐页工作流即使对 25 页 PPT 也不会耗尽上下文
4. **领域即指令** —— 麦肯锡方法论存在于参考文件中，而非代码中
