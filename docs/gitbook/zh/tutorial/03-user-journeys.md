# 3. 用户旅程映射

## 本章学习目标

在本章中，你将为备课方案设计器 Solution 规划端到端的用户旅程。用户旅程追踪用户完成一个目标的每一步——从打开应用到看到最终结果。在编写代码之前映射这些旅程，可以确保没有遗漏任何交互环节，并且 AI Agent、后端和前端都被设计为支持相同的工作流。

完成本章后，你将能够：

- 将功能分解为分步用户旅程
- 识别 AI Agent、Solution 后端和前端各自在哪里发挥作用
- 将 WebSocket 事件映射到旅程中的特定时刻
- 设计同时处理 AI 生成内容和用户编辑内容的旅程
- 在设计阶段尽早发现缺口和边界情况

## 为什么在编码前映射旅程

在传统 Web 应用中，数据流很简单：用户填写表单、提交、后端保存。在 CCAAS Solution 中，有第三个参与者——AI Agent——它引入了新的交互模式：

- 用户输入自然语言请求，而不是填写表单
- AI Agent 解释请求、生成结构化数据，并将其推送到表单
- 用户审查、编辑并确认 AI 生成的内容
- 后端保存最终结果

如果跳过旅程映射步骤，你会在实现过程中才发现这些交互模式，导致返工。

## 架构回顾：直连模式

在映射旅程之前，回顾[第 1 章](01-architecture.md)中的 CCAAS 架构。前端通过 WebSocket **直接连接** CCAAS 进行所有 AI 交互。Solution 后端仅处理领域数据（备课方案、教材等的 CRUD 操作）：

```
┌──────────────┐         ┌──────────────────┐         ┌──────────────┐
│   Solution   │  WS     │  CCAAS 后端      │  stdin/  │   AI Agent   │
│   前端       │◄───────►│  (NestJS)        │  stdout  │   进程       │
└──────────────┘         └──────────────────┘◄────────►└──────────────┘
  Vue + SDK                会话管理                      Claude Code
  @ccaas/vue-sdk           技能路由
                           事件流

┌──────────────┐         ┌──────────────────┐
│   Solution   │  REST   │  Solution        │
│   前端       │◄───────►│  后端            │
└──────────────┘         └──────────────────┘
                           领域 CRUD
                           (备课方案、教材)
```

两条独立通道：
- **AI 通道**：前端 ↔ CCAAS（WebSocket + REST）
- **领域数据通道**：前端 ↔ Solution 后端（REST）

### 此旅程中的 WebSocket 事件

前端通过与 CCAAS 的直连 WebSocket 接收这些事件：

```typescript
// 1. Agent 开始思考
agent_status: { status: 'thinking' }

// 2. Agent 决定使用 write_output
tool_activity: { toolName: 'write_output', phase: 'start' }

// 3. 学习目标区域同步
output_update: {
  payload: {
    data: {
      field: 'learningObjectives',
      value: [
        { description: '理解100以内加法', type: 'knowledge', bloomLevel: 'understand' },
        { description: '运用竖式加法方法', type: 'skill', bloomLevel: 'apply' }
      ]
    }
  }
}

// 4. 工具调用完成
tool_activity: { toolName: 'write_output', phase: 'end' }

// 5. 更多区域依次同步...
output_update: { payload: { data: { field: 'textbookAnalysis', value: { ... } }}}
output_update: { payload: { data: { field: 'studentAnalysis', value: '...' }}}
output_update: { payload: { data: { field: 'learningTasks', value: [...] }}}
output_update: { payload: { data: { field: 'homeworkTasks', value: [...] }}}

// 6. Agent 完成
agent_status: { status: 'idle' }
```

### 这个旅程揭示的设计决策

1. **编辑器必须处理部分更新。** 各区域逐个到达，而不是一次性全部到达。`LessonPlanEditor` 需要在 `output_update` 事件到达时增量更新每个区域。
2. **用户在 AI 生成后编辑。** 每个编辑器区域（如 `LearningObjectivesEditor`、`LearningTasksEditor`）在 AI 生成期间和之后都必须可编辑。
3. **保存是独立操作。** AI Agent 不保存——它只填充表单。用户必须显式点击"保存"来持久化修改。
4. **Solution 后端拥有数据。** 保存请求（步骤 15）发送到 Solution 后端，而不是 CCAAS。CCAAS 仅处理 AI 中继。
5. **两条独立连接。** 前端维护与 CCAAS 的 WebSocket 用于 AI 事件，以及与 Solution 后端的 REST 调用用于领域数据。这是两条独立的通道。

### 这个旅程揭示的设计决策

1. **上下文很重要。** AI Agent 需要知道备课方案的当前状态。`FormStateSynchronizer` 将当前表单状态发送给 CCAAS，Agent 启动时接收它作为上下文。
2. **部分更新是必要的。** AI Agent 应该只更新用户要求的区域（`learningTasks`），保持 `learningObjectives`、`textbookAnalysis` 等不变。
3. **前端必须合并而不是替换。** 当 `learningTasks` 字段的 `output_update` 到达时，前端只替换该区域，保持其他所有区域不变。

### 这个旅程揭示的设计决策

1. **AI 修改的区域需要视觉反馈。** 当 `output_update` 到达时，受影响的区域应显示高亮效果（如蓝色光晕动画），让教师知道 AI 修改了哪些区域。`LessonPlanNewView` 中的 `aiModifiedFields` 机制通过 3 秒淡出动画实现这一效果。
2. **逐区域编辑至关重要。** `LessonPlanEditor` 支持内联编辑，每个区域（课程要求、教材分析、学习目标等）都可以单独编辑、保存或取消。这让教师可以逐一批准各区域。
3. **大纲面板辅助导航。** 备课方案包含多个区域（9+ 个区域），`OutlinePanel` 组件通过滚动监听让教师可以快速导航到任意区域。
4. **草稿持久化保护工作成果。** `useDraftPersistence` 组合式函数自动将未保存的编辑保存到 localStorage，防止浏览器意外关闭导致数据丢失。

## 旅程地图总结

以下是所有旅程及涉及组件的概览：

| 旅程 | 用户操作 | AI Agent | Solution 后端 | CCAAS |
|------|---------|----------|--------------|-------|
| 创建备课方案 | 选择章节，与 AI 对话 | 通过 write_output 生成各区域 | 创建/保存备课方案 | 路由消息，管理会话 |
| 编辑区域 | 请求 AI 改进某区域 | 更新特定区域 | 更新备课方案 | 路由消息 |
| 审查和批准 | 审查 AI 输出，逐区域编辑 | 不参与（生成后阶段） | 保存逐区域编辑 | 不参与 |
| 浏览方案 | 导航 UI，筛选 | 不参与 | 提供备课方案列表 | 不参与 |

## 需要考虑的边界情况

旅程映射还帮助你尽早发现边界情况：

### 用户在 AI 生成时编辑怎么办？

AI 发送了 `learningObjectives` 区域的 `output_update`，但用户正在编辑该区域。

**决策：** 通过 `AIEditingOverlay` 显示视觉指示器（蓝色光晕动画）表明 AI 正在更新此区域。`FormStateSynchronizer` 追踪每次更新的来源（`'agent'` vs `'manual'`），前端可以区分 AI 生成的内容和用户编辑的内容。

### AI 生成了无效数据怎么办？

AI 发送了一个 `learningObjective`，其 `bloomLevel: "mastery"`，但有效值是 `remember`、`understand`、`apply`、`analyze`、`evaluate` 和 `create`。

**决策：** 在 MCP Server 的 `write_output` 处理器中校验。用清晰的错误消息拒绝无效值，让 AI 可以自我修正。

### 生成过程中网络断开怎么办？

AI 已经通过 `output_update` 发送了 `textbookAnalysis` 和 `learningObjectives`，但在 `learningTasks` 到达之前连接断开了。

**决策：** 使用会话重连（`reconnect_session` 事件）。前端应优雅地处理部分状态——显示已接收的区域，让用户手动完成其余部分或请求 AI 重新生成。

### 用户在生成过程中取消怎么办？

AI 还在调用 `write_output` 时用户点击了"取消"。

**决策：** 通过 Socket.io 向 CCAAS 发送 `cancel` 事件。前端应停止从后续的 `output_update` 事件更新编辑器区域。已更新的区域保留在编辑器中，供用户丢弃或保留。

### 教师想恢复某个区域怎么办？

AI 更新了 `learningTasks` 区域，但教师更喜欢原始版本。

**决策：** `useDraftPersistence` 组合式函数保留草稿副本。结合逐区域的编辑/取消功能，教师可以取消区域编辑来恢复到上次保存的版本。
