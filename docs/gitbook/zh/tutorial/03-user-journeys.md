# 3. 用户旅程映射

## 本章学习目标

在本章中，你将为任务管理器 Solution 规划端到端的用户旅程。用户旅程追踪用户完成一个目标的每一步——从打开应用到看到最终结果。在编写代码之前映射这些旅程，可以确保没有遗漏任何交互环节，并且 AI Agent、后端和前端都被设计为支持相同的工作流。

完成本章后，你将能够：

- 将功能分解为分步用户旅程
- 识别 AI Agent、后端和前端各自在哪里发挥作用
- 将 WebSocket 事件映射到旅程中的特定时刻
- 设计同时处理 AI 生成内容和用户编辑内容的旅程
- 在设计阶段尽早发现缺口和边界情况

## 为什么在编码前映射旅程

在传统 Web 应用中，数据流很简单：用户填写表单、提交、后端保存。在 LoopAI Solution 中，有第三个参与者——AI Agent——它引入了新的交互模式：

- 用户输入自然语言请求，而不是填写表单
- AI Agent 解释请求、生成结构化数据，并将其推送到表单
- 用户审查、编辑并确认 AI 生成的内容
- 后端保存最终结果

如果跳过旅程映射步骤，你会在实现过程中才发现这些交互模式，导致返工。

## 旅程 1：通过 AI 辅助创建任务

这是最主要的旅程。用户请求 AI Agent 创建任务，审查生成的内容，然后保存。

### 旅程中的关键时刻

| 步骤 | 组件 | 发生了什么 | 事件 |
|------|------|----------|------|
| 1 | 用户 | 输入自然语言请求 | -- |
| 2 | 前端 | 通过 Socket.io 发送聊天消息 | `chat` |
| 3 | Solution 后端 | 转发到 CCAAS REST API | HTTP POST |
| 4 | CCAAS | 启动带有 Skill 上下文的 AI Agent | -- |
| 5 | AI Agent | 对每个字段调用 `write_output` | tool_activity |
| 6 | CCAAS | 通过 WebSocket 推送结构化数据 | `output_update` |
| 7 | 前端 | 实时更新表单字段 | -- |
| 8 | 用户 | 审查并可选编辑 | -- |
| 9-12 | 前端 + 后端 | 标准表单提交 | HTTP POST |
| 13 | 前端 | 显示确认消息 | -- |

### 此旅程中的 WebSocket 事件

前端按以下顺序接收这些事件：

```typescript
// 1. Agent 开始思考
agent_status: { status: 'thinking' }

// 2. Agent 决定使用 write_output
tool_activity: { toolName: 'write_output', phase: 'start' }

// 3. title 字段同步
output_update: { payload: { data: { field: 'title', value: '部署新的 API' }}}

// 4. 工具调用完成
tool_activity: { toolName: 'write_output', phase: 'end' }

// 5. 更多字段依次同步...
output_update: { payload: { data: { field: 'description', value: '...' }}}
output_update: { payload: { data: { field: 'status', value: 'todo' }}}
output_update: { payload: { data: { field: 'dueDate', value: '2026-03-01' }}}
output_update: { payload: { data: { field: 'tags', value: ['backend', 'deployment'] }}}

// 6. Agent 完成
agent_status: { status: 'idle' }
```

### 这个旅程揭示的设计决策

1. **表单必须处理部分更新。** 字段逐个到达，而不是一次性全部到达。表单需要增量更新。
2. **用户在 AI 生成后编辑。** 表单在 AI 生成期间和之后都必须可编辑。
3. **保存是独立操作。** AI Agent 不保存——它只填充表单。用户必须显式保存。
4. **Solution 后端拥有数据。** 保存请求（步骤 10）发送到 Solution 后端，而不是 CCAAS。

## 旅程 2：用 AI 编辑现有任务

用户有一个任务，想让 AI 更新特定字段。

### 分步流程

```
 用户                    前端                     Solution 后端
  │                         │                         │
  │  1. 打开任务 #42        │                         │
  │────────────────────────▶│                         │
  │                         │  2. GET /tasks/42       │
  │                         │────────────────────────▶│
  │                         │  3. 任务数据             │
  │                         │◀────────────────────────│
  │  4. 表单填充现有数据    │                         │
  │◀────────────────────────│                         │
  │                         │                         │
  │  5. "给描述加点细节，   │                         │
  │   再加上测试相关的标签" │                         │
  │────────────────────────▶│                         │
  │                         │                         │
  │  ... (AI 生成, output_update 事件) ...             │
  │                         │                         │
  │  6. 只有 "description"  │                         │
  │     和 "tags" 字段更新  │                         │
  │     其他字段不变        │                         │
  │◀────────────────────────│                         │
  │                         │                         │
  │  7. 用户点击"保存"     │                         │
  │────────────────────────▶│                         │
  │                         │  8. PATCH /tasks/42     │
  │                         │────────────────────────▶│
  │                         │  9. 200 OK              │
  │                         │◀────────────────────────│
```

### 这个旅程揭示的设计决策

1. **上下文很重要。** AI Agent 需要知道当前打开的是哪个任务。Skill 指令应该指定如何读取现有上下文（通过 `read_context` 或在聊天消息中接收当前表单状态）。
2. **部分更新是必要的。** AI Agent 应该只更新用户要求的字段，保持其他字段不变。
3. **前端必须合并而不是替换。** 当 `description` 字段的 `output_update` 到达时，前端只替换该字段，保持 `title`、`status` 等不变。

## 旅程 3：批量创建任务

用户想根据项目描述一次创建多个任务。

### 分步流程

```
 用户                    前端                     AI Agent
  │                         │                       │
  │  1. "把这个项目拆分     │                       │
  │   成任务：构建用户认证  │                       │
  │   系统，包含登录、注册  │                       │
  │   和密码重置"           │                       │
  │────────────────────────▶│                       │

  │                         │──────────────────────▶│
  │                         │                       │
  │                         │  output_update:       │
  │                         │  field: "tasks"       │
  │                         │  value: [             │
  │                         │    { title: "...",    │
  │                         │      description: ... │
  │                         │      status: "todo"}, │
  │                         │    { title: "...",    │
  │                         │      ...},            │
  │                         │    { title: "...",    │
  │                         │      ...}             │
  │                         │  ]                    │
  │                         │◀──────────────────────│
  │                         │                       │
  │  2. 任务列表渲染        │                       │
  │     3 个新任务          │                       │
  │◀────────────────────────│                       │
  │                         │                       │
  │  3. 用户审查，          │                       │
  │     删除一个，          │                       │
  │     编辑另一个          │                       │
  │────────────────────────▶│                       │
  │                         │                       │
  │  4. 用户点击            │                       │
  │     "全部保存"          │                       │
  │────────────────────────▶│                       │
  │                         │  POST /tasks/bulk     │
  │                         │──────────────────────▶│
```

### 这个旅程揭示的设计决策

1. **数组字段需要特殊处理。** 当 AI 生成多个任务时，它以数组形式发送。前端需要列表视图，而不是单项表单。
2. **同步字段契约扩展。** 除了每个字段的同步字段外，你可能需要一个接受任务对象数组的 `tasks` 同步字段。
3. **批量保存需要独立端点。** 单个 POST 到 `/tasks` 创建一个任务；你需要一个 `/tasks/bulk` 端点来批量创建。

不是每个旅程都涉及 AI Agent。有些是纯 CRUD 操作。

  │  1. 打开项目            │                         │
  │     "API 后端"          │                         │

### 这个旅程揭示的设计决策

1. **Solution 后端需要带过滤功能的列表端点。** 你需要 `GET /projects/:id/tasks` 并支持 status、tags 和日期范围等查询参数。
2. **不是所有操作都经过 CCAAS。** 领域数据的读取操作直接访问 Solution 后端——只有在调用 AI Agent 时才涉及 CCAAS。

## 旅程地图总结

以下是所有旅程及涉及组件的概览：

| 旅程 | 用户操作 | AI Agent | Solution 后端 | CCAAS |
|------|---------|----------|--------------|-------|
| 创建任务 | 发送聊天消息 | 通过 write_output 生成字段 | 保存任务 | 路由消息，管理会话 |

| 查看任务 | 导航 UI | 不参与 | 提供任务列表 | 不参与 |

## 需要考虑的边界情况

旅程映射还帮助你尽早发现边界情况：

### 用户在 AI 生成时编辑怎么办？

AI 发送了 `description` 字段的 `output_update`，但用户正在编辑该字段。

**决策：** 显示一个视觉指示器表明 AI 正在更新此字段。如果用户已做编辑，提供选择：保留用户版本还是接受 AI 版本。

### AI 生成了无效数据怎么办？

AI 发送了 `status: "completed"`，但有效值是 `todo`、`in_progress` 和 `done`。

**决策：** 在 MCP Server 中校验。用清晰的错误消息拒绝无效值，让 AI 可以自我修正。

### 生成过程中网络断开怎么办？

AI 已经通过 `output_update` 发送了 `title` 和 `description`，但在 `status` 和 `tags` 到达之前连接断开了。

**决策：** 使用会话重连（`reconnect_session` 事件）。前端应优雅地处理部分状态——显示已接收的字段，让用户手动完成其余部分。

### 用户在生成过程中取消怎么办？

AI 还在调用 `write_output` 时用户点击了"取消"。

**决策：** 通过 Socket.io 发送 `cancel` 事件。前端应停止从后续的 `output_update` 事件更新表单字段。已更新的字段保留在表单中，供用户丢弃或保留。

## 本章小结

在本章中你学到了如何：

- **追踪完整的用户旅程**，从初始操作到最终结果
- **识别每个组件的角色**（前端、Solution 后端、CCAAS、AI Agent）在每个步骤中
- **将 WebSocket 事件映射**到旅程中的特定时刻
- **发现旅程揭示的设计决策**（部分更新、合并 vs. 替换、保存 vs. 生成）
- **设计 UI 布局**，基于交互模式
- **在编写任何代码之前识别边界情况**
