D-01

问: SSE 怎么做 auth?

Backend · SSE
触发 实现完 SSE auth 写完单元测试都过,wire test 一跑发现浏览器收到 HTTP 200 + 流静默关闭,根本看不到 401/403。 约束 NestJS 的 @Sse() 装饰器在订阅 Observable 之前就提交了 HTTP response status + Content-Type: text/event-stream。Observable 内部 error() 时已经晚了。
OptionApproachVerdict
A 在 Observable 内部用 from(authPromise).pipe(switchMap(...)) 包一层,auth fail 就 subscriber.error()。 ❌ 拒 —— @Sse 已经提交 200,error 只是静默关流。
B 把 auth 提到 NestJS Guard,在 canActivate 阶段跑。Guard 抛 UnauthorizedException / ForbiddenException。 ✓ 选 —— Guard 在 handler 之前跑,状态码能正确送到浏览器。
为什么这么选

NestJS 的请求生命周期是 Guard → Pipe → Handler → Interceptor → Response。 一旦 handler 返回 Observable 给 @Sse,response 头部就已经在 wire 上了 —— 这时候 Observable 内部的 error 只能体现为 "stream 中断",无法回写 status code。

Guard 是 NestJS 唯一一个 pre-handler 的边界,在 response 头部确定之前能 throw HTTP exception。 对所有未来的 SSE endpoint 都适用 —— Guard 化是 SSE auth 的唯一正确解。

Locked in
所有 ccaas SSE 路由都用 @UseGuards() 模式。Guard 类是 Reusable 边界。
Locked out
per-event auth(每条 message 重新鉴权)做不到。如果未来需要 token rotation mid-stream 要换设计。

D-02

问: Tenant resolver port 的 API 形状?

Design · Port
触发 原本写成 resolveTenant(projectId): tenantId | null,code reviewer 指出这个签名让 multi-tenant 场景下有隐患 —— 如果一个 projectId 被多个 tenant 都 bind 过(理论可能),"who owns this project" 没有唯一答案。 约束 project 在 ccaas-core 是不透明字符串,ccaas 不知道 project 是怎么分配 tenant 的 —— 必须由 solution 端回答。
OptionAPI shapeVerdict
A resolveTenant(projectId): Promise<string | null> —— "who owns this project" ❌ 拒 —— 隐式假设 project ↔ tenant 是 1:1,实现层可能选错 tenant。
B verifyProjectAccess(projectId, callerTenantId): Promise<boolean> —— "does this tenant own this project" ✓ 选 —— 把 caller 的身份注入,问题变成可判定的 pair-check。
为什么这么选

Option A 的形状让 resolver 的实现 决定"哪个 tenant 拥有这个 project", 这是个开放性问题 —— 实现可以做任意映射。结果是 ccaas-core 失去对 tenant 边界的控制。

Option B 把问题转成 布尔判定:caller 声称自己是 tenant X,resolver 只回答 yes/no。 ccaas-core 仍然持有 tenant 边界的语义(从 API key 解析出 tenant),resolver 只回答 "这个 tenant 能不能访问这个 project"。 API 的"问题域"更小,实现的自由度更窄,安全边界更清晰。

Locked in
每次 verify 都需要 caller 已经被 ccaas 解析出 tenantId(API key 校验通过)。
Locked out
无法用此 port 列举 "tenant X 拥有哪些 projects" —— 那是另一个 query,需要另一个 port 或反向接口。

D-03

问: ccaas 怎么知道一个 project 属于哪个 tenant?

Runtime · Resolution
触发 D-02 的 port 形状定了,但 ccaas-default 怎么实现?最初方案让每个 solution 写自己的 resolver(查它的 project 表)。 约束 Solution 是跨进程的;ccaas → solution callback 要走 HTTP,加延迟、加失败模式;每个 solution 要 schema migration 加 tenantId 列;接入门槛高。
OptionApproachVerdict
A 每个 solution 实现 resolver(查自己的 CourseProject.tenantId 列),ccaas 通过 HTTP 调它。 ❌ 拒 —— 每个 solution 加 column + 加 endpoint + 跨进程 latency。
B ccaas 默认 resolver 直接查自己的 session_metadata 表,该表已经被 bind-project 流程写入 (tenantId, projectId) 行。 ✓ 选 —— 零 solution 改动,单次索引化 SQLite lookup。
为什么这么选

bind-project 流程在 ccaas 这边已经把 (sessionId, tenantId, projectId) 三元组写到了 session_metadata 表 —— 这是 ccaas 本来就知道 的事实, 不需要再问 solution 一遍。

Option A 是"找 source of truth 的源头去问",Option B 是"复用既有 cache"。 权衡的关键: bind-project 必然先发生(用户必须先 bind 才能开始 sync), 所以这条信息在我们需要时一定已经存在。代价: 没 bind 过的 project resolve 不到 → 403, 但这正是 "用户没 bind 就不该有访问" 的语义。

Locked in
Solution 端只要走 bind-project,就自动获得 SSE 鉴权能力,零代码。
Locked out
想要 "未 bind 也能预览" 这类流程的话需要换 resolver(可以通过覆盖 PROJECT_TENANT_RESOLVER token 实现)。

D-04

问: 浏览器要不要持有 ccaas API key?

Security · Boundary
触发 写 Phase 2b 文档时我建议 creator app 用 localStorage.setItem('ccaas:apiKey', ...) 让运营粘贴 key。用户直接打回: "按我的设计,solution 才是 ccaas 的 tenant,end user 属于 solution,浏览器不该见 ccaas key"。 约束 浏览器要订阅 SSE,而 ccaas SSE 需要 auth。如果浏览器不持有 ccaas key,必须有人代理这次调用。
OptionApproachVerdict
A localStorage 持有 ccaas key,浏览器直接打 ${VITE_CCAAS_URL}/projects/.../changes?token=... ❌ 拒 —— 违反 "ccaas knows tenants, not end users" 边界。
B Solution backend (live-lesson) 持有 env CCAAS_API_KEY,新加 proxy controller,浏览器只打同源 /api/projects/.../changes ✓ 选 —— 浏览器零 ccaas 状态;solution backend 一个 key。
为什么这么选

边界问题不是技术问题,是角色定义问题。 ccaas 这个平台的"用户"是 solution,不是 end user。Solution 自己有它的鉴权体系(cookie / JWT / 任何形式), end user 通过 solution 的体系登录;solution backend 是 ccaas 的客户。

一旦把这条线画清楚,Option A 就是把 ccaas 的 master credential 暴露给 solution 的 end user, 这是 credential confusion —— end user 拿到的是 solution 给他的 session,不该有 ccaas 的 API key。 Option B 通过 proxy 让 solution backend 重新成为 ccaas 的唯一调用方,边界还原。

Locked in
Solution backend 必须为每条需要的 ccaas 调用提供 proxy endpoint。增加了 boilerplate。
Locked out
浏览器无法直接 debug ccaas API,必须通过 solution 那一层。

D-05

问: Binary 支持做完整 vertical 还是 MVP?

Design · Scope
触发 Phase 2b-4 启动时问过 "MVP(只做 port + engine)还是 Full(再加 REST adapter + syncer)?"。当前没有任何 in-tree consumer 需要 binary。 约束 MVP ~1 天,Full ~3-4 天。但 API shape 一旦定下来,以后的 binary consumer 会按这个形状写代码 —— shape 设计错了要返工。
OptionScopeVerdict
A MVP — 只做 port 和 SyncEngine.planBinary;REST + syncer 等真出现 consumer 再做 ❌ 拒 —— shape 没被"完整 vertical 跑通"验证过,容易设计错。
B Full — port + engine + REST adapter + syncer + 端到端单元测试 ✓ 选 —— 把 shape 在完整 vertical 中跑通一遍,后续接入零返工。
为什么这么选

API 的设计正确性只有在 完整的调用链跑通 之后才能验证。 只做 port + engine 的话,REST 传输层(streaming、Content-Length、size cap)的边界条件没被试过, 后续接入第一个 binary consumer 时大概率发现需要返工 API。

多花 2-3 天换 "未来的 binary 用户不用改 ccaas",是合算的 —— 因为修改一个已发布的 port 形状 的代价比一开始就把它做对要大几倍(下游消费者要跟着改)。这是 "在没人用之前先把 shape 做对" 的投资。

Locked in
整个 binary vertical 在 ccaas 端已存在并被测试覆盖。未来的 binary solution 实现 REST 即可接入。
Locked out
~2-3 天的 ccaas-side 工作量提前花了。如果 binary 这个方向最终没用,这部分代码是 dead weight。

D-06

问: Invalidate proxy 上游 4xx 时怎么处理?

Backend · Error
触发 Code reviewer 提出:invalidate 上游 401/403 时,如果一律 swallow 成 {accepted: false} 给浏览器,运营永远不会发现 CCAAS_API_KEY 配错了。 约束 invalidate 在设计上是 "optional optimization" —— 即使失败,下一轮 turn 也会自动 sync,不算 critical path。但 401/403 是 operator-actionable 的配置错误,跟 5xx (upstream busy) 不同。
OptionApproachVerdict
A 所有上游 4xx + 5xx 一律返回 {accepted: false},只 warn log ❌ 拒 —— auth 配错静默永久 degraded。
B 401/403 抛 BadGatewayException(502) + error log;5xx/404 保留 {accepted: false} + warn log ✓ 选 —— 分类处理:auth 失败 fail-loud,其他保留 best-effort。
为什么这么选

"Optional optimization" 的语义是说调用失败不影响功能, 不是调用失败的原因不重要。401/403 是配置错, 这种错沉默下去会变成 "agent 永远不实时 sync,但 SSE 又看起来正常",运营调试这个 bug 要小时级。

反观 5xx (upstream busy)、404 (project not yet bound) —— 这些是瞬时或预期的状态, 下一轮 turn 会自动恢复,值得保留 best-effort 行为。 所以 4xx 的二分(auth-class vs not) 比 4xx vs 5xx 的二分更准确地反映了问题域。

Locked in
Operator 配错 CCAAS_API_KEY 在第一次 invalidate 调用就会看到 502 + 日志。
Locked out
GUI 调用 invalidate 时需要处理 502(虽然实践中一旦配对就稳定)。

OQ-01

什么时候 introduce 短期 exchange token?

问题SSE 用 ?token=master-key 是因为 EventSource 不能 set headers。Master key 写在 URL 这件事在 access log / proxy log 里会留痕;真正的 multi-tenant 公网部署应该改成短期 scoped token 交换(POST /sessions/exchange)。

为什么 open当前 ccaas 是 single-replica + 内网部署;log leak 风险低。但什么阈值后必须做交换?没明确标准。可能的触发: 第一次部署到公网域名 / 出现 audit 合规要求 / 出现第一个非内部 solution。

OQ-02

Live key rotation 怎么做?

问题当前 rotation = restart。但生产环境如果想做 zero-downtime rotation(K8s rolling update + vault secret reload),tenantId cache 怎么 invalidate?401 上游响应怎么触发重新 resolve?

为什么 open不知道未来的 ops shape。可能就是 K8s rolling restart(不需要 live)。也可能是 vault webhook → cache invalidate。设计取决于 ops 流程。Tracked in backlog.md 但没设计。

OQ-03

多 instance ccaas 的故事是什么?

问题ccaas 当前是 single-replica(SQLite + agentfs FUSE 都假设单进程)。如果要 scale 出去,SyncEngine 的 per-projectId mutex、session_metadata 的 SQLite、agentfs 的 FUSE mount 都需要换。

为什么 open没有 capacity pressure 驱动 multi-instance;单实例可以扛到什么规模也没测。设计层面 mutex 可以换 Postgres advisory lock,SQLite 换 PG,但 agentfs FUSE 这一块是真正的 single-host constraint。
