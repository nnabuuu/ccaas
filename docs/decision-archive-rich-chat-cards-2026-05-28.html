结构化进度:打通三类富卡片。

creator-v7-rich-chat-doc.md (8 节正式规格) 描述了三类 chat 富消息卡片 (Todo / Questions / Verify), 完整 mockup 在 creator-v7-rich-chat.jsx 已实现 —— 但 shipped 创作器只渲染纯文本气泡。 用户带着假设进来: "实现可能要走 ccaas-core 的 'solution-defined tools + sessionTemplate' 机制, 但担心已 broken"。

第一个意外发现: 假设错了。 两套机制都 LIVE + tested —— solution.json 注册 MCP server + toolEventTriggers 配置就能把 tool 结果转为 output_update SSE event。 Live-lesson 一直在用它跑 classroom 场景, 只是 creator 没用过。 POC 真正的工作是第一次把 creator 接入这条已经在跑的轨道。

D-09

问: 富 chat 卡片走 ccaas-core MCP 还是 hack 私有渠道?

Contract · Architecture
触发 用户假设要走 "solution-defined tools + sessionTemplate" 但担心 broken。 研究发现机制完整 + classroom 在用 (3 个 MCP 工具配置了 toolEventTriggers), 只是 creator 场景没用过。 问题变成: 我们也走这条路, 还是 live-lesson 自己 hack 一个绕开 ccaas 的私有渠道? 约束 creator 当前用 HTTP 直连 ccaas (CcaasChatProxyController), 已经能塞任意数据进 chat。 直接 hack 一个 emit_card endpoint 跳过 MCP 协议是可能的。 但任何后续solution想要类似富消息能力, 都要重新发明轮子。
OptionApproachVerdict
A live-lesson 加私有 HTTP endpoint, 绕开 ccaas-core MCP 完全自管 ❌ 拒 —— 重复造轮子 + 跟 ccaas-core 解耦了 future-proof 但也丢了所有 future-proof
B 注册 MCP server + toolEventTriggers: [{ toolName, eventType:'output_update', field:'card' }] ✓ 选 —— 0 ccaas-core 改动, 复用现成机制, 任何 solution 都能照样做
C 改 EventMapperService 加一个 hardcoded "live-lesson 富卡片" branch ❌ 拒 —— ccaas-core 不该知道某个 solution 的具体 UI 概念
为什么这么选

这是个 "走对了既有架构 vs 偷工" 的选择。 ccaas-core 的 MCP + toolEventTriggers 机制本来就是为这个场景设计的 —— field: 'card' 把工具结果包成 { field: 'card', value: <parsed> } 然后 emit 为 output_update SSE event。 这是已经在跑的通路, 我们只是第一次用它。

Option A 看着诱人 (live-lesson 全栈在手, 添个 endpoint 几小时搞定), 但它把 ccaas-core 当成了哑管道。 一旦其他 solution 想要类似功能, 它们要么各自 hack 一遍, 要么我们最终被迫做 B 的扩展 —— 反正都得做, 不如一次做对。 Phase 0 烟雾测试只用了 10 分钟就确认了 (跑现有的 event-mapper.service.spec.ts 10/10 全绿), 假设破除。

Locked in
任何新富消息类型只要: 加 MCP 工具 + 注册 toolEventTriggers + 写 frontend 渲染器。 0 ccaas-core 改动。
Locked out
MCP server 进程生命周期 + 启动延迟由 ccaas-core 控制; solution 无法精调。 工具 schema 是 raw JSONSchema (不是 Zod-to-JSONSchema bridge), 手工同步两边。

D-10

问: 富卡片怎么持久化, 页面刷新后还能看到?

Persistence · Schema
触发 POC 阶段卡片只在 live SSE 流中渲染, 刷新页面就丢了。 生产环境必须持久化, 但当前 messages 表只有 role + text, 没 type 字段。 选什么持久化路径? 约束 不想 migrate messages schema (跨包改动, ccaas-core + 多 solution 都要追)。 但工具调用结果已经被 ccaas-core 写到了 tool_events 表 (toolName + toolId + toolOutput), 只是前端 history 渲染只把它们当"调用 X 工具"通用标签显示。
OptionApproachVerdict
A messages 表加 type + payload 列 (schema migration) ❌ 拒 —— 跨包 schema 改动 + 要同步改 ccaas-core message persistence
B 复用 tool_events.toolOutput 存 card payload, 前端 history reload 时按 toolName 重建 card message ✓ 选 —— 0 migration, 现成持久化通路, frontend pure helper 完成 rebuild
为什么这么选

tool_events.toolOutput 已经在持久化我们要的东西 —— MCP 工具的返回值就是 card payload (我们的 echo 设计让这一点天然成立)。 所以"持久化"的真问题不是 schema, 是前端 history 渲染时把这些 toolOutput 重新当成卡片消息显示。 辅助函数 expandHistoryMessage 做这个: 遍历 toolEvents, 凡是 emit_*_card 工具且 toolOutput 能 parse 出 kind, 就 hoist 出一条独立的 card-type ChatMessage。

代价: 前端的两条渲染路径 (live SSE / history reload) 必须语义等价。 活路径 push 一条 {type:'card', card}; 重建路径同样 push 一条, 用 toolId 做 React key (后端分配, 跨 reload 稳定)。 一致性 review 后特意做了 "card 插入位置" 的修正 —— 活路径要把 card 插在 in-flight agent bubble之前, 这样跟 history reload 的 hoist-before-text 顺序一致, 用户刷新页面不会看到消息顺序突变。

Locked in
所有 emit_*_card 工具的 toolOutput 是持久化的唯一数据源。 schema 演进只要保持 kind 字段 + backward-compat。
Locked out
toolOutput 持久化 size cap 限制了卡片体量 (verify 几十检查项是上限)。 卡片提交后的state (Questions 的 submitted=true) 不持久化 —— 刷新后 submit 按钮重现, 但下游 user message 已发, agent 已 reply, 实际不会出 bug。

D-11

问: POC 只做最简单的 Todo, 还是 3 个卡片同时做?

Design · Scope
触发 Phase 1 plan 默认 POC 只做 Todo (最简单, 单向数据流), Questions + Verify 留后续 sprint。 但用户决定: 三个都上 (~7.5 天 vs 3 天)。 约束 Todo 是单向 (LLM → 卡片), Verify 多一个 running 动画 (前端 setInterval 模拟), Questions 多一个回流 (教师提交 → user message → agent). 复杂度递增。
OptionApproachVerdict
A 只 Todo, 验证 architecture, 另两个留后续 ❌ 拒 —— architecture 验证不需要 3 个; 但UX 价值验证需要
B 三个一起做 (Todo + Verify + Questions), 把 architecture 跟 UX 完整价值同时落地 ✓ 选 —— +4.5 天换"教师真能用"
C Todo + Verify (跳过 Questions 的回流复杂度) ❌ 拒 —— Questions 是唯一双向卡, architecture 真正的"完整测试"需要它
为什么这么选

POC 的目的不只是"证明能做", 而是"教师真能用得上"。 Todo 单独验证了数据从后端到前端的渲染管道; Verify 验证了前端 stateful 卡片 (动画) 的渲染管道; Questions 验证了用户输入回流到 agent 的反向通路。 三个一起做才把 architecture 跑完整 —— 反向通路是唯一新 architecture 点, 不做就只是"单向 SSE 验证", 跟其它 output_update event 没本质区别。

+4.5 天的代价主要在 Phase 3 (component 实现): Verify 1d, Questions 1.5d, vs Todo 0.5d。 但测试规模化几乎免费 (每个 card 5-8 个 component test); review-driven fix 也是分摊 (3a 的反 motion 修复让 3b/3c 受益)。 最大风险在 Questions 的 submit-back 设计 —— 哪个文本格式 agent 能 reliably 解析? 这次选 - [id] label: value 结构化前缀, 但属于 contract decision, 真接入下游 prompt 可能要调。

Locked in
完整 architecture 端到端 (单向 + 双向) 验证。 Todo / Verify / Questions 三个 react 组件 + 完整 component tests 是 future 富卡片的模板。
Locked out
+4.5 天的实施 effort。 review cycle 也长了 (CRITICAL 的 questions 静默 freeze bug 只有在 Questions 真做了才暴露)。

D-12

问: Questions 卡片提交后, 答案怎么回流给 agent?

Contract · Submit-back
触发 原型只有 setSubmitted(true), 没设计提交后的 agent 通信。 但实际场景里 agent 问了问题, 它需要收到答案才能继续。 怎么把教师的填写发回 agent? 约束 不想新增 user message type (跟 D-10 同样的"避免 schema migration"动机)。 chat 协议唯一标准 user → agent 通路是 user message text。 但 review 暴露了一个 CRITICAL: 如果 send 被 silently 丢弃 (chat 忙), 卡片 freeze 显示 "✓ 已确认", agent 永远收不到答案。
OptionApproachVerdict
A 新增 user message type 'questions_answer', agent 解析结构化字段 ❌ 拒 —— schema 改动 + 跨包契约工作量超过 POC 预算
B 提交时自动 send 一条 user message 文本 (已确认以下选择:\n- [q1] label: value\n...), agent 文本解析 ✓ 选 —— 0 协议改动, agent prompt 教它格式即可
为什么这么选

User message text 是已经存在的 user → agent 唯一通路。 任何 "新通路" 都要在 chat 协议上加 type 字段, ccaas-core 跟所有 solution 都要追。 文本通路的代价是解析的脆弱性 —— 这是 review 阶段提出的 (Warning #2): 当 label 含 : 或重复时 agent 难分清。 修复: 文本格式加 [id] 前缀 (- [q1] 矩阵维度: ...), 给 agent 一个稳定 anchor 去 regex 匹配。

但review 还抓到一个更深的 bug: 提交流程是 "card 调 onSubmit → setSubmitted(true)" 这两步是同步的。 如果 onSubmit 是 sendNow(text), 而 chat 当时 isThinking=true, sendNow no-op 返回 void —— card 仍然显示"已确认"但 agent 没收到。 Silently 丢失 user 输入是 chat UI 的最大灾难。 修复让 sendNow 返回 boolean (true=已发, false=guarded), card 检查返回值, false 时不冻结允许重试。 没这条修, 整个回流 architecture 就是个谎言。

Locked in
submit-back 用 user message 文本通路, agent prompt 教它解析 [id] 前缀。 Drop-detection (sendNow boolean return) 是所有 chat-bridge 路径的新契约 (action://fix 也受益)。
Locked out
agent prompt 要稳定地按格式解析; 如果 LLM 偶尔漂移 (省略 id 前缀 / 用别的语言), 解析失败。 prompt-tuning 是后续工作。

DEF-05

LLM prompt 生产化 (per-skill + per-card-type 调优)

现状POC 在 solution.json 的 appendSystemPrompt 里塞了一段总指令告诉 agent 三个工具的用法。 真集成需要 per-skill 决策 (谁能用 emit_questions_card, 在什么 surface 上)。

推迟理由需要真实 dogfood 数据 (agent 用得对不对? 教师吃不吃这套?), 否则 prompt 调成什么样都是猜。 POC 阶段 prompt 够 demo。

重新审视的触发第一个真实班级用 creator 产生 ≥10 个真实 chat 对话之后, 拉数据看 emit_*_card 触发率 + 命中率。

DEF-06

卡片 schema 升级路径 (v1 → v2 backward-compat)

现状今天三个卡片的 kind + 字段都是 v1 隐含的 (没显式版本号). 持久化的 toolOutput 也没 schemaVersion 字段。

推迟理由没真实 schema 演进发生; 提前设计版本协议属于过度工程。 frontend 已经做了 forward-compat (不识别的 kind 静默忽略 + 不识别字段不破坏渲染), 倒退兼容性能撑。

重新审视的触发第一次需要给某个 card 加 required 字段时。 加 schemaVersion: 'v2' + 前端按版本路由。

DEF-07

ChatTodoCard 流式更新 (中途改 active item)

现状后端一次性发完整 card payload。 如果 agent 想说"item 3 完成了, 现在做 item 4", 它需要重发整个 card (会渲染成第二张卡片而非更新第一张)。

推迟理由POC 简化决定, 真用起来教师可能会 expecting incremental update。 但解决方案需要在 SSE 协议层加 "patch this previous card by id" 概念, 是新 contract。

重新审视的触发第一次 dogfood 反馈 "为什么 todo 重复出现" 类型抱怨。

DEF-08

QuestionsCard 键盘 a11y (radio arrow-key navigation)

现状radio options 是 <button role="radio"> 列表, 没有 ArrowUp/ArrowDown 导航。 鼠标 OK, 屏幕阅读器在 VoiceOver/NVDA 下也工作 (会读 role + aria-checked), 但纯键盘用户必须 Tab 过每个 option。

推迟理由真 fix 需要要么换原生 <input type="radio"> (样式重写麻烦) 要么自己实现 keyboard handler (~30 行 + 测试)。 review 提了, 但不阻塞 POC。

重新审视的触发实际有 a11y 用户反馈, 或 a11y audit 要求。

OQ-04

submit-back text 格式跟 agent 解析在真 LLM 上的可靠性

问题POC 选了 - [q1] label: value 文本格式 + prompt 教 agent 按 [id] regex 匹配。 但 Claude / 其他 LLM 在多轮 chat 后是否仍能稳定按格式 emit + parse?

OQ-05

card 应该不应该有"撤销"按钮?

问题prototype 没有撤销。 Questions 提交后 freeze 就 freeze 了; Todo 没"取消任务"; Verify 没"重跑"。 但教师真用起来, 万一选错了选项, 当前的恢复路径是"在 chat 里发一条文字纠正"。 体验够好吗?

为什么 open取决于教师真使用模式。 撤销按钮本质是card state 跟 chat history 的关系问题 —— card 已经把 user message 发出去了, 撤销意味着从 history 里删掉那条 user message + 让 agent 知道, 涉及 ccaas-core 行为。

OQ-06

classroom 跟 creator 的两套 MCP server 该不该合并?

问题live-lesson 现在有两个 MCP server 进程: mcp-server/ (classroom 用, 板书工具) 跟 creator-mcp-server/ (creator 用, 富卡片工具)。 是不是该合并成一个?

为什么 open合并的好处: 一个进程少一份生命周期管理。 坏处: 两组工具的语义很不一样 (classroom 是"操作真课堂", creator 是"在 UI 上渲染卡片"), 合在一起 schema 文件会很混乱。 sessionTemplate 已经能按 template 分配工具集了, 拆开更清晰。 当前的决定是拆开, 但如果未来 solution 数量大涨这个判断可能要重审。
